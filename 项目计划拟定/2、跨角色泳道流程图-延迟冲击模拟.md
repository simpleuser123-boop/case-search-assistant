### 2. 跨角色切换点风险扫描表

| 切换点                    | 输出角色 | 输入角色 | 交付物                                  | 切换标准                                           | 如果交接延误的影响链                                         |
| ------------------------- | -------- | -------- | --------------------------------------- | -------------------------------------------------- | ------------------------------------------------------------ |
| PM → 后端                 | PM       | 后端     | 筛选后的失败查询日志、需求边界定义      | 日志质量达标、需求边界已签字确认                   | 若日志不可用或需求模糊，后端日志分析无法启动，直接冲掉第1天所有开发任务 |
| 后端 → 测试（第1天）      | 后端     | 测试     | 日志分析结论（失败模式总结）            | 日志分析完成，可指导样本集覆盖失败类型             | 延误导致样本集构建缺乏指向，样本质量下降，后续评测置信度不足 |
| 测试 → 前端（第2天启）    | 测试     | 前端     | 评测样本集（20条query）                 | 样本集已构建完成（不依赖后端规则）                 | 前端静态改造可独立进行，实际影响较小；但若样本集严重延误，会间接挤压第3天评测准备 |
| 前端 → 后端（第2天）      | 前端     | 后端     | 前端静态改造完成的页面结构              | 静态框架就绪，后端规则可参照字段名设计排序理由字段 | 若前端静态框架未定，后端规则输出的排序理由字段可能需返工适配 |
| 后端 → 前端（第2天）      | 后端     | 前端     | 可部署的排序权重规则集+排序理由字段定义 | 规则定稿，字段名、取值逻辑确定                     | 前端动态展示强依赖规则输出；规则延误将直接压缩前端动态开发时间，甚至导致前端展示无法完成 |
| 前端 → 测试/集成（第3天） | 前端     | 测试     | 完成的前端页面+埋点                     | 页面可交互，埋点事件就绪                           | 前端未完成则集成联调无法进行，评测无法部署，整个第3天任务链断裂 |
| 测试 → PM（第3天）        | 测试     | PM       | 评测报告（NDCG@10对比、bug清单）        | 报告结论清晰（改善/恶化明确）                      | 若报告结论模糊（部分改善、部分恶化），PM 上线决策陷入胶着，可能延期上线或做出高风险决策 |

**特别关注的后端→前端切换**：规则编写耗时 3h，若因术语映射等复杂度过高膨胀至 5h+，会直接吃掉第 2 天下午时间，使前端动态部分只能挪至第 3 天，进而挤压测试与上线窗口。

### 3. 延迟冲击模拟

#### 模拟 A：后端/算法角色任务整体延迟 2 天（原 5h → 实际 15h，日志分析+规则编写占满第1、2天）
- **影响链**：第 1 天仅完成 PM + 日志分析；第 2 天全天规则编写；前端 C1/C2 完全无法在第 2 天开始；测试 D1 已在第 1 天完成，但 D2/D3 无法启动。
- **总工期影响**：基线 3 天 → 至少 5 天（前端 1 天 + 集成测试 1 天不可压缩）。
- **降级方案**：
  - 砍前端动态展示改造，仅保留静态页面和埋点框架；
  - 后端规则上线后直接观测埋点数据（点击率），不做 NDCG 评测；
  - 评测样本从 20 条压缩至 5 条核心 query，仅做人工评估。

#### 模拟 B：测试/评测角色任务延迟 1 天（样本构建从 2h 膨胀至 4h，或评测发现规则不达预期需返工）
- **影响链**：若返工发生在第 3 天下午（如规则修改后重新评测），上线窗口丧失；评测报告无法及时交付 PM，上线决策瘫痪。
- **总工期影响**：至少延至第 4-5 天。
- **降级方案**：
  - 不进行完整 NDCG 评测，仅选取 Top 5 核心 query 做人工对比判断；
  - 直接上线并密切监控线上“点击率”指标，若 24h 内提升 < 10% 即触发回滚；
  - 功能回归测试仅做冒烟，不覆盖全 case。

#### 模拟 C：失败查询日志质量极差，无法支撑规则优化（MoSCoW 预判风险）
- **根因**：日志中口语化查询零碎、无上下文，难以提取法律术语映射模式。
- **影响**：原计划“日志驱动 → 规则编写”链条断裂，Must 需求“检索结果高精准度”空心化。
- **备选方案**：
  - A：律师人工标注 10-20 组口语→术语映射对，替代日志驱动，但耗时增加至少 1 天；
  - B：放弃术语映射与段落匹配，仅调整 LLM 重排序的 prompt，让 LLM 自行完成口语→法言法语转换；
  - C：承认 3 天不足以做出可感知提升，将迭代目标降级为“建立评测基线和失败日志收集机制”，本次不上线排序优化。

### 4. 缓冲方案

#### 建议增加缓冲的切换点
| 切换点               | 建议缓冲           | 理由                                                         |
| -------------------- | ------------------ | ------------------------------------------------------------ |
| 后端 → 前端（第2天） | **+30min**         | 规则集定稿后需与前端字段对齐，缓冲用于文档同步与接口确认，避免返工 |
| 前端 → 测试（第3天） | **+15min**         | 联调环境准备、埋点验证通常会出现小问题，预留修补时间         |
| 测试 → PM            | **已含切换 15min** | 评测报告生成可在评测执行最后 15min 同步撰写，无需额外缓冲    |

#### 可提前并行准备的任务
- **前端静态改造**：在 PM 确认需求后即可启动，无需等待后端任何产出。应将页框、埋点代码框架、样式调整与后端规则解耦，提前完成。
- **测试样本集构建**：与后端日志分析并行（虽同人不可并行，但样本集部分可基于历史查询先粗选 10 条，待日志分析后补充 10 条，减少后段纯手工时间）。
- **集成联调脚本**：可在规则编写阶段提前写好自动化部署脚本，减少联调时手动操作。

#### 强制压缩工期时的最优降级路径
**砍测试（样本 20→5） > 砍前端展示改造（仅上线后端规则） > 砍规则复杂度（只做案由加权）**
- 测试样本压缩风险：评测结论置信度下降，但配合线上埋点可快速验证。
- 前端展示砍掉不影响排序核心逻辑，用户仍可通过结果质量提升感知。
- 规则复杂度砍掉会削弱优化效果，可能达不到 15% 点击率提升目标，但可积累经验。

#### “不上线比强行上线更危险吗？”——决策标准
若第 3 天评测结论为“部分 query 改善、部分恶化”，应遵循以下决策矩阵：
- **上线条件**：
  - NDCG@10 整体提升 ≥ 5%，且无致命 bug；
  - 改善 query 数 ≥ 恶化 query 数，且恶化幅度 ≤ 10%；
  - 回滚机制已就绪（可一键切回基线）。
- **不上线条件**：
  - 任一核心 query 结果严重恶化（如首位不相关）；
  - 埋点数据验证未通过（如点击率统计口径错误）；
  - PM 无法在 30min 内做出清晰判断。
- **风险分析**：强行上线可能导致用户对搜索信任度下降，留存率不升反降；不上线则丢失一次真实反馈机会。建议当结论模糊时，**延迟 1 天**，补充 3-5 条高优先级 query 的人工评测后再定，避免“赌徒式上线”。
