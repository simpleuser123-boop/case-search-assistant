**2. 溢出风险对抗检查**

**对抗问题1：最不靠谱的Story攻击——E1-F2-T1（映射表编写，1.5h/15组）**
- *工期过于乐观*：即使压缩至15组，法律术语映射要求验证“违约金条款”与“违约责任”、“解除合同”与“合同解除”等在裁判文书中的实际表述差异。按5-10分钟/组核算（含查阅文书片段），实际需1.25-2.5h，1.5h处于乐观下沿。
- *前置依赖被忽略*：映射表质量完全依赖全栈工程师的法律领域理解，若缺乏法律背景，易产生“合理但错误”的映射（如将“定金”映射至“预付款”），错误会在评测阶段才暴露，引发隐性返工。
- *验收标准模糊*：“15组映射对”仅是数量验收，质量验收后置到E1-F4-T2。若评测发现映射错误，修复+重新评测的时间未预留，直接威胁上线决策。
- *结论*：该任务溢出概率高，应提前准备15组预设兜底映射对作为Plan B，并设定“映射表编写耗时上限1.5h，超出则直接使用预设映射对”的硬止损。

**对抗问题2：隐性工作量最大的Story**
- **E1-F5-T1（集成联调，1.5h）**：2h并非充裕，即使砍掉前端，后端规则联调仍包含：环境变量对齐、向量数据库查询参数调整、规则生效顺序调试、与现有检索逻辑的兼容性验证。若规则中有潜在逻辑冲突（如案由加权与映射表同时命中时的叠加方式），定位与修复极易额外消耗1h。
- **E1-F4-T2（评测执行，1.5h）**：单轮评测时间勉强够用，但评测后发现NDCG反降或Precision无改善的概率不低。此时需要分析per-query原因、微调权重，再跑评测。1.5h仅够“出报告”，不含任何迭代优化。
- **E1-F5-T2（灰度上线，2h）**：包含上线部署、灰度流量配置、监控面板配置、数据回收脚本验证。若出现上线后异常（如接口超时），排查和回滚决策将大量消耗PM角色带宽，挤占测试工作时间。

**对抗问题3：紧急需求插入的挤出效应**
若第2天内测律师反馈“漏掉重要判例”，触发“查全率提示”紧急需求：
- *被挤出的Story候选*：E1-F2-T3（段落匹配规则）首当其冲被牺牲，因为它属于Stretch Goal且非上线必要条件。其次是E1-F4-T2评测的精细度——可能被迫从10条样本压缩至5条、放弃NDCG只做Precision@5快速判断。
- *Sprint Goal可否达成*：砍掉T3后，Goal仍可通过“映射表+案由加权”上线达成，但精准度提升天花板降低。若评测也被挤压，可能只能上线“有希望改善但缺乏量化信心”的规则，增加无效上线的风险。

**3. 多角色Sprint计划校验**

- **PM视角**：计划在第1天上午做出Go/No-Go决策，确保第2天上午可并行启动映射表与评测样本构建，节奏紧凑但合理。第3天下午上线决策依赖E1-F4-T2评测结论，若结论为“部分改善部分恶化”，上线依据将模糊。建议预设决策矩阵：如NDCG@10提升≥5%且Precision@5不下降则上线；否则仅保留日志回收机制，不启用新规则，避免负向影响。里程碑层面，本次Sprint即使未上线，也需达成“建立评测基线与日志回收管道”的最低目标。

- **研发视角（全栈工程师）**：第1天后端→测试→PM高密切换；第2天上午若做段落匹配为后端高强度编码，下午评审会后立即切为测试角色跑评测，认知负载高；第3天上午联调又切回后端，下午上线决策需PM角色清醒判断。风险点在于第3天下午的灰度上线决策——经过连续3天多角色切换，疲劳可能导致对上线信号的敏感度下降（如忽视轻微报错、误判数据回收有效性）。建议第3天上午联调提前完成，下午预留0.5h单纯作为PM决策缓冲，不进行编码。

- **测试视角**：评测时间1.5h为理论单轮，未含返工。评测发现需要修复映射表或调整权重时，修复（0.5-1h）+ 重新评测（1.5h）会导致第3天上午联调严重延迟，可能挤走上线。已通过“限定映射表编写时间上限1.5h、评测仅单轮”的止损策略降低此风险。样本量压缩至10条在初期可接受，但需确保10条query能覆盖买卖合同和侵权纠纷的核心场景，否则评测结论置信度不足。

**4. 缓冲策略**

**Must Have（无论如何必须完成，约11.5h）**：
1. 日志收集与Go/No-Go决策（2.5h）
2. 法律术语映射表（15组）+ 案由加权规则（3h）
3. 评测样本构建10条 + 评测执行单轮（2.5h）
4. 集成联调（后端规则）（1.5h）
5. 灰度上线 + 数据回收（2h）

**Stretch Goal（容量允许时顺序执行，最多+2h）**：
1. 段落匹配加权规则（2h） → 若启动，联调范围同步包含此规则
2. 评测样本扩充至20条（+0.5h） → 在段落匹配完成后且评测仍未开始时方可追加
3. 前端展示改造+埋点（3.5h） → **本次Sprint明确不做**，即使有容量剩余也不启动，避免成就感驱动挤压核心任务

**降级方案（按触发顺序）**：
1. **砍范围**（实时触发）：若第1天E1-F2-T1映射表编写耗时接近1.5h仍仅完成<10组，立即切换为15组预设兜底映射对；评测样本在准备时若耗时超1h，直接截断至5条核心query。
2. **砍段落匹配**（第1天结束时触发）：若第1天Must Have未全部完成，直接取消第2天的Stretch Goal E1-F2-T3，腾出容量保证评测与联调。
3. **评测止损**（第2天评测时触发）：若首轮评测NDCG@10为负或Precision@5无改善，不再尝试修复，保留规则代码但不上线，Sprint目标切换为“建立评测基线与日志回收机制”。
4. **不上线**（第3天下午触发）：若联调未完成或评测结论为“无显著改善”，放弃灰度上线，将本次Sprint产出定义为“已验证的技术方案+评测基线+日志管道”，纳入下一迭代。

**Stretch Goal优先级陷阱警告**：E1-F2-T3（段落匹配）是Stretch Goal但技术深度高，容易被错误赋予“核心优化”的心理权重而挤占评测时间。执行顺序强制为：先保证评测执行完整，再考虑段落匹配；段落匹配未完成不应阻碍上线，其效果可在下一轮迭代验证。
