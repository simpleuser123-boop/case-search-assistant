# 竞争假设实验

### 步骤A：假设分解

团队假设：“语义理解能力不足是根因，引入 RAG + 语义搜索能显著提升检索效率和准确性”

拆分为以下可独立检验的子假设：

| 子假设 | 内容                                                         | 可检验性                                               |
| ------ | ------------------------------------------------------------ | ------------------------------------------------------ |
| H1     | 律师在检索类案时，确实习惯用自然语言描述案情（而非提炼关键词），但现有工具只支持关键词匹配，导致必须反复转换表达方式 | 高 —— 可直接观察用户检索行为或回顾搜索日志             |
| H2     | 律师知道正确的法律关键词（如“未按约履行”），但手动输入这些词耗时费力，且容易遗漏同义表述，因此希望用自然语言一次性命中 | 中 —— 需区分“懒得试”还是“不知道怎么转译”【置信度：中】 |
| H3     | 律师不知道如何将生活化案情转译为裁判文书中的规范法律语言，语义鸿沟是认知层面的障碍，而非工具交互问题 | 高 —— 可测试用户对同一案情写出关键词的能力             |
| H4     | 检索结果在技术上是相关的（即关键词命中较多），但律师面对一堆案例时，难以快速判断哪个案例在事实/争议焦点上最相似，导致筛选耗时 | 高 —— 可通过用户对结果排序或归因分析检验               |
| H5     | 律师需要的不是“更多案例”，而是“最相关的3-5个高价值案例”，现有工具返回的结果数量过多且排序不佳，迫使人工二次筛选 | 高 —— 观察用户使用的案例数量及放弃率                   |
| H6     | 语义理解提升后，律师的检索效率会显著提升（如单次检索时间从20分钟降到5分钟以内） | 高 —— 可通过对照实验直接测量                           |

> 注：可检验性均标记为“高”或“中”，因为大部分可通过行为数据或简单访谈验证。

### 步骤B：竞争假设生成

如果“语义理解不足”不是根因，以下竞争假设值得考虑：

#### 竞争假设 C1：律师的痛点不是“查不到相关案例”，而是“查到的案例权威性不足或效力不确定”
- **内容**：律师需要的是来自上级法院、本院近期、或法官有明确倾向的案例。现有工具（如无讼）能返回语义相关的案例，但缺乏对案例“参考价值”（如审判层级、法官知名度、是否被后续判决推翻）的标注和排序。
- **间接证据**：律师常用筛选“法院层级”“审级”“判决年份”等功能，且频繁剔除基层法院案例。【置信度：中】
- **证明成立的数据**：用户访谈中，超过60%的律师将“权威性/效力”列为比“语义相关”更重要的维度；或A/B测试显示，在语义搜索基础上增加权威性排序后，用户留存率显著上升。
- **证明不成立的数据**：用户在使用语义搜索后，即使结果权威性分布不变，其检索时间仍大幅下降且满意度提高。

#### 竞争假设 C2：律师的检索效率瓶颈不在“找到案例”，而在“从案例中提取有用信息”
- **内容**：找到3-5个相关案例后，律师需要阅读全文，对比事实、争议焦点、判决理由、法律适用等，这个阅读和对比过程平均耗时30-60分钟，远高于检索本身。语义搜索只是加快了“找”，但没解决“读”和“比”。
- **间接证据**：现有工具如Alpha提供“案例摘要”“裁判要旨”但律师仍会读原文；律师经常制作类案对比表。【置信度：低→中】
- **证明成立的数据**：通过用户日志发现，用户从点击搜索结果到停止浏览的平均时长很长（如>10分钟），且阅读模式多为逐篇细读而非快速扫描；或访谈中多数律师表示“找案例很快，但分析案例很慢”。
- **证明不成立的数据**：语义搜索后，用户从点击到关闭案例详情页的平均时间显著缩短，且对“信息提取效率”评分高。

#### 竞争假设 C3：律师真正需要的是“基于类案的预测和论证支持”，而非单纯检索结果
- **内容**：律师检索类案的最终目的是预判本案胜率、说服法官或撰写代理词。现有工具只返回案例列表，但律师需要的是“类似事实下，法院的判决倾向是什么”“哪些要素对判决结果有决定性影响”。语义搜索不能直接生成这类洞察。
- **间接证据**：律师常在检索后手动统计“支持/驳回比例”，或询问“这个案子在XX法院会怎么判”。【置信度：中】
- **证明成立的数据**：用户对“判决倾向分析”功能（如团队假设中提到的）使用率极高，且表示“帮我省了统计时间”；或对比测试中，提供预测分析的结果比纯语义搜索更能提升用户决策信心。
- **证明不成立的数据**：用户仍主要查看原始案例全文，很少使用分析看板，且认为“机器分析不可靠”。

### 步骤C：假设优先级排序

按“如果为真，对产品决策的影响程度 × 当前证据缺乏程度”排序。影响程度高指该假设若成立，会显著改变产品方向（如需要做语义搜索还是做别的）。证据缺乏程度高指目前团队对该假设是否成立几乎没有数据。

| 排名 | 假设                                                   | 影响程度                                                     | 证据缺乏程度                                               | 综合优先级 |
| ---- | ------------------------------------------------------ | ------------------------------------------------------------ | ---------------------------------------------------------- | ---------- |
| 1    | **竞争假设 C2**：效率瓶颈在“读和比”而非“找”            | 高 —— 若成立，产品应优先做案例摘要、争议焦点抽取、类案对比表，而不是花大力气优化语义检索 | 高 —— 团队假设默认“找”是痛点，未验证后续阅读耗时           | ★★★★★      |
| 2    | **竞争假设 C1**：痛点在于权威性/效力不足               | 高 —— 若成立，语义搜索需叠加裁判层级、审级、引用频次等排序权重，甚至需要引入案例效力图谱 | 中 —— 常见筛选功能存在，但不确定是否为核心痛点             | ★★★★☆      |
| 3    | **团队子假设 H5**：用户需要“最相关的3-5个”而非大量案例 | 中 —— 若成立，语义搜索需大幅优化结果排序和去重，而不是单纯提升召回率 | 中 —— 常见抱怨“结果太多”，但不确定用户是否真会只读3-5个    | ★★★☆☆      |
| 4    | 团队子假设 H1/H3 (自然语言输入需求)                    | 中 —— 若成立，需设计自然语言输入框；但若C2/C1更真，优先级应后移 | 低 —— 团队已有定性观察，证据较充分                         | ★★☆☆☆      |
| 5    | 竞争假设 C3                                            | 中 —— 若成立，需要做分析预测模型，但产品MVP已包含“判决倾向分析”，部分覆盖 | 低 —— 用户对类似功能有明确需求（如聚法案例的“裁判大数据”） | ★★☆☆☆      |

> **Top3 假设**：
> 1. **竞争假设 C2**：瓶颈在于从案例中提取有用信息（阅读+对比）
> 2. **竞争假设 C1**：瓶颈在于案例权威性/效力评估
> 3. **团队子假设 H5**：用户需要的是少量高相关案例，而非大量候选

### 步骤D：调研设计建议

针对 Top3 假设，分别设计验证方案：

| 假设                           | 核心研究问题                                                 | 访谈用户类型与最低人数                                       | “验证通过”的操作性定义                                       |
| ------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **C2：检索后阅读对比耗时更长** | “律师在找到一个相关案例后，具体花多少时间阅读、对比、提取信息？过程中最耗时的操作是什么？” | 诉讼律师（民商/刑事方向），3-5年经验，每周检索类案≥5次。**最低5人**（饱和时可增至8人） | 在用户使用现有工具（如无讼）完成一次真实检索任务后，计时发现：<br>- 检索耗时 < 5分钟，但阅读+对比耗时 ≥ 15分钟<br>- 且用户明确表示“找案例不难，难的是看完后知道怎么用”<br>【达到3/5人即验证通过】 |
| **C1：权威性/效力是关键**      | “用户如何判断一个案例的参考价值？哪些元数据（法院层级、审级、判决年份、法官、是否被引用）比其他因素（如语义相似度）更重要？” | 诉讼律师（尤其资深律师/合伙人），需参与过上诉或类案检索策略制定。**最低5人** | 在卡片排序测试中，用户对“案例排序依据”将“权威性相关属性”排在前2位的比例 > 60%；<br>或访谈中 ≥4人主动提及“效力不确定导致我不敢用”作为首要痛点 |
| **H5：需要少量高相关案例**     | “用户在一次典型检索中，真正仔细阅读的案例数量是多少？过多数量的结果会引发什么负面行为？” | 诉讼律师（不限经验），日常使用类案检索。**最低6人**（观察+访谈） | 回顾最近3次检索行为（可通过日志或回顾性访谈）：<br>- 用户打开案例详情页的数量中位数 ≤ 5；<br>- 且超过80%的用户表示“返回前20条之后的基本不看”；<br>- 且对“只给我最相关的3条”这一功能表示强烈兴趣（Likert 5点中≥4分） |
