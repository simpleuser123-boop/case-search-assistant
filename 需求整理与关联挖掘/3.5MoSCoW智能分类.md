## Must 对抗性挑战

### Must #2：检索结果高精准度

- **反对者挑战**：“精准度提升是一个无底洞，需要持续的数据标注、模型迭代，3 天根本做不出可感知的提升。而且，如果没有同时优化自然语言理解，排序模型能做的只是在错误的理解上排出‘更相关’的错误结果，用户依然找不到，留存率不会变。这条需求根本不是 Must，而是一个需要长期投入的史诗，不应该放在一个为期 3 天的迭代里当 Must。”

- **我的回应**：“你说得对，完整解决精准度确实需要长期投入。但本次迭代的 Must 定义不是‘完美解决精准度’，而是‘做出可感知的精准度提升，使部分用户完成一次成功检索并留存’。我们并不需要从头训模型。策略是聚焦**查询改写和排序权重调整**——基于上一版本积累的失败查询日志，做几项高杠杆规则优化：比如法律术语映射、案由字段加权、裁判文书关键段落匹配。这些改动可以在 3 天内完成并上线 A/B 测试。即使 NLU 不完美，一个更聪明的排序能把‘碰巧蒙对’的结果提上来，让一部分用户第一次搜到想要的东西。留存率从 0% 到 20% 并不要求所有人都留下，只要比现在好就行。这条 Must 瞄准的是那个‘可实现的提升’。”

- **如果做错的最快发现方式**：“上线后立即看埋点数据——关键行为‘搜索结果点击率’和‘二次搜索率’。如果点击率没有 15% 以上的相对提升，或二次搜索率居高不下，说明新排序未达预期。第一天就能出结论，直接回滚止损。”

- **真的没有替代方案吗？** “没有。如果这一迭代完全不碰排序，只做知识辅助或 UI 改动，留存率不可能从 0 跳到 20%。这个是唯一能直接作用于核心痛点的杠杆。”

## 自检

指出本次 MoSCoW 分类中最可能出错的 3 个判断：

1. **哪条 Must 最可能在迭代中途被发现其实可以不做的？**  
   → **检索结果高精准度**。如果实际动手后发现，现有日志质量极差，无法支撑有效的规则优化，或者一个简单的规则调整只能带来 2~3% 的点击率提升，远不足以撬动留存，团队可能会意识到，单靠排序规则无法达成目标，必须联手自然语言理解或语料清洗才能见效。这时 Must 就可能“空心化”，执行中感觉在做无用功。

2. **哪条 Won't 最可能在迭代末期因为外部压力被紧急升级？**  
   → **查全率与风险提示**。法律从业者对“漏案”极其敏感，哪怕精准度尚可，如果内测用户或早期客户发现某次重要检索遗漏了关键案例，可能会引发信任危机，并施压要求立刻加上风险提示。管理层可能迫于客户压力，在迭代最后两天要求紧急插入此需求。

3. **哪条 Should 实际比某些 Must 对目标贡献更大，但被低估了？**  
   → **自然语言案情检索**。虽然被标为 Should，但如果精准度优化主要作用于排序，而自然语言理解本身错误率很高（比如把“离婚”识别为“合同纠纷”），那么排序再高明也无济于事。换句话说，提升 NLU 本身就是精准度提升中最关键的子集。如果本次 Must 的精准度工作不包含查询理解改进，那么这条 Should 很可能是被低估的真 Must。
