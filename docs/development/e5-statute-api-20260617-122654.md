## 端点契约

| 方法 | 路径 | 入参（extra=forbid 白名单） | 出参 |
|---|---|---|---|
| POST | `/api/statute/search` | SearchProfile 白名单五字段 + `mode`/`limit` | `StatuteRef[]`（带 `statute_anchors[text_id]`，无裁判正文） |
| POST | `/api/statute/by-case` | `case_id`（必填）+ `mode`/`limit` | 关联 `StatuteRef[]`（类案→法条互跳） |
| POST | `/api/statute/cases-by-statute` | `statute_id`（必填）+ `mode`/`limit` | `CandidateRef[]`（法条→类案互跳，白名单七字段 + `source_anchors`，无正文） |

请求体 `extra=forbid`：任何裁判正文 / PII / 原始案情 / 模型生成条文型键在 pydantic 层即 422（第一道闸）；`service` 层经 E4-2 `sanitize_intake_profile_payload` fail-closed（第二道闸，转 400 `STATUTE_PROFILE_REJECTED`）。

## 消费 E5-3 方式

`StatuteQueryService` 经构造函数注入默认 `app.kernel.rag.StatuteSearchService`，三入口分别调用 `search_statutes` / `statutes_by_case` / `cases_by_statute`。`/search` 在装配 `SearchProfile` 前先过 `app.kernel.guardrails.sanitize_intake_profile_payload`（E4-2 防御层）。不直连 `retrieval/rerank/summary/query_processing`，不深引内核私有，不 import 任何其它产品包（AST 扫描 0 命中）。

## 互跳行为

互跳只走契约对象，两侧均无对侧裁判正文：
- 类案→法条（`/by-case`）：出 `StatuteRef[]`，条文只来自语料、带 `text_id` 锚点；缺锚点经内核 `sanitize_statute_ref` fail-closed 丢弃。
- 法条→类案（`/cases-by-statute`）：出 `CandidateRef[]`，复用 E-1 白名单七字段 + `source_anchors`（每条至少 `case_id` + `source_chunk_id`）。

## 降级行为

`ENABLE_STATUTE_SEARCH=false`（默认）：三端点均返回 403 `STATUTE_SEARCH_DISABLED`，不检索 / 不落库 / 不回显查询，与 intake `INTAKE_DISABLED` 同款语义。
