## 1. 做了什么

新建 E 系列第三个产品能力包 `app/drafting/`，并接入受 `ENABLE_DRAFTING` 门控的 4 个端点，
把检索/清单沉淀的 `CandidateRef`（类案）+ 可选 `StatuteRef`（法条，经 E5 互跳）**组装**为
`DraftDescriptor`（结构骨架=段落标题 + 锚定引用 + 用户短字段），并持久化（只存元数据/引用/短字段）。
**只组装、不起草**：service 绝不生成任何段落正文/结论/胜负判断/诉讼结果预测。

## 3. 端点契约

| 方法 | 路径 | 鉴权 | 门控关闭 | 行为 |
| --- | --- | --- | --- | --- |
| POST | `/api/drafting/drafts` | 需登录 | 403 `DRAFTING_DISABLED` | 组装+持久化，出已收敛 `DraftDescriptorView`（零正文） |
| GET | `/api/drafting/drafts` | 需登录 | 403 | 列出当前租户可见草稿（零正文） |
| GET | `/api/drafting/drafts/{draft_id}` | 需登录 | 403 | 读取单个；越权/不存在 → 404 `DRAFT_NOT_FOUND` |
| PUT | `/api/drafting/drafts/{draft_id}` | 需登录 | 403 | 更新骨架/引用/短字段（仍只存元数据）；非 owner → 404 |

错误码：`DRAFTING_DISABLED`(403) / `DRAFTING_REQUIRES_LOGIN`(401) / `DRAFT_REJECTED`(400,
sanitize fail-closed) / `DRAFT_NOT_FOUND`(404) / `VALIDATION_ERROR`(422, pydantic extra=forbid)。

## 4. 持久层列清单（证明无正文列）

`draft_descriptor` 表共 **13 列**，全部为元数据/引用/结构骨架(标题)/短字段/结构化关系：

```
draft_id, owner_user_id, team_id, visibility,
structure_skeleton(JSON,标题清单), candidate_refs(JSON,带锚点), statute_refs(JSON,带锚点),
note, tag, status, reason_code, created_at, updated_at
```

**无任何正文/胜负/凭据列**：不含 `draft_body / draft_text / generated_text / paragraph_body /
conclusion / full_text / chunk_text / judgment_text / summary_text / raw_case / raw_query /
win_probability / verdict / password / token`。`test_persisted_row_has_no_body_columns` 对列集
做精确相等断言 + 禁列空集断言。`structure_skeleton` 虽是 JSON 文本列，但只承载段落标题清单
（每项 ≤ 60 字，由 E6-1 契约校验，超长即视为正文 fail-closed）。

## 5. 组装而非起草的实现方式

- `DraftingService.assemble_draft(payload)` 直接复用 E6-1 内核纯函数 `sanitize_draft_descriptor`：
  - `structure_skeleton` 每项做标题校验（非空 + ≤ 60 字），超长（疑似正文）fail-closed 抛错；
  - `candidate_refs` / `statute_refs` 逐项收敛，**缺锚点引用 fail-closed 丢弃**（保留项 100% 有锚点）；
  - 起草正文型 / 裁判正文型 / PII 型 / 胜负结论型键 fail-closed 抛 `ContractViolationError`。
- service / store / router 源码**零** LLM / 模型 / 检索客户端 import，零文本生成调用入口
  （`test_drafting_service_does_not_call_text_generation` 标记扫描 + `test_drafting_*` AST 双证）。
- 「如需重新取锚点展示内容经 app.kernel.rag、不落库」：本步不触发该路径（端点入参已自带引用与锚点），
  DraftDescriptor 持久层只存引用与锚点，不存取回的展示正文。

## 6. 鉴权 / 租户隔离 / 降级

- **默认 `visibility=private`**；所有端点需登录（复用 M5-2 会话，`ENABLE_ACCOUNT_SYSTEM` 关时会话无效）。
- 读取仅 `list_visible` / `get_visible` 两入口，均强制 `_tenant_clause(ctx)`（行级隔离，与
  `SedimentationObject` 同口径）：单用户私有态只见自己私有行；团队态见「自己私有行 + 本团队
  `visibility=team` 行」；跨用户/跨团队在 SQL 层即排除，**无无过滤读取的对外路径**。
- 写/更新经 `_assert_write_within_tenant` + owner 校验；`update_owned` 非 owner 取不到 → None → 404
  （不泄露他人草稿存在性差异）。
- E6-2 默认单用户私有态（`team_id=None`），不放权客户端写 `visibility/team_id`（团队放权留待后续步骤）。
- 关闭 `ENABLE_DRAFTING`（默认）时全部端点 403 `DRAFTING_DISABLED` 安全降级，响应不泄露内部信息。
- **日志脱敏**：只写 `user_id_hash / draft_id_hash / *_count / note_meta(len+hash) / reason_code`；
  绝不写起草正文/裁判正文/原始案情/note 全文（`test_note_not_logged_in_full` 守门）。

## 7. 边界（产品包纪律）

- `app/drafting/*.py` 对内核的依赖仅 `app.kernel.guardrails` / `app.kernel.identity` **顶层公开面**；
  基础设施依赖 `app.core.db/config/logging` + `app.api.errors` + `app.schemas`（既有范式，同 sharing/statute）。
