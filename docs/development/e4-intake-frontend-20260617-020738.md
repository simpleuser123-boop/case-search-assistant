## 1. 实现概述

录入端前端页落地「原始案情 → 本地脱敏预览 → 用户确认 → 仅发 SearchProfile → 展示 CandidateRef」闭环。原始口语化案情只存在于浏览器 React 内存态，绝不写入任何浏览器存储、不进入任何网络请求体或 URL query。入口与路由严格受 `VITE_ENABLE_INTAKE` 门控，默认 `false` 时不渲染。

三阶段流程：

1. 本地输入：原始案情文本框 + 结构化要素辅助输入（案由/地域/审级倾向/争议焦点关键词，均可选）。
2. 脱敏预览：调用 E4-2 前端 `sanitize.ts` 的 `buildSearchProfileFromRaw`，把原始案情转为 SearchProfile 白名单五字段，并展示「将要发送的脱敏后内容」。用户结构化辅助输入也逐项再脱敏后合并。
3. 用户显式点击「确认并仅发送脱敏内容」后，才 POST SearchProfile 到 `/api/intake/search`；展示返回的 `CandidateRef[]`（无正文，带 `source_anchors`，可跳检索助手）。

## 3. 本步完成的 E4 子目标

- 录入端前端页实现本地输入 → 本地脱敏预览 → 用户确认 → 仅发 SearchProfile → 展示 CandidateRef。
- 原始案情零上送：网络请求体只含白名单五字段（+ mode/limit），无浏览器存储写入，无 URL 携带。
- 入口/路由严格受 `VITE_ENABLE_INTAKE` 门控（三重：路由不注册 + 入口不渲染 + 页面返回 null），默认 off 不渲染。
- 无任何「服务端 AI 增强」UI（`VITE_ENABLE_INTAKE_AI_EXTRACTION` 默认 off、无 on 路径）。

## 4. 原始案情零上送的可执行保证

请求体由 `toIntakeRequestBody` **逐字段显式**取白名单键组装，而非 spread 任意对象。测试以「在 profile 对象上注入 `raw_case`/`raw_query`/`name`/`id_card`」验证：出参仅含 7 个键，序列化后不含任何注入的 PII/正文键。组件测试进一步验证：含假 PII 的原始案情经预览后，预览 DOM 与发送的 profile 均不含 `张三`/`李四`/`13800138000`/住址等。
