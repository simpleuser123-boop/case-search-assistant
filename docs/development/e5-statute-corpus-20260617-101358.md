## 2. 本步完成的 E5 子目标

1. **法条语料规整**：每条法条记录含 `statute_id / law_name / article_no / text_id / article_text / source_corpus / effective_status / coverage_domain`，与案件 `cases.jsonl/chunks.jsonl` 物理隔离（独立文件名），不覆盖、不修改案件产物。
2. **法条索引构建**：独立 Chroma collection `statute_chunks_bge_m3_v1`（≠ 案件 `case_chunks_bge_m3_v1`）+ 独立 persist 目录（`CHROMA_STATUTE_PERSIST_DIR` 或 `<case_dir>_statute`，硬断言永不等于案件目录）；脚本幂等、可重跑、参数化，未引入对外业务 flag。
3. **类案→法条关联标注**：复用既有 `law_articles`（源自 JuDGE「Law Articles」）建立 `case_id → 法条 text_id/statute_id` 映射，作为 E5-3 互跳数据底座；**只读**案件产物。
4. **覆盖范围如实标注**：刑事覆盖（《中华人民共和国刑法》182 条）；民事/行政为已知缺口，与 M5-7 路线 B 扩语料同依赖，本期不强求扩充。

## 4. 种子缺位与 catalog 模式说明

工作区**未放置** JuDGE `law_corpus.jsonl` 种子（仅文档引用），且无「案件→法条」qrels 文件。为不杜撰条文（E5 红线）同时交付可用产物：

- `build_statute_corpus.py` 实现 **seed** 与 **catalog** 双模式，二者用同一 `derive_statute_id` 口径，可平滑合流。
- 本步以 **catalog 模式**运行：从 `cases.jsonl` 的 `law_articles` 派生法条目录，每条带 `text_id` 锚点（catalog 命名空间 `cncl::...`），`article_text=null`、`effective_status=unverified`。**绝不杜撰条文**。
- **seed 模式已实现并被单测覆盖**（条文原样保留、缺 text_id 丢弃、chunk_type=`statute_article`）。一旦 JuDGE `law_corpus.jsonl` 到位，运行 `--mode seed --seed data/raw/law_corpus.jsonl` 即可补齐权威 `article_text`，`statute_id` 不变、关联映射无需重建。

