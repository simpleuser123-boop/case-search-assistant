## 1. 做了什么

新增受 `VITE_ENABLE_DRAFTING` 门控的文书工作台前端页（对标 E4-4 录入页 / E5-5 法条页 gated 范式）：

草稿态仅 React 内存 + 后端持久化，禁用任何浏览器存储，引用零正文，草稿/引用不入 URL。

## 3. flag 三重门控方式

`VITE_ENABLE_DRAFTING=false`（默认）时：

1. **路由不注册**：`router.tsx` 中 `if (isDraftingEnabled())` 为假，`/drafting` 不进 routes。
2. **入口不渲染**：`HomePage` / `SearchPage` 的 `draftingEnabled ? <Link/> : null` 渲染 null，DOM 无可达入口。
3. **页面组件返回 null**：`DraftingPage` 顶部 `if (!isDraftingEnabled()) return null;`。

`isDraftingEnabled()` 直接读 `VITE_ENABLE_DRAFTING`，**不走** `readM1M5UiFlag`，与
`VITE_ENABLE_M1_M5_ACCEPTANCE` / `VITE_ENABLE_INTAKE` / `VITE_ENABLE_STATUTE_SEARCH` 正交，
互不联动（featureFlags.test.ts 4 条守门断言）。

## 4. 组装而非起草的 UI 设计

- **结构骨架**：纯段落标题增删改（`SkeletonEditor`），单项 `maxLength=60`、超长提示「应为标题非正文」、
  最多 64 项；**无**「自动起草段落正文」按钮/能力。
- **引用区**：`CandidateRefSection`/`StatuteRefSection` 让用户填 `case_id`+`source_chunk_id`（类案）/
  `statute_id`+`law_name`+`text_id`（法条）构造**带锚点**引用；无锚点引用前端拦截（`hasCandidateAnchor`/
  `hasStatuteAnchor`），只展示元数据 + 锚点出处，`article_text` 恒为 null（前端不生成条文）。
- **短字段**：`note`（≤200）/`tag`（≤40）输入。
- **保存/列表**：`SaveBar` 仅「保存草稿/更新草稿/新建空白草稿」；`SavedDrafts` 列出本人/团队可见草稿（计数 + 标签，零正文）。
- **无任何**起草/生成/预测入口：`DraftingPage.test.tsx` 断言 DOM 无「自动起草/一键起草/AI 起草/生成段落/
  生成正文/生成结论/自动预测/预测胜负」文案，且无按钮文本匹配 `/起草|生成|预测|胜负|胜率/`。

## 5. 引用零正文证明

- `toDraftRequestBody` 顶层只逐字段取 5 键（`structure_skeleton`/`candidate_refs`/`statute_refs`/`note`/`tag`），
  **绝不 spread**；注入 `draft_body`/`generated_text`/`conclusion`/`win_probability`/`verdict`/`raw_case`/
  `password` 等键时，序列化结果 0 命中（draftingApi.test.ts 断言）。
- `candidate_refs` 逐项只取 CandidateRef 白名单七字段 + 锚点；注入 `summary`/`highlight`/`chunk_text`/
  `judgment_text` 等正文键被丢弃。
- `statute_refs` 只取白名单字段 + 锚点；`structure_skeleton` 只发标题字符串数组（去空白/裁项数/裁长度）。
- 无锚点引用在组装阶段即被丢弃（`hasCandidateAnchor`/`hasStatuteAnchor` 过滤）。
