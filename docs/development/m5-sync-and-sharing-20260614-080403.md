## 1. 目标与基调

把 M4 的本地沉淀（历史 / 收藏 / 清单 / 报告）同步到服务端按 owner 归属的存储，并支持团队内显式共享。共享是跨用户数据可见性的引入点、最高隐私风险，因此全程遵循：默认 private、默认不共享、显式授权才可见；同步只上送元数据 / 引用 / 用户自填短字段，绝不上送正文或原始案情；`ENABLE_TEAM_SHARING` 默认 false，关闭即回到 M4 本地沉淀末态。

## 2. 数据结构

### 2.1 共享账本 `m5_shared_object`（新增）

显式共享动作的账本，用于留痕、审计与撤销。白名单字段：`share_id / object_id / object_type / owner_user_id / shared_with_team_id / visibility / anchor_count / status / reason_code / created_at / updated_at`。无任何正文列；`anchor_count` 仅整数（校验通过的锚点条数），用于审计「共享内容是否可溯源」，不含锚点内容。

**关键设计**：账本不参与读取放权。真正的可见性仍由 M5-3 的 `SedimentationObject.visibility + team_id` 承载，并经唯一过滤点 `tenant_visibility_clause` 强制隔离。这样可见性判定不出现「第二条路径」，跨团队 / 跨用户隔离不被共享破坏。

### 2.2 复用 `m5_sedimentation_object`（M5-3 已有）

复用其 `owner_user_id / team_id / visibility / source_anchors` 槽位承载可见性与来源锚点。M5-5 不新增任何正文列。

## 3. 同步规则

入口 `POST /api/sharing/sync`：只接受元数据 / 引用 / 来源锚点 / 用户自填短字段，**不接受 team_id / visibility**（共享是另一个显式动作）。同步永远以单用户私有上下文写入（team_id=None / visibility=private）。正文 / 凭据 / 未知键被双重拦截：schema `extra=forbid`（422）+ 复用 M5-3 store 的白名单 / 禁用键清洗（forbidden_field）。同步由用户显式触发、可清除。

## 4. 共享规则

共享必须显式动作（`POST /api/sharing/share`），默认 private。只有「对象 owner」且「目标团队活跃成员」可共享；非 owner / 非成员一律被拒。案例侧 AI 内容承载型对象（`case_list` / `report_template`）共享前强制校验来源锚点：必须至少有一条合法锚点（同时带非空 `case_id` 与 `source_chunk_id`），任何出现的锚点都必须合法，否则拒绝（`missing_source_anchor` / `invalid_source_anchor`）。纯引用型（`case_favorite`）可不带锚点，但若带则必须合法。

共享通过后把对象 `visibility` 升为 `team`、写入 `shared_with_team_id`（经 team_id 承载），可见性随即由 `tenant_visibility_clause` 唯一承载，团队成员据此可见。取消共享（`POST /api/sharing/unshare`）由 owner 把对象降回私有。共享后受 M5-4 权限模型约束（viewer 只读）。

## 5. 降级与兼容

后端 `ENABLE_TEAM_SHARING`、前端 `VITE_ENABLE_TEAM_SHARING` 均默认 false。关闭态：所有端点返回 403 `TEAM_SHARING_DISABLED`、懒初始化零副作用不建表，前端 `SharingPanel` 渲染 null，回到 M4 本地沉淀末态（单用户、纯前端、不上送服务端）。同步 / 共享失败给安全提示（reason_code），不影响主链路与本地沉淀。
