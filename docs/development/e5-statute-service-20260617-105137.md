## 2. 服务入口与互跳实现

| 入口 | 入参 | 出参 | 互跳数据底座 |
| --- | --- | --- | --- |
| `search_statutes` | `SearchProfile` / `str` | `StatuteSearchResult(statute_refs=StatuteRef[])` | 法条索引/语料（默认词法匹配，确定性） |
| `statutes_by_case` | `CandidateRef` / `case_id` | `StatuteSearchResult(statute_refs=StatuteRef[])` | E5-2 `case_statute_links.jsonl`（类案→法条） |
| `cases_by_statute` | `StatuteRef` / `statute_id` | `StatuteCaseRefResult(candidate_refs=CandidateRef[])` | E5-2 关联标注反向（法条→类案）+ `chunks.jsonl` 取代表性 `source_chunk_id` 锚点 |

互跳只走契约对象：类案→法条出 `StatuteRef`（经 `sanitize_statute_ref` 收敛白名单 + `text_id` 锚点），法条→类案出 `CandidateRef`（经 `sanitize_candidate_ref` 收敛 E-1 白名单七字段 + `source_anchors`）；**两侧均不携带对侧正文**。

## 3. 本步完成的 E5 子目标

1. **内核法条检索能力**：在 `app.kernel.rag` 新增 `StatuteSearchService`，对标 E3 `InternalSearchService`（依赖构造注入、纯转换、零正文泄露）；默认 `JsonlStatuteCorpus` 只读 E5-2 法条产物。
2. **法条条文锚定 / 不杜撰**：`article_text` 原样透传（只来自语料，catalog 模式为 `None`），服务**不生成/改写/续写**任何条文；`StatuteRef` 一律经 `sanitize_statute_ref` 收敛白名单。
3. **缺锚点 fail-closed**：法条命中缺 `text_id` → 丢弃并记 `STATUTE_REF_DROPPED_NO_ANCHOR`；互跳类案缺 `source_chunk_id` → 丢弃并记 `STATUTE_CASE_REF_DROPPED_NO_ANCHOR`。绝不展示不可溯源候选。
4. **行为对既有检索零影响**：未改 `InternalSearchService.execute/search_candidate_refs`、`/api/search`、`/api/search/expand`、intake 端点；未重建案件索引；未改排序/召回/rerank 默认。法条检索是新增并行能力，案件检索路径逐位不变。
