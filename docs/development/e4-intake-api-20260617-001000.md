## 2. 端点契约

`POST /api/intake/search`

请求体（`IntakeSearchRequest`，`extra="forbid"`）：
- `case_cause: str | None`、`region: str | None`、`trial_level_preference: str | None`、`dispute_focus_keywords: list[str]`、`query_text: str | None`（SearchProfile 白名单五字段）
- `mode: "standard" | "expanded"`（默认 standard）、`limit: int`（1–50，默认 10）——仅结构化检索参数，不入 profile 载荷
- 任何 raw_case / raw_query / name / id_card / phone / address / email / full_text / content 等 PII / 正文型键 → pydantic 层即 422（第一道闸）

响应体（`IntakeSearchResponse`，零正文）：
- `query_session_id: str`
- `candidate_refs: IntakeCandidateRefView[]`（白名单七字段：case_id / case_number / court / trial_level / case_cause / judgment_date / source_anchors）
- `candidate_count: int`、`degraded: bool`、`degraded_reasons: list[str]`、`search_mode`
- 不回传 query_text / 原始案情 / timings / coverage 明细

错误码：
- `403 INTAKE_DISABLED`（ENABLE_INTAKE=false 安全降级，E-1 关闭语义一致）
- `422 VALIDATION_ERROR`（schema extra=forbid 拒绝非白名单 / PII / 正文键）
- `400 INTAKE_PROFILE_REJECTED`（E4-2 后端防御层 fail-closed，消息只含口径说明、零 PII 回显）

## 3. 消费 E3 方式

- `IntakeSearchService` 仅依赖 `app.kernel` 公开面：`app.kernel.rag`（InternalSearchService / InternalSearchRequest / InternalSearchResult / SearchProfile）+ `app.kernel.guardrails`（sanitize_intake_profile_payload / ContractViolationError）。
- 流程：请求体白名单五字段 → E4-2 `sanitize_intake_profile_payload`（键级红线 + 值级脱敏）→ `SearchProfile` → `InternalSearchRequest` → `InternalSearchService.search_candidate_refs()` → `CandidateRef[]`。
- 不复制检索主路径、不深引内核内部、不直连 retrieval / rerank / summary / query_processing（AST 静态断言通过）。

## 4. 降级行为

- ENABLE_INTAKE=false（默认）：`403 INTAKE_DISABLED`，不检索、不落库、不泄露内部信息。
- 检索内部降级（数据源不可用 / summary 不可用等）：透传 `degraded=true` + `degraded_reasons`（结构化原因码，零正文），候选可为空。
- 锚点不完整候选：由 E3 服务 fail-closed 丢弃（CANDIDATE_REF_DROPPED_NO_ANCHOR），不暴露不可溯源候选。

## 5. 鉴权 / 租户隔离

- 与 `/api/search` 同款：检索端点为会话态（`query_session_id` 中间件注入），不要求登录令牌；`ENABLE_INTAKE`（默认 false）作为访问闸，关闭即不对外暴露。
- 未新增任何「检索端点之外」的对外端点；无未鉴权写库 / 写历史端点。
