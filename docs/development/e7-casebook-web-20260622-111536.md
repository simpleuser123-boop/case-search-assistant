## flag 三重门控

1. **路由不注册**：`router.tsx` 仅在 `isCasebookEnabled()` 为真时 `routes.push({ path: "/casebook" })`。
2. **入口不渲染**：HomePage / SearchPage 头部仅在 `casebookEnabled` 为真时渲染「协作工作台」Link。
3. **组件返回 null**：`CasebookPage` 首行 `if (!isCasebookEnabled()) return null;`，DOM 无任何可达节点。

`isCasebookEnabled()` 直读 `import.meta.env.VITE_ENABLE_CASEBOOK`（默认 false），与
`VITE_ENABLE_M1_M5_ACCEPTANCE` / `VITE_ENABLE_INTAKE` / `VITE_ENABLE_STATUTE_SEARCH` /
`VITE_ENABLE_DRAFTING` 完全正交，互不联动（featureFlags 测试覆盖默认 false + 三类正交 + 仅自身 on）。

## 归集而非起草的 UI 设计

页面只有四类输入区，无任何文本生成/起草/归纳/预测控件：

- **案情摘要区**：仅 5 个脱敏白名单子集输入框（案由/地域/审级倾向/争议焦点关键词/已脱敏检索文本），
  与后端 `SEARCH_PROFILE_FIELDS` 逐字段一致；明确提示「不要填写原始口语化案情」。
- **归集区**：手动选入带锚点 CandidateRef（case_id + source_chunk_id）与 DraftDescriptor（draft_id +
  段落标题骨架）；无锚点类案 / 无标题骨架前端拦截不可加入（与后端「无锚点丢弃」一致）。
- **短字段区**：title/note/tag，逐字段长度上限（80/200/40）。
- **保存栏 + 已保存列表**：保存/更新协作夹、新建空白、载入编辑；列表只读展示 visibility 标识
  （仅本人可见 / 团队可见），无共享切换控件（留 E7-4）。

测试断言：无 `/生成|起草|综述|归纳|预测/` 匹配的按钮文案与 placeholder（用 affirmative 短语断言，
避开免责文案中「不…起草…」否定式的误判）；无 `role="switch"`，无「切换为团队 / 设为团队可见」控件。

## 引用零正文证明

- `casebookApi` 所有请求体由白名单键**显式逐字段组装**，绝不 spread：
  - 顶层 create 体只含 6 键 `search_profile_summary / candidate_refs / draft_descriptors / title / note / tag`
    （create 永不发 visibility）；update 体同集 + 可选 visibility（仅调用方显式给出时）。
  - `search_profile_summary` 只发 5 个脱敏子集键；注入 `raw_narrative / defendant_name / phone` 时出参只剩 5 键。
  - `candidate_refs` 只发 CandidateRef 7 字段 + 锚点；注入 `summary / chunk_text / judgment_text` 时出参只剩 7 键。
  - `draft_descriptors` 只发骨架标题 + 锚定引用 + 短字段；注入 `draft_body / generated_text` 被丢弃。
  - 无锚点引用 / 无骨架标题在组装阶段即被 `filter` 丢弃。
- `CasebookPage` 渲染只展示元数据 + 锚点出处（case_id / 案号 / 法院 / 来源片段 ID / 段落标题计数 /
  visibility）；正文型 key（`judgment_text/chunk_text/article_text/draft_body/案件综述/胜诉概率`）
  仅出现在免责/否定文案，无任何实际渲染。
