## 三、现状持久层基线（关键事实）

M4-2 至 M4-6 的全部沉淀能力（历史、草稿、收藏、清单、导出、报告）都是**纯前端 `localStorage`、单浏览器、单用户、零后端持久层**。后端 `apps/api/app/core/db.py` 只有数据库引擎，无任何业务表；不存在用户身份概念、账号体系或服务端归属。

这意味着团队复用不是在既有服务端能力上加一层权限，而是一次从「无服务端用户态」到「多用户协作」的体系级跃迁，归属 M5 商业化范畴。M4-7 只做能力评估和字段位预留，不补建任何服务端基础，也不改动现有单用户行为。

## 六、对照 Gate 3 的行为证据

Gate 3（进入 M4 / 工作流扩展前置）要求检索成功率与复访率、收藏/复制/详情行为、重复检索/复用清单的证据。

- **检索成功率与复访率**：部分就绪。M1/M2 已建检索成功率与降级覆盖证据；复访率依赖真实多会话用户数据，单用户本地态尚无法直接度量。
- **收藏/复制案号/查看详情行为**：部分就绪。M3-7 复制案号 + M4-3 收藏 + 详情查看能力已具备且默认可关闭；「显著存在」需上线后脱敏埋点（`case_favorite_action`、`citation_copy_action`）累积。
- **重复检索/复用类案清单行为**：部分就绪。M4-2 历史/草稿 + M4-4 清单 + M4-5 导出 + M4-6 报告模板已提供复用载体；「明确行为证据」需真实用户数据。

总体结论：**能力载体在 M4 已全部就绪且默认安全关闭，行为证据待上线后积累**。团队复用真正落地的前置是 M5 账号/共享/权限/隔离体系。本步只评估，不补建。

## 七、M5 团队空间前置条件

- **账号**：用户实体 `user_id`、认证与会话（登录/SSO/令牌）、凭据与个人数据合规存储。
- **共享**：可见性模型 `private/team`、显式共享动作、默认 private 默认不共享。
- **权限**：团队/组织实体、角色模型 `owner/editor/viewer`、对象级鉴权中间件与审计。
- **隔离**：`team_id` 强隔离、行级/命名空间隔离、跨团队默认不可见。
- **持久层**：从纯前端 `localStorage` 升级到服务端按 `owner` 归属的持久层；迁移即复用 M4-1 持久化字段白名单（只存元数据/引用/用户自填短字段，不存正文）。

## 八、现有持久层从单用户扩展到团队的改造点与隐私风险

| 改造点 | 当前（单用户） | 目标（团队） | 隐私风险 |
| --- | --- | --- | --- |
| 存储位置 | 浏览器 localStorage（单浏览器） | 服务端按 owner/team 归属 | 从本地可清除变为服务端长期保存，须复用字段白名单 + 脱敏 + 保留周期 |
| 归属 | 隐式单用户（无 owner） | 显式 `owner_user_id` + 可选 `team_id` | 归属错误导致越权；owner 缺省必须等同私有 |
| 可见性 | 仅本人本浏览器可见 | `private`（默认）/ `team`（显式开启） | 默认值必须 private；默认 team 可见即重大事故 |
| 锚点与正文边界 | 前端 sanitize 丢弃非白名单键 | 服务端入库同样强制白名单 + 锚点校验 | 共享放大正文泄露面，须服务端再次执行 no_anchor_no_delivery 与禁正文 |
| 日志/埋点 | 脱敏 action 事件 | 增加 owner/team 维度但仍不带正文/案号/note/tag | 多用户维度可能反推个人，须保持脱敏聚合 |

## 九、最小预留（仅声明，不实现业务）

仅声明字段位与默认值，**不在 M4-7 注入到任何在用 TS 类型或服务端表**，不改变现有 sanitize / 序列化行为；物理落地留待 M5。

| 字段位 | 默认值 | 含义 | 是否改变现有行为 |
| --- | --- | --- | --- |
| `owner` | `private` | 记录归属；缺省等同当前单用户私有 | 否 |
| `visibility` | `private` | 可见范围；默认仅本人可见 | 否 |
| `sharing` | `disabled` | 共享开关；默认关闭 | 否 |
| `team_id` | `null` | 团队归属；空即单用户私有 | 否 |
| `owner_user_id` | `null` | 用户归属；空即匿名单用户 | 否 |

预留默认值与 M4-1 合同一致：`owner=private`、`visibility=private`、`sharing=disabled`。

> 之所以只「声明」而不注入在用类型：M4-1 已冻结持久化字段白名单，且收藏/清单的 `sanitize*` 反序列化逻辑会**主动丢弃非白名单键**。此时若把 `owner/visibility` 塞进现行 TS 类型，要么被 sanitize 丢弃（无意义），要么需放宽白名单（越界、可能引入回归）。因此预留停留在评估/合同层，等 M5 连同服务端持久层与账号体系一并落地，符合「最小预留、不改现有行为」。
