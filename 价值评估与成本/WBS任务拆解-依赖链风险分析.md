```
请将以下功能需求拆解为WBS任务树，并对依赖链进行风险分析。按以下推理链执行：

**推理链**：逐层拆解（Epic→Feature→Task→Sub-task）→ 依赖关系建模 → 依赖链脆弱性扫描 → 瓶颈任务识别 → 级联延迟模拟

功能：类案检索RAG助手MVP迭代 —— "检索结果高精准度"优化。基于RAG（检索增强生成）的法律类案语义检索工具，本次迭代聚焦于：通过对查询改写和排序权重进行调整，使检索结果精准度有可感知提升，让目标用户（「判例猎人王律」——执业5-8年的诉讼律师）在用自然语言描述案情时，前几条结果能命中事实情节高度类似的类案。
目标：将关键行为"搜索结果点击率"较基线提升至少15%，降低二次搜索率，推动用户留存率从当前0%提升至20%以上。验证核心假设：一个更聪明的排序能把"碰巧蒙对"的结果提上来，让至少一部分用户第一次搜到想要的东西。本次迭代不追求"完美解决精准度"，而是"做出可感知的提升"。
约束：
- 时间：1个Sprint = 3天，目标MVP上线+埋点就绪
- 人力：1名全栈工程师（共3人天，80%缓冲阈值2.4人天）。无专职后端、算法、测试。
- 技术：现有RAG系统架构——基于法律语料的语义检索引擎（嵌入模型+向量数据库+LLM重排序链路）。本次迭代**不能推翻现有检索引擎架构重来**，聚焦在查询层（Query Rewriting）和排序层（Ranking Weight Adjustment）做规则型优化。技术手段包括：法律术语映射表、案由字段加权、裁判文书"本院查明"段落匹配加权。
- 数据：基于上一版本积累的失败查询日志做驱动分析。日志可能存在质量差（query过于简略、缺乏用户行为反馈闭环）的风险。
- 业务：不能破坏现有关键词检索用户的体验。优化应为增量式、可回滚的。

**阶段一：WBS拆解**
拆解规则：
1. 层级不超过4层（Epic → Feature → Task → Sub-task）
2. 最底层任务1-3天可完成
3. 每个任务标注：预估工时+工时置信度【高|中|低】、前置依赖任务编号、交付物描述、验收标准（可测试，非"完成"这类模糊词）
4. 标注哪些任务在关键路径上

输出格式：Markdown嵌套列表树形结构 + Mermaid flowchart依赖图，所有任务使用统一编号体系（如E1-F2-T3-S1）

**阶段二：依赖链脆弱性分析**
对依赖链进行风险扫描，回答以下问题：
1. **级联延迟模拟**："如果任务[X]延迟3天，沿依赖链向下追溯，哪些任务会依次延迟？最终对项目总工期的影响是多少天？"（选择3个处于依赖链关键位置的任务分别模拟）
2. **单点瓶颈识别**：标注出那些"被≥3个任务依赖"且"没有并行替代方案"的任务——这些是项目的阿喀琉斯之踵。特别注意：本次只有1名全栈工程师，不存在"增加人手并行"的可能——所有任务本质上都是串行的，单点瓶颈的杀伤力会加倍。
3. **依赖类型分类**：将依赖标注为三种类型——FS（完成-开始，前置完成才能开始后继）/ SS（开始-开始，同时开始但进度关联）/ FF（完成-完成，需要同时完成），标注每种依赖类型的风险等级
4. **最坏情况延迟链**：从所有任务中找出"如果每个任务都按最悲观工期完成"的最长延迟链，标注这条链上的每一个任务及其悲观工期。特别说明：如果最悲观链长度超过3天可用的缓冲（因为只有1个人且总工期仅3天），建议的降级方案是什么。

**阶段三：风险任务清单**
用Markdown表格输出"高风险任务清单"：
| 任务编号 | 任务名 | 风险类型 | 风险描述 | 如果延迟，一级影响 | 如果延迟，二级影响 | 缓解方案 |
风险类型包括：依赖过多（≥3个）、工时估算置信度低、关键人依赖（仅1名全栈工程师，无备选人力）、技术不确定性高（法律术语映射准确率未知、失败日志质量未知）、需求可能变更（MoSCoW自检已指出"自然语言检索"可能是被低估的真Must——如果本次精准度工作不包含查询理解改进，可能需要临时调整范围）
```
