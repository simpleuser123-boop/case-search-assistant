# M2 可信检索分步骤文档

本文档承接 `M1.3x-10 合法候选稳健性验证与防过拟合门禁`：

- `docs/development/m1.3x-legal-candidate-robustness-20260611-170419.md`
- `docs/development/m1.3x-legal-candidate-robustness-20260611-170419.json`
- `docs/development/m1.3x-legal-candidate-robustness-parameter-version-20260611-170419.json`

最新 M1.3x-10 结论是：

| 结论项 | 状态 |
| --- | --- |
| selected candidate | `m1_3x_legal_score_shape_router_v2_candidate` |
| selected candidate robustness | `ROBUST_GO` |
| before->after `REGRESSED` | `0` |
| after-vs-baseline `REGRESSED` | `0` |
| `METRIC_REGRESSION` | `0` |
| `RECALL_MISS` | `7`，已解释为不阻断 M2 的剩余召回边界 |
| aggregate Top10 hit rate | `0.72` |
| aggregate NDCG@10 | `0.1954`，高于 baseline `0.134` |
| aggregate Precision@5 | `0.056`，高于 baseline `0.032` |
| `ENABLE_WEIGHTED_RERANK` 默认状态 | `false` |
| M2 是否已经进入 | `false` |

因此，M2 可以作为下一阶段开始执行；但 M2 不是上线放量版，也不是默认开启 weighted rerank。M2 的目标是把“能找到相似案例”推进为“用户能核验结果来源、理解数据覆盖、复核可能遗漏”，并且不能承诺绝对查全。

---

## 1. M2 定位

M2 对应 `01-演进计划总览.md` 中的 `M2 可信检索`，在 `07-开发步骤.md` 中落到 `Sprint 3：可信与查全`。

### 1.1 版本目标

解决用户看到结果后仍然不敢信的问题：

1. 每条用户可见的 AI 加工内容都能回到来源案例和来源片段。
2. 用户能看到数据来源、数据截止日期、候选数量、降级状态等覆盖信息。
3. 主结果和低置信度候选被清楚区分。
4. 扩展检索只能表达“扩大复核范围”，不能表达“已查全”。
5. 用户对相关/不相关的反馈可被脱敏记录，但不立即改变线上排序。
6. 不利案例和风险提示只作为复核线索，必须有来源锚点，不输出法律结论。

### 1.2 成功标准

M2 结束时，至少要同时满足以下条件，才允许讨论进入 M3 阅读提效：

| 指标 | 门槛 |
| --- | --- |
| 来源锚点覆盖 | 结果卡片、详情、摘要、风险提示中的 AI 加工内容 `100%` 绑定 `case_id` 和 `source_chunk_id` |
| 无来源内容处理 | 无来源锚点的 AI 内容不展示，降级为原文片段或空状态 |
| 数据覆盖声明 | 结果页可展示数据来源、数据截止日期、索引版本或候选统计、降级状态 |
| 隐私 | 用户原始案情不持久化；日志、埋点、报告只保存脱敏字段 |
| 扩展检索文案 | 不出现“已查全”“保证无遗漏”“查全率”等绝对话术 |
| 低置信度候选 | 主结果和低置信度候选在 API 字段、前端视觉和埋点中可区分 |
| 反馈事件 | 相关/不相关反馈只记录脱敏事件，不直接改当前排序 |
| 质量回归 | M1.3x-10 candidate comparison 核心门禁继续通过 |
| 性能 | 搜索链路 warm P95 `< 3s`，扩展检索和详情不造成主结果白屏 |
| 回滚 | 新增 M2 能力可通过 feature flag 或降级路径关闭 |
| 默认开关 | `ENABLE_WEIGHTED_RERANK=false`，`.env.example` 不改为 true |
| 正文泄露 | 产物不得包含原始 query、案情正文、候选正文或 chunk 正文 |

### 1.3 本版本不做

- 不承诺“已查全”“保证无遗漏”“查全率 99%”。
- 不进入 M3 的裁判要旨深度摘要、事实对比、案例对比视图。
- 不进入 M4 的收藏、历史、导出、类案报告和团队复用。
- 不默认开启 `ENABLE_WEIGHTED_RERANK`。
- 不让 qrels、label、relevance 参与线上排序、source selection 或前端分组。
- 不根据 query id、case id 做排序特判。
- 不手工改 rank，不修改 qrels，不删除坏样本，不改历史评测结果。
- 不把用户反馈直接写入线上排序权重。
- 不输出原始 query、案情正文、候选正文或 chunk 正文到开发报告。

## 2. 执行原则

1. 先做可信展示和可核验链路，再做阅读提效。
2. 所有生成或解释性内容必须先有来源锚点，后有展示。
3. 覆盖声明只描述数据和策略边界，不描述“已查全”。
4. 低置信度候选只用于复核，不挤占主结果。
5. 扩展检索必须由用户动作或明确低结果状态触发，并可关闭。
6. qrels 只能用于离线评测，不能进入运行时排序决策。
7. 反馈先作为脱敏观测数据，不直接训练或改线上排序。
8. 每一步都要输出 Go/No-Go；任何隐私、来源、绝对话术门禁失败都必须 `NO_GO`。

## 3. 分步骤总览

建议把 M2 拆成 8 个独立步骤，每一步可以单开一个 Codex 会话执行：

| 步骤 | 标题 | 目标 |
| --- | --- | --- |
| M2-1 | 入口证据冻结与可信检索合同 | 冻结 M1.3x-10 入场证据，定义 M2 API/UI/日志合同 |
| M2-2 | 来源锚点链路闭环 | 保证结果卡片、详情和 AI 加工内容都有可追溯锚点 |
| M2-3 | 数据覆盖声明与隐私展示 | 在 API 和前端展示数据覆盖、降级、隐私边界 |
| M2-4 | 低置信度候选分层 | 区分主结果和低置信度候选，避免覆盖主排序 |
| M2-5 | 扩展检索受控入口 | 建立用户触发的扩展检索和回滚门禁 |
| M2-6 | 相关/不相关反馈闭环 | 记录脱敏反馈事件，但不让反馈直接影响排序 |
| M2-7 | 不利案例与风险提示雏形 | 输出有来源的复核线索，不输出法律结论 |
| M2-8 | M2 验收与 M3 入口结论 | 汇总门禁、测试、性能、隐私和下一阶段 Go/No-Go |

## 4.2 步骤 M2-2：来源锚点链路闭环

### 新会话标题

```text
类案检索助手 M2-2 来源锚点链路闭环
```

### 目标

让搜索结果、详情面板和任何 AI 加工内容都能追溯到 `case_id` 和 `source_chunk_id`。无锚点内容不展示。

### 开始前必须读取

- M2-1 产物
- `落地设计文档/02-技术选型与架构设计.md`
- `落地设计文档/04-数据层设计.md`
- `apps/api/app`
- `apps/api/tests/test_summary_service.py`
- `apps/api/tests/test_search_api_fallback_smoke.py`
- `apps/web/src`

### 主要任务

1. 梳理当前搜索、详情和摘要链路中已有的来源字段。
2. 为结果卡片定义最小锚点结构：
   - `case_id`
   - `source_chunk_id`
   - `chunk_type`
   - `anchor_type`
   - `source_url` 或可追溯来源标识
3. 为详情页定义锚点展示策略：
   - 摘要句子有锚点才展示。
   - 高亮片段有锚点才展示。
   - 裁判要旨如无锚点则不展示，回退到来源片段。
4. 新增后端 focused tests：
   - 无锚点生成内容被过滤。
   - 搜索和详情返回的 AI 加工字段必须有锚点。
   - 日志和报告不输出正文。
5. 新增前端 focused tests：
   - 有锚点时可见来源入口。
   - 无锚点时不显示 AI 加工内容。

### 允许修改

- 搜索/详情 API schema。
- 摘要或片段展示的安全过滤。
- 前端结果卡片和详情面板的锚点展示。
- focused tests。

### 禁止修改

- 禁止为了补锚点编造 `source_chunk_id`。
- 禁止无来源内容以“AI 解释”形式展示。
- 禁止改排序权重来配合锚点展示。

### 产物

- `docs/development/m2-source-anchor-closure-<timestamp>.json`

```powershell

cd ../web

```

### 验收

- 用户可见 AI 加工内容 `100%` 有锚点。
- 无锚点内容被稳定降级或隐藏。
- API、日志、报告分层清楚，报告无正文泄露。

### 止损

如果现有数据没有足够 chunk id 支持来源锚点，则本步骤只能输出 `NO_GO` 和数据修复清单，不得用伪锚点绕过。

## 4.4 步骤 M2-4：低置信度候选分层

### 新会话标题

```text
类案检索助手 M2-4 低置信度候选分层
```

### 目标

把主结果和低置信度候选区分开，让用户知道哪些是主要推荐，哪些只是可能相关、需要复核的候选。

### 开始前必须读取

- `落地设计文档/03-前端架构与设计系统.md`

### 主要任务

1. 定义低置信度候选进入条件，只允许使用运行时非标签特征：
   - score 区间
   - 召回来源
   - 法律要素命中数量
   - 主结果数量
   - 降级状态
2. 明确禁止使用：
   - qrels
   - relevance
   - label
   - query id / case id 特判
3. API 区分：
   - `results`
   - `low_confidence_candidates`
   - `confidence_level`
   - `confidence_reasons`

   - 主结果保留主视觉权重。
   - 低置信度候选使用独立面板。
   - 文案使用“部分相关，仅供复核”。
5. focused tests：
   - qrels 不传入 runtime router。
   - query id / case id 不影响分层。

- 置信度分层逻辑。

- 低置信度候选前端面板。
- focused tests。

### 禁止修改

- 禁止用低置信度候选改主排序指标。
- 禁止用 qrels/label/relevance 做 source selection。
- 禁止把低置信度候选包装成“查全结果”。

## 4.5 步骤 M2-5：扩展检索受控入口

### 新会话标题

```text
类案检索助手 M2-5 扩展检索受控入口
```

### 目标

实现用户主动触发或低结果状态触发的扩展检索能力，用于复核可能遗漏，不影响标准检索的默认稳定性。

### 开始前必须读取

- M2-1 至 M2-4 产物
- `落地设计文档/02-技术选型与架构设计.md`
- `落地设计文档/03-前端架构与设计系统.md`
- `apps/api/app/core/config.py`
- `.env.example`
- `apps/api/app/retrieval`
- `apps/api/tests/test_feature_flag_rollback.py`
- `apps/web/src`

### 主要任务

1. 复核 `ENABLE_EXPANDED_SEARCH` 和 `VITE_ENABLE_EXPANDED_SEARCH` 当前默认值。
2. 明确扩展检索触发条件：
   - 用户点击扩展检索。
   - 主结果少于 5 条时展示入口。
   - 无结果时展示入口和修改建议。
3. 明确扩展检索策略：
   - 放宽相似度阈值。
   - 放宽案由限制。
   - 增加低置信度候选。
   - 不改变标准检索默认主结果。
4. 增加后端测试：
   - flag 关闭时接口返回 disabled 状态。
   - flag 开启时返回 mode=`expanded`。
   - 扩展检索不输出绝对查全文案。
5. 增加前端测试：
   - flag 关闭时不显示扩展入口。
   - 低结果状态下显示入口。
   - 展示“扩大复核范围”而不是“保证无遗漏”。

### 允许修改

- 扩展检索 API 或已有接口 mode。
- feature flag 读取和回滚测试。
- 前端扩展检索入口。

### 禁止修改

- 禁止输出法律结论。

## 6. M2 Go/No-Go 口径

### ROBUST_M2_GO

同时满足：

- M1.3x-10 合法候选稳健性仍通过。
- M2-1 至 M2-8 全部硬门禁通过。
- 生成内容来源锚点覆盖 `100%`。
- 数据覆盖声明可展示且不夸大。
- 低置信度候选和主结果清晰分离。
- 扩展检索可关闭、可回滚、无绝对查全文案。
- 反馈事件脱敏，且不影响排序。
- 风险提示有来源、非结论性。
- API、日志、报告无正文泄露。
- `ENABLE_WEIGHTED_RERANK=false`。

### NO_GO

出现任一硬门禁失败即为 `NO_GO`。最终报告必须明确：

- 失败步骤。
- 失败门禁。
- 受影响能力。
- 是否需要回滚。
- 是否阻止进入 M3。
- 下一轮建议从哪个 M2 子步骤继续。
