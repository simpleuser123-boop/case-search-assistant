# 多产品生态全链路闭环设计文档

本文档承接 `M5 商业化扩展` 已建成的多租户/账号/团队/权限/共享/批量导入能力，把「类案检索助手」单一产品扩展为**以检索为核心、多个网站产品串成全链路闭环的产品生态**。

适用前提（沿用历史里程碑已冻结的口径，本文件不放宽任何一条）：

| 前提项 | 状态 |
| --- | --- |
| 来源锚点 | 用户可见的 AI 加工内容必须有 `source_anchors`（`case_id` + `source_chunk_id`），无锚点不进入交付物 |
| 隐私 | 不持久化用户原始案情；持久层、报告、JSON、日志、导出文件、测试快照不得保存正文型内容 |
| 禁用文案 | 不承诺绝对覆盖，不输出诉讼结果概率或确定性法律结论 |
| 默认排序 | 不改变线上排序、source selection 或 rerank 默认开关 |
| 多租户 | 跨团队/跨用户数据强隔离；`team_id`/`owner_user_id` 为空时行为与单用户私有完全一致 |
| 凭据安全 | 不在工具内代填/代管/代存支付或登录凭据；密码不明文存储；令牌不写入日志/产物 |
| 默认开关 | 每个新能力都有 feature flag（默认 `false`），关闭后回到上一里程碑末态 |

---

## 1. 定位与边界

### 1.1 为什么是"生态"而不是"加功能"

M1~M5 解决的是单一产品「类案检索助手」的找得准、信得过、读得快、可沉淀、可商业化。本阶段的目标不同：把检索助手作为**核心枢纽**，在它前面接「案源进入」的入口产品，在它后面接「研究→应用→管理」的下游产品，让一个律师/律所的工作流从案源进来到工作产物沉淀，全程在一个生态内闭环，且沉淀产生的脱敏使用数据回流反哺检索质量。

闭环链路：

```
案情录入端 ──结构化检索画像──▶ 类案检索助手(核心) ──检索结果──┬──▶ 法条检索(研究)
   ▲                                                          ├──▶ 文书工作台(应用)
   │                                                          └──▶ 案件协作台(管理/沉淀)
   └────────────── 脱敏使用数据/团队规则回流(仅聚合) ◀──────────────┘
```

### 1.2 架构选型（已决策：共享平台 + 多前端）

经评估，本生态采用**共享平台 + 多前端**架构，而非独立站点 + SSO 集成。

决策理由：本项目全程的核心约束是「护栏不能漂移」。共享平台让账号（M5-2 已有）、字段白名单契约（M4-1 已冻结）、RAG 检索核心、护栏纪律（仅锚定来源才展示、无来源降级、不落正文、不输出胜负概率）**只实现一次**，所有产品共用同一份契约与同一个检索后端。独立站点方案下每个站点都要重新实现一遍护栏，极易漂移，且闭环里跨产品传数据时隐私红线更难统一守住。

代价与应对：共享平台耦合度高、后端演进影响面大。应对方式是把后端按「共享内核 + 各产品能力包」分层（见 §3），能力包之间不直接互相依赖，只依赖共享内核；每个产品仍有独立 feature flag，可单独关闭而不影响其他产品和基础搜索。

### 1.3 本阶段做什么

v1 闭环纳入 5 个产品（检索助手为已建核心，另 4 个为本阶段新建）：

1. **案情智能录入端**（入口）：把当事人/律师的口语化案情结构化为脱敏检索画像，产出喂给检索的 query。
2. **类案检索助手**（核心，已建）：M1~M5 成果，本阶段作为被调用的检索服务 + 自身前端。
3. **法条法规检索**（研究）：复用现有向量库与 JuDGE 自带法条标注，做法条/法规的语义检索。
4. **文书辅助工作台**（应用）：在 M4 报告模板骨架上前进一步，只组装锚定来源、不起草结论。
5. **案件协作工作台**（管理/沉淀）：复用 M5 多租户/团队/权限，把检索结果、清单、文书归到「案件」维度下管理，是闭环的尾部与反哺入口。

### 1.4 本阶段不做

- 不做独立站点各自部署（已选共享平台）。
- 不让任何产品在录入端把用户原始案情上送服务端持久层（结构化后只传白名单字段）。
- 不让文书工作台起草结论性内容或输出胜负判断。
- 不让回流数据携带任何正文/个案明细（只走聚合计数与脱敏规则）。
- 不默认开启任何跨产品/跨用户/跨团队可见性。
- 不在任何产品内代填/代管支付或登录凭据。
- 不默认开启 `ENABLE_WEIGHTED_RERANK`。

## 3. 共享平台架构

### 3.1 分层

```
┌─────────────────────────────────────────────────────────────┐
│ 前端层 (apps/web 内多入口路由，或多个 app 共享组件库)         │
│  录入端 │ 检索助手 │ 法条检索 │ 文书工作台 │ 案件协作台        │
├─────────────────────────────────────────────────────────────┤
│ 产品能力包层 (apps/api/app/<product>/，互不直接依赖)          │
│  intake/ │ (检索沿用现有) │ statute/ │ drafting/ │ casebook/  │
├─────────────────────────────────────────────────────────────┤
│ 共享内核层 (所有产品依赖，护栏只实现一次)                      │
│  RAG核心: retrieval/ rerank/ query_processing/ summary/       │
│  身份与租户: account/ team/ permission/ sharing/              │
│  契约与护栏: M4-1字段白名单 / source_anchors校验 / 禁用文案守门 │
│  数据: pipeline/ + Chroma向量库 + case_store/                 │
└─────────────────────────────────────────────────────────────┘
```

设计约束：

- 产品能力包**只能依赖共享内核**，不能互相 import（避免耦合成一团）。跨产品的数据流转走显式的「契约对象」传递（见 §4），不走代码直连。
- 护栏（来源锚点校验、字段白名单 sanitize、禁用文案守门、多租户过滤）全部在共享内核实现，能力包调用，不各自重写。
- 每个产品一个总开关 feature flag，默认 `false`。关闭某产品 = 它的前端入口不渲染、它的 API 端点返回安全降级，其余产品与基础搜索不受影响。

### 3.2 统一身份与单点登录（站内）

虽然采用共享平台（非独立站点 SSO），站内多前端仍需统一会话。复用 M5-2 `account/` + `auth.py`：

- 一次登录，会话在所有产品前端共享（同源 + 统一 session/token）。
- 令牌不写入日志/产物（沿用 M5 凭据安全红线）。
- `team_id`/`owner_user_id` 贯穿所有产品的对象归属；为空时各产品行为与单用户私有态一致。

### 3.3 跨产品调用：检索助手作为内部服务

法条检索、文书工作台、案件协作台都需要调用检索能力。检索助手对内暴露稳定的检索服务接口（输入脱敏 query / 输出带锚点的候选 + 元数据），其他产品作为消费方。这样检索逻辑只有一份，护栏在检索侧统一执行。

## 4. 闭环数据流与契约对象

跨产品的数据流转不走代码直连，只通过显式「契约对象」传递。每个契约对象都受 M4-1 字段白名单约束，sanitize 时主动丢弃非白名单键（含潜在正文）。

### 4.1 四个契约对象

1. **检索画像 (SearchProfile)** ── 录入端 → 检索助手
   - 字段：结构化案件要素（案由/地域/审级倾向/争议焦点关键词）+ 脱敏后的查询文本。
   - 红线：原始口语化案情**不在内**（仅浏览器本地）；只携带结构化白名单字段 + 已脱敏 query。

2. **检索结果引用 (CandidateRef)** ── 检索助手 → 法条/文书/协作台
   - 字段：`case_id` + 元数据（case_number/court/trial_level/case_cause/judgment_date）+ `source_anchors`（`case_id`+`source_chunk_id`）。
   - 红线：**不含候选正文/chunk 正文/裁判文书全文**；下游要展示正文需自行实时取且不落库。

3. **文书草稿描述 (DraftDescriptor)** ── 文书工作台 → 协作台
   - 字段：文书结构骨架（段落标题，非正文）+ 引用的 CandidateRef 列表 + **可选** StatuteRef 列表（经 E5 互跳，E6-1 合同变更追加）+ 用户自填短字段（note/tag）。
   - 红线：只存引用与骨架，不存起草正文；`structure_skeleton` 标题非正文（单项长度上限）；引用 100% 有锚点（缺锚点 fail-closed 丢弃）；导出走 M4-5 免责头机制。

4. **案件聚合 (CaseFolder)** ── 协作台内，归集上述三者
   - 字段：`case_folder_id` + owner/team/visibility + 关联的 SearchProfile 摘要(脱敏) / CandidateRef 清单 / DraftDescriptor 列表 + 时间戳。
   - 红线：默认 `visibility=private`；对象级鉴权；持久层只存元数据/引用/短字段。

> **合同变更登记（E5-1 引入第 5 个契约对象，2026-06-17）**：法条检索端 `statute/` 落地需要一个「法条检索结果」契约对象，原四对象不覆盖，故经本登记新增 `StatuteRef`（**非擅自加字段**）。详见下方 §4.1.1 与文档 17 §3.5。

> **合同变更登记（E6-1 为 DraftDescriptor 追加可选 statute_refs，2026-06-18）**：E6 文书工作台需引用经 E5 互跳而来的法条 `StatuteRef`。`DraftDescriptor` 的 E-1 核心五字段（`draft_id`/`structure_skeleton`/`candidate_refs`/`note`/`tag`）**保持不变**，经本登记追加**可选** `statute_refs` 字段（沿 StatuteRef 范式，**非擅自加字段**）；持久层另补 `created_at`/`updated_at`/`owner_user_id`/`team_id`/`visibility`（同 CaseFolder，默认 `visibility=private`）。E6 权威白名单冻结为独立常量 `DRAFT_DESCRIPTOR_E6_FIELDS`（落地 `app/kernel/guardrails/contracts/drafting_contract.py`，纯数据 + 纯函数，E6-1 不接线），**不进** `CONTRACT_FIELD_WHITELIST`（E-1 四对象口径不动）。详见文档 17 §3.3 与文档 21 §1。

#### 4.1.1 StatuteRef（检索助手/法条检索 → 法条结果消费方）── E5-1 新增

5. **法条检索结果 (StatuteRef)** ── 法条检索端产出 → 前端展示 / 法条↔类案互跳
   - 字段：`statute_id`（法条标识）+ `law_name`（法名）+ `article_no`（条号）+ `statute_anchors`（**必填非空**，每条至少 `text_id` 指向法条语料）+ 可选 `article_text`（条文文本）/ `source_corpus`（语料来源）/ `effective_status`（时效状态）/ `related_case_refs`（`list[CandidateRef]`，法条→类案互跳）。
   - 红线（法条特有）：
     - **条文必锚定语料、不得由模型生成**：`article_text`（若有）只来自法条语料库且带 `text_id` 锚点；模型不杜撰/改写/续写条文；命中无锚点则降级不展示（契约层以「`statute_anchors` 必填非空、每条至少 `text_id`」表达「无锚点不展示」）。
     - **互跳只走契约对象**：`related_case_refs` 只承载 `CandidateRef` 同款字段（白名单七字段、必有 `source_anchors`、**零裁判正文**）；互跳不内嵌对侧正文，`CandidateRef` 字段不因互跳被增删。
     - 拒绝任何裁判正文型键（`full_text`/`content`/`chunk_text`/`summary_text`/`highlight_text`/`matched_text`）、PII 型键、以及模型生成条文型键（`generated_article`/`llm_text`/`paraphrased_article`/...），出现即 `NO_GO`。
   - flag：受 `ENABLE_STATUTE_SEARCH` 门控（默认 `false`，E-1 已冻结，不新增产品级 flag）；关闭时法条端点 403 降级、前端无入口。

### 4.2 反哺回流（只走聚合）

协作台与各产品产生的使用数据，回流改进检索质量，但**只走聚合、绝不含正文/个案明细**：

- 允许回流：检索命中后的脱敏交互聚合（如某案由下结果被加入清单的聚合计数、降级触发率、result_count 分布）、团队自定义的检索偏好规则（如优先本地法院，规则本身不含案情）。
- 禁止回流：任何 raw_query、案情正文、候选/chunk/裁判文书正文、具名个案明细、qrels/label/relevance（这些只用于离线评测，不进入线上排序）。
- 回流如要影响排序，必须经独立的 `ENABLE_WEIGHTED_RERANK` 路径评估，默认 `false`，不改主排序默认。

## 5. 隐私与安全红线（生态级强化）

在 M4/M5 红线基础上，生态架构新增三条跨产品红线：

1. **录入端原始案情零上送**：用户在录入端输入的原始口语化案情，仅在浏览器本地用于生成结构化画像，结构化完成后即可清除；服务端持久层 `user_raw_case_persisted_on_server=false`（沿用 M4-1）。任何把原始案情写入服务端的实现即触发 `NO_GO`。

2. **跨产品传递只走契约对象**：产品间不直接共享内存对象或数据库表，只通过 §4 的四个契约对象传递；每个契约对象过白名单 sanitize。跨产品出现正文型字段即触发 `NO_GO`。

3. **回流聚合无正文**：反哺数据 100% 聚合/规则，抽查任意回流记录 bodyHits 必须为空、rawQueryLeak 必须为 false。

其余沿用：来源锚点继承、禁用文案守门（不"已查全"/不胜负概率）、多租户强隔离、对象级鉴权、凭据不代管、离线评测隔离、默认排序不变、性能 warm P95 < 3s、每能力可 flag 关闭回退。

## 6. 分步骤执行计划（沿用 Mx-y 同款门禁结构）

本阶段记为 **E 系列（Ecosystem）**，每步独立 Go/No-Go，前置须全 GO，止损条件 = 正文泄露/破坏主链路/默认开启跨用户可见性即 `NO_GO`。

| 步骤 | 内容 | 核心产物 | 关键依赖 |
| --- | --- | --- | --- |
| E-1 | 生态入口合同 | 共享平台分层契约 + 4 个契约对象字段白名单 + 5 个产品 flag 冻结（默认 false） | M5-10 收官 |
| E-2 | 共享内核抽取 | 把 RAG 核心/身份租户/护栏从检索助手抽为内核层，检索助手改为内核消费方（行为零变化回归） | E-1 |
| E-3 | 检索内部服务接口 | 检索助手对内暴露检索服务（输入 SearchProfile / 输出 CandidateRef），护栏在检索侧统一执行 | E-2 |
| E-4 | 案情智能录入端 | intake/ 包 + 录入前端；原始案情本地 only，产出脱敏 SearchProfile | E-3 |
| E-5 | 法条法规检索 | statute/ 包 + 前端；复用向量库+法条标注，命中带锚点，法条↔类案互跳 | E-3 |
| E-6 | 文书辅助工作台 | drafting/ 包 + 前端；M4-6 骨架进一步，只组装锚定来源 + 免责导出 | E-3 |
| E-7 | 案件协作工作台 | casebook/ 包 + 前端；CaseFolder 聚合，复用 M5 多租户/权限/共享 | E-3、E-4~E-6 契约就绪 |
| E-8 | 闭环反哺（聚合 only） | 脱敏使用数据/团队规则回流管道；默认不影响排序 | E-7 |
| E-9 | 生态验收 + 运营入口 | 全链路闭环 E2E 验收 + 三类全 GO 判定 + 后续 M6 入口 | E-1~E-8 全 GO |

### 6.1 顺序依赖说明

E-2/E-3 是地基（抽内核 + 暴露检索服务），必须先做，否则后面每个产品会各自直连检索逻辑导致护栏漂移。E-4~E-6 三个产品在 E-3 之后可并行设计、串行落地（沿用记忆里"并行 bash 会冲突、必须串行验证"的环境约束）。E-7 协作台依赖前面产品的契约对象就绪。E-8 反哺放最后，确保前面隐私边界都稳了再开数据回流。

### 6.2 成功标准（生态级）

| 指标 | 门槛 |
| --- | --- |
| 闭环连通 | 录入端→检索→研究/应用→协作台 全链路可走通，跨产品只传契约对象 |
| 护栏单点 | 来源锚点/白名单/禁用文案/多租户过滤全部在共享内核执行，能力包不重写 |
| 隐私 | 原始案情零上送；契约对象/持久层/回流 100% 无正文；抽查 bodyHits=[]、rawQueryLeak=false |
