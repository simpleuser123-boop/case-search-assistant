# M1.3x 回归清零版分步骤文档

本文档承接 `M1.3 Final Verification`：

- `docs/development/m1.3-final-verification-20260610-152812.md`
- `docs/development/m1.3-release-gate-20260610-152812.json`

M1.3 的正式结论是：

| 结论项 | 状态 |
| --- | --- |
| 基础搜索是否继续可用 | `GO` |
| 新 weighted rerank 是否允许成为灰度候选 | `NO_GO` |
| 是否可以进入 M2 可信检索准备 | `NO_GO` |
| `ENABLE_WEIGHTED_RERANK` 默认状态 | `false` |

因此，M1.3x 不是 M2 功能实施版，也不是新能力扩展版。它是 M1 阶段内的第三轮回归清零版，目标是继续处理 M1.3 final 阻断项：`after-vs-baseline REGRESSED`、`METRIC_REGRESSION` 和剩余 `RECALL_MISS`。

`x` 表示本轮允许作为 M1.3 的修复补丁序列继续推进；但每一步仍必须独立验收，不允许把未清零的阻断项包装成“建议观察”。

---

## 1. 当前起点

依据 M1.3 final verification，当前状态如下：

| 项 | 当前状态 |
| --- | --- |
| 基础搜索 | `GO`，继续可用 |
| weighted rerank 灰度候选 | `NO_GO` |
| M2 readiness | `NO_GO` |
| `ENABLE_WEIGHTED_RERANK` 默认状态 | `false`，必须继续关闭 |
| product eval current Precision@5 | `0.048` |
| product eval current NDCG@10 | `0.171` |
| product eval current Top10 hit rate | `0.68` |
| product eval bad cases | `METRIC_REGRESSION=7`，`RECALL_MISS=7` |
| M1.3 best combined candidate | Top10 `0.68`，before->after `REGRESSED=0`，after-vs-baseline `REGRESSED=7`，`METRIC_REGRESSION=7`，`RECALL_MISS=7` |
| M1.2 runner 兼容对照 | currentAfter Top10 `0.64`，before->after `REGRESSED=4`，after-vs-baseline `REGRESSED=9`，`METRIC_REGRESSION=9` |
| performance smoke | warm P95 `551ms` |
| rollback drill | max rollback `16ms`，无需重建索引 |

M1.3x 的第一优先级不是继续追求 aggregate Top10，而是回答：

1. 为什么 Top10 已达到 `0.68`，但 `METRIC_REGRESSION` 仍为 `7`。
2. 为什么 best candidate 仍有 after-vs-baseline `REGRESSED=7`。
3. M1.2 runner 兼容对照与 M1.3 candidate comparison 的差异如何解释，哪条线作为最终门禁。
4. 是否能在不伤害基础搜索和成功样本的前提下，把固定回归和 metric regression 清零。

---

## 2. 版本目标

### 2.1 主目标

把 M1.3 的“基础搜索可用、Top10 aggregate 达标，但 per-query 回归阻断”推进到“至少一个候选版本通过全部硬门禁，或明确 M1.3x 仍然 `NO_GO`”。

### 2.2 成功标准

M1.3x 结束时，至少要同时满足以下条件，才允许重新讨论 M2 readiness：

| 指标 | 门槛 |
| --- | --- |
| product-local eval | 可重复运行，无 blocked item |
| Precision@5 | `current >= baseline` |
| NDCG@10 | `current > baseline` |
| Top10 hit rate | `current >= 0.60` |
| fixed regression before -> after | `REGRESSED=0` |
| fixed regression after vs baseline | `REGRESSED=0` |
| bad case | `METRIC_REGRESSION=0` |
| 剩余 `RECALL_MISS` | 有 query id 级解释、止损和不进入默认路径的边界 |
| M1.3 candidate comparison | exactly one selected candidate `GO`，或明确全部 `NO_GO` |
| M1.2 runner 兼容对照 | 作为对照保留；如与 M1.3 runner 结论不一致，必须解释差异 |
| 性能 | warm P95 `< 3s` |
| 回滚 | `< 60s`，无需重建索引 |
| 默认开关 | `ENABLE_WEIGHTED_RERANK=false` |
| 隐私 | 不写入原始 query、案情正文、候选正文或 chunk 正文 |

### 2.3 分层结论

M1.3x 最终必须输出三类正式结论，状态只能是 `GO` 或 `NO_GO`：

| 结论项 | 允许状态 |
| --- | --- |
| 基础搜索是否继续可用 | `GO` 或 `NO_GO` |
| 新 weighted rerank 是否允许成为灰度候选 | `GO` 或 `NO_GO` |
| 是否允许进入 M2 可信检索准备 | `GO` 或 `NO_GO` |

### 2.4 本版本不做

- 不做 M2 的来源引用、引用锚点、覆盖声明、低置信度候选产品化扩面。
- 不做 M3 的裁判要旨摘要、事实对比、阅读提效扩展。
- 不做 M4 的检索历史、收藏、导出、类案报告。
- 不做团队空间、权限、商业化能力。
- 不改 qrels、评测样本、历史评测结果。
- 不删除退化样本，不伪造指标，不手工改结果。
- 不把 `ENABLE_WEIGHTED_RERANK` 默认改成 `true`。
- 不把 M1.3x 通过等同于对外发布；最多只允许成为受控灰度候选。

---

## 3. 执行原则

1. 先清零 per-query 回归，再讨论 M2。
2. 先解释 M1.3 final 阻断项，再改召回或排序。
3. M1.3 runner 是候选横评主线；M1.2 runner 必须保留为兼容对照。
4. 任何 aggregate 指标不能覆盖 `METRIC_REGRESSION>0` 或 fixed regression `REGRESSED>0`。
5. 不做 case-id hardcoding，不做 query-id 特判排序。
6. 所有新增诊断字段只允许包含 query id、case id、rank、score、label、count 和状态。
7. 每一步都要明确是否修改线上搜索逻辑；如只是评测或诊断，必须写明。

---

## 4. 分步骤总览

建议把 M1.3x 拆成 8 个独立步骤，每一步可以单开一个 Codex 会话执行：

| 步骤 | 标题 | 目标 |
| --- | --- | --- |
| M1.3x-1 | 阻断证据冻结与门禁口径对齐 | 冻结 M1.3 final 阻断项，明确主门禁和兼容对照 |
| M1.3x-2 | `METRIC_REGRESSION` 根因矩阵 | 对剩余 7 个 metric regression 做分数与排序归因 |
| M1.3x-3 | baseline 保护门禁与测试补强 | 防止 aggregate 变好但 baseline 成功样本被压低 |
| M1.3x-4 | rerank guard v2 定点修复 | 只修 after-vs-baseline 回归，不扩大 M2 能力 |
| M1.3x-5 | 剩余 `RECALL_MISS` 闭环与止损 | 对剩余 recall miss 给出可修复结果或不可修复解释 |
| M1.3x-6 | 候选版本统一横评与参数冻结 | 横评 guard v2、recall closure、combined v2，选一个或全部 `NO_GO` |
| M1.3x-7 | 灰度候选 readiness 复核 | 只复核默认关闭、API schema、性能、回滚和前端稳定 |
| M1.3x-8 | M1.3x 验收与 M2 入口结论 | 输出正式 Go/No-Go 和下一阶段建议 |

- 不包含原始 query、案情正文、候选正文或 chunk 正文。
- 不修改搜索逻辑、qrels、评测样本或参数。

### 止损

如果 M1.3 final 产物缺失，或 blocker register 无法解释 M1.3 与 M1.2 runner 的差异，则停止后续修复，输出 `NO_GO`。

## 5.2 步骤 M1.3x-2：`METRIC_REGRESSION` 根因矩阵

### 新会话标题

```text
类案检索助手 M1.3x-2 METRIC_REGRESSION 根因矩阵
```

### 目标

对剩余 `METRIC_REGRESSION=7` 和 after-vs-baseline `REGRESSED=7` 做可修复归因，区分是排序 guard、score bucket、candidate ordering、baseline 保护缺失还是数据边界问题。

### 开始前必须读取

- M1.3x-1 产物
- `docs/development/m1.3-rerank-regression-repair-20260610-134631.md`
- `docs/development/m1.3-rerank-regression-repair-20260610-134631.json`
- `docs/development/m1.3-final-candidate-comparison-20260610-152812.json`
- `apps/api/app/rerank/service.py`
- `apps/api/app/retrieval/merge.py`
- `apps/api/app/eval/product_eval.py`
- `apps/api/scripts/m1_3_candidate_comparison.py`

### 主要任务

1. 对每个 `METRIC_REGRESSION` query 输出脱敏分数拆解：
   - baseline relevant rank
   - M1.3 candidate relevant rank
   - vector score bucket
   - legal element score
   - case cause score
   - key paragraph effective score
   - authority effective score
   - guard reasons
2. 给每个 query 标记主因：
   - `BASELINE_PROTECTION_GAP`
   - `VECTOR_BUCKET_TIE_BREAK_ERROR`
   - `MULTI_SOURCE_BONUS_OVERBOOST`
   - `WEAK_SIGNAL_RESIDUAL_BOOST`
   - `RECALL_POOL_ORDERING_SIDE_EFFECT`
   - `QRELS_OR_DATA_BOUNDARY`
   - `UNREPAIRABLE_WITH_CURRENT_SIGNALS`
3. 明确哪些 query 进入 M1.3x-4 排序修复，哪些进入 M1.3x-5 recall/解释闭环。
4. 如需新增 runner 诊断字段，只允许加评测输出，不改变线上排序。

### 产物

- `docs/development/m1.3x-metric-regression-root-cause-*.md`
- `docs/development/m1.3x-metric-regression-root-cause-*.json`

### 验收

- `METRIC_REGRESSION=7` 的每个 query 都有主因标签。
- 能明确说明哪些项可修，哪些项只能作为剩余风险解释。
- 不用原始 query 或正文解释问题。

### 止损

如果无法归因，不允许直接进入 rerank 修复；先补诊断字段和 runner 输出。

### 新会话标题

```text
类案检索助手 M1.3x-3 baseline 保护门禁与测试补强
```

### 目标

把“不能低于 baseline 成功样本”的规则写成可测试门禁，防止 Top10 aggregate 提升掩盖 per-query 回归。

### 开始前必须读取

- `apps/api/tests/test_rerank_service.py`

### 主要任务

1. 在评测或 gate 层补齐 baseline 保护字段：
   - baseline best relevant rank
   - candidate best relevant rank
   - rank_delta_vs_baseline
   - metric_regression_reason
2. 增加 focused tests：
   - aggregate Top10 提升但 after-vs-baseline `REGRESSED>0` 时必须 `NO_GO`
   - `METRIC_REGRESSION>0` 时必须 `NO_GO`
   - selected candidate 必须 exactly one

4. 不修改线上检索逻辑。

- 测试能证明 per-query 回归无法被 aggregate 指标覆盖。
- M1.3 candidate comparison 仍能读取旧产物并输出新字段。
- 不改变 `ENABLE_WEIGHTED_RERANK` 默认值。

### 止损

如果 gate 补强导致历史结果不可读，必须先修兼容；不能通过破坏历史对比来制造 `REGRESSED=0`。

## 5.4 步骤 M1.3x-4：rerank guard v2 定点修复

### 新会话标题

```text
类案检索助手 M1.3x-4 rerank guard v2 定点修复
```

### 目标

只修 M1.3x-2 确认的排序退化，不扩大召回、不引入 M2 功能、不默认开启 weighted rerank。

### 开始前必须读取

- M1.3x-1 到 M1.3x-3 产物
- `docs/development/m1.3-rerank-regression-repair-20260610-134631.md`
- `docs/development/m1.3-rerank-regression-repair-20260610-134631.json`
- `apps/api/app/rerank/service.py`
- `apps/api/tests/test_rerank_service.py`
- `apps/api/tests/test_feature_flag_rollback.py`

### 主要任务

1. 针对根因矩阵设计最小 guard v2 策略：
   - 限制 weak-signal-only boost。
   - 收紧或分层 multi-source bonus。
   - 修正 vector bucket 内 tie-break。
   - 保护强向量/强事实支持候选，避免过度 guard。
2. 每个策略都必须生成候选版本，不直接替换默认路径。
3. 每个候选版本必须同时评测：
   - baseline
   - M1.3 final combined candidate
   - guard v2 candidate
4. 禁止 case-id hardcoding、query-id 特判、手工改 rank。

### 产物

- `docs/development/m1.3x-rerank-guard-v2-*.md`
- `docs/development/m1.3x-rerank-guard-v2-*.json`
- 必要时包含 `apps/api/app/rerank/service.py` 与测试的小范围改动。

### 验收

- before->after `REGRESSED=0`。
- after-vs-baseline `REGRESSED` 必须下降，目标为 `0`。
- `METRIC_REGRESSION` 必须下降，目标为 `0`。
- Top10 hit rate 保持 `>=0.60`。
- 不重新引入 key paragraph / authority weak-signal 过度抬分。

### 止损

如果 guard v2 让 Top10 继续提升但 after-vs-baseline 或 `METRIC_REGRESSION` 仍不为 `0`，不得进入灰度候选，只能保留为离线候选。

## 5.5 步骤 M1.3x-5：剩余 `RECALL_MISS` 闭环与止损

### 新会话标题

```text
类案检索助手 M1.3x-5 剩余 RECALL_MISS 闭环与止损
```

### 目标

对剩余 `RECALL_MISS=7` 做闭环：能修的只做定点 recall 修复，不能修的给出证据级解释和止损边界。

### 开始前必须读取

- M1.3x-1、M1.3x-2、M1.3x-4 产物
- `docs/development/m1.3-recall-repair-20260610-113200.md`
- `docs/development/m1.3-recall-repair-20260610-113200.json`
- `apps/api/app/retrieval/service.py`
- `apps/api/app/retrieval/merge.py`
- `apps/api/app/retrieval/bm25_fallback.py`
- `apps/api/app/query_processing/term_mapping.py`

### 主要任务

1. 对每个 `RECALL_MISS` query 重新确认：
   - relevant case 是否在 vector topK
   - 是否在 BM25 topK
   - 是否被 merge/dedupe/gating 丢失
   - 是否属于 qrels/data boundary
2. 只允许定点 recall 修复：
   - controlled BM25 supplement
   - recall-only query mapping
   - merge admission rule
3. 不允许默认扩大所有 query 的候选池。
4. 对不可修复项输出止损说明：
   - 为什么当前信号不足
   - 为什么不能靠 M1.3x 安全修复
   - 是否会阻断 M2

### 产物

- `docs/development/m1.3x-recall-miss-closure-*.md`
- `docs/development/m1.3x-recall-miss-closure-*.json`
- 必要时包含召回层小范围修复和测试。

### 验收

- `RECALL_MISS` 有下降，或每个剩余项都有不可修复解释。
- 不新增 `METRIC_REGRESSION`。
- 不新增 fixed regression。
- 性能不明显恶化，warm P95 `<3s`。

### 止损

如果 recall closure 修复了 miss 但新增任何 fixed regression 或 `METRIC_REGRESSION`，该策略不得进入 combined candidate。

## 8. M1.3x Definition of Done

M1.3x 完成必须同时满足：

- 有冻结后的 blocker register。
- 有 `METRIC_REGRESSION` 根因矩阵。
- 有 baseline 保护门禁和测试。
- `METRIC_REGRESSION` 明确下降到 `0`，或最终明确 `NO_GO`。
- fixed regression before->after `REGRESSED=0`。
- fixed regression after-vs-baseline `REGRESSED=0`。
- 剩余 `RECALL_MISS` 有 query id 级解释和止损。
- 至少一个候选参数版本通过全部硬门禁，或明确全部候选 `NO_GO`。
- 新 weighted rerank 是否允许成为灰度候选有正式门禁。
- 是否进入 M2 readiness 有正式 Go/No-Go。
- 默认 feature flags 没有被悄悄打开。
- 没有把 M2、M3、M4 功能提前混进来。
- 没有把原始案情文本写进日志、报告正文或新增 JSON 明文字段。

## 9. M2 入口条件

只有 M1.3x-8 同时给出以下结论时，下一轮才允许进入 M2 可信检索准备：

| 条件 | 必须满足 |
| --- | --- |
| 基础搜索 | `GO` |
| weighted rerank 灰度候选 | `GO`，或明确 M2 不依赖该排序且 baseline 精准度已达标 |
| product-local Top10 hit rate | `>= 0.60` |
| fixed regression | `REGRESSED=0` |
| bad case | `METRIC_REGRESSION=0`，剩余 `RECALL_MISS` 有解释和止损 |
| 性能 | warm P95 `<3s` |
| 回滚 | `<60s`，无需重建索引 |
| 隐私 | 无原始案情/候选正文/chunk 正文写入 |
| 默认开关 | `ENABLE_WEIGHTED_RERANK=false`，除非另有明确批准的灰度开关策略 |

如果任何一项不满足，下一轮继续做 M1.3x 内部修复或输出阶段性 `NO_GO`，不得进入 M2 实施。
