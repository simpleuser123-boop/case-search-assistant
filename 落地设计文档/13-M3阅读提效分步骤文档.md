# M3 阅读提效分步骤文档

本文档承接 `M2-8 M2 验收与 M3 入口结论`：

- `docs/development/m2-final-verification-20260612-191945.md`
- `docs/development/m2-release-gate-20260612-191945.json`
- `docs/development/m2-parameter-version-20260612-191945.json`

最新 M2-8 结论是：

| 结论项 | 状态 |
| --- | --- |
| 基础搜索继续可用 | `GO` |
| M2 可信检索完成 | `GO` |
| M3 入口 | `GO` |
| M2-1 至 M2-7 | 全部 `GO` |
| 来源锚点 | 用户可见 AI 加工内容必须有 `source_anchors` |
| 隐私 | 不持久化用户原始案情，报告、JSON、日志和测试快照不得保存正文型内容 |
| 禁用文案 | 不承诺绝对覆盖，不输出诉讼结果概率或确定性法律结论 |
| 默认排序 | 不改变线上排序、source selection 或 rerank 默认开关 |
| `ENABLE_WEIGHTED_RERANK` 默认状态 | `false` |

因此，M3 可以作为下一阶段开始执行；但 M3 不是工作流沉淀版，不做收藏、历史、导出、类案报告或团队复用。M3 的目标是把“用户找到一组可信结果”推进为“用户能更快判断某篇或几篇案例是否可用”。

---

## 1. M3 定位

M3 对应 `01-演进计划总览.md` 中的 `M3 阅读提效`，在 `07-开发步骤.md` 中落到 `Sprint 4：阅读提效`。

### 1.1 版本目标

减少用户逐篇打开裁判文书、手动比对事实和裁判理由的时间：

1. 在详情页提供有来源锚点的裁判要旨摘要。
2. 提炼争议焦点和关键法律要素，但只作为阅读导航，不输出法律结论。
3. 将本次输入中的事实维度与案例事实维度做受控对照，不持久化用户原始案情。
4. 在详情页定位与输入最相关的来源片段，帮助用户快速回到原文。
5. 支持少量案例的受控对比视图，帮助用户横向判断可用性。
6. 支持复制案号和基础引用格式，但不做导出清单、收藏、历史或报告。

### 1.2 成功标准

M3 结束时，至少要同时满足以下条件，才允许讨论进入 M4 工作流沉淀：

| 指标 | 门槛 |
| --- | --- |
| 来源锚点覆盖 | 裁判要旨、争议焦点、事实对比、相似片段、案例对比中的案例侧 AI 加工内容 `100%` 绑定 `case_id` 和 `source_chunk_id` |
| 用户输入隐私 | 用户原始案情、自由文本、长片段上下文不持久化，不写入报告、JSON、日志、测试快照或埋点 |
| 无来源内容处理 | 无 `source_anchors` 的案例侧 AI 内容不展示，降级为来源片段入口或空状态 |
| 阅读辅助边界 | 所有摘要、焦点、对比和高亮只表达阅读线索，不表达确定性法律结论 |
| 事实对比边界 | 用户侧事实只使用本次请求内的临时抽象字段，不保存原文，不作为排序特判 |
| 对比视图边界 | 仅对用户当前选择的少量结果做阅读对比，不建立收藏、历史、导出或报告能力 |
| 引用复制边界 | 只复制案号、法院、审级、日期等元数据和基础引用格式，不生成类案清单或报告 |
| 默认排序 | 不改变主结果排序、source selection、rerank 默认开关和 M2 扩展检索默认行为 |
| 离线评测隔离 | qrels、label、relevance 只用于离线评测，不进入运行时摘要、分层、对比或高亮决策 |
| 性能 | 主搜索 warm P95 继续 `< 3s`；M3 阅读辅助可懒加载、超时降级，不造成主结果白屏 |
| 回滚 | 新增 M3 能力可通过 feature flag 或安全降级关闭 |
| 正文泄露 | 产物不得包含原始 query、案情正文、候选正文、chunk 正文或裁判文书长正文 |

### 1.3 本版本不做

- 不做 M4 的收藏、历史、导出、类案清单、轻量报告和团队复用。
- 不自动起草代理词、诉状、答辩状或庭审提纲。
- 不输出胜诉概率、败诉概率、风险定级分数或确定性法律结论。
- 不承诺绝对覆盖，不使用“已查全”“保证无遗漏”“查全率”等话术。
- 不默认开启 `ENABLE_WEIGHTED_RERANK`。
- 不改变 M1/M2 已验证的线上排序、召回、source selection 或扩展检索默认行为。
- 不让 qrels、label、relevance 参与运行时摘要、对比、高亮、排序或分组。
- 不根据 query id、case id 做排序、摘要、对比或高亮特判。
- 不把用户原始案情、候选正文、chunk 正文或裁判文书长正文写入开发报告。

## 2. 执行原则

1. 先做阅读辅助合同，再做具体摘要、对比和高亮能力。
2. 案例侧生成或解释性内容必须先有 `source_anchors`，后有展示。
3. 用户侧输入只允许在请求内临时抽象，不持久化原文。
4. M3 能力默认是详情页和对比视图的阅读辅助，不反向影响主搜索排序。
5. 每个阅读辅助模块都必须有隐藏、降级或关闭路径。
6. 文案必须表达“复核线索”“阅读定位”“对照维度”，不能表达“结论已确定”。
7. 每一步都要输出 Go/No-Go；任何隐私、来源、绝对话术、排序污染门禁失败都必须 `NO_GO`。

## 3. 分步骤总览

建议把 M3 拆成 8 个独立步骤，每一步可以单开一个 Codex 会话执行：

| 步骤 | 标题 | 目标 |
| --- | --- | --- |
| M3-1 | 阅读提效入口合同 | 冻结 M2 GO 证据，定义 M3 API/UI/日志/feature flag 合同 |
| M3-2 | 裁判要旨摘要来源闭环 | 在详情页提供有来源锚点的裁判要旨摘要和降级路径 |
| M3-3 | 争议焦点与关键要素提炼 | 输出可复核的争议焦点和关键要素阅读导航 |
| M3-4 | 相似事实对比 | 做用户输入事实维度与案例事实维度的隐私安全对照 |
| M3-5 | 相似片段高亮与关键段落定位 | 将阅读辅助落到可点击、可追溯的来源片段 |
| M3-6 | 案例对比视图受控入口 | 支持当前结果内少量案例横向对比，不沉淀为工作流 |
| M3-7 | 复制案号与引用格式边界 | 支持复制基础引用信息，不生成清单、导出或报告 |
| M3-8 | M3 验收与 M4 入口结论 | 汇总门禁、测试、隐私、性能和下一阶段 Go/No-Go |
