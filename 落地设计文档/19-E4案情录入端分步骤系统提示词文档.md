## 1. E4 总目标

E4 的目标是落地 E 系列**第一个真正的产品能力包 + 第一个前端产品**：案情智能录入端。它把律师口语化的「原始案情」转成一份**已脱敏、结构化、字段白名单**的 `SearchProfile`，再调用 E3 的内部检索服务得到 `CandidateRef[]`，形成「案源 → 检索」这一段闭环的起点：

```text
原始案情（口语化、含当事人/隐私）   ← 仅浏览器本地内存，零上送服务端
  -> 本地脱敏 + 要素抽取（规则/本地）
  -> SearchProfile（白名单：case_cause / region / trial_level_preference / dispute_focus_keywords / query_text<已脱敏>）
  -> intake 后端包（仅做防御性校验 + 透传）
  -> E3 InternalSearchService.search_candidate_refs(SearchProfile)
  -> CandidateRef[]（无正文，带 source_anchors）
```

E4 的本质是**第一个产品包落地 + 第一条生态红线（原始案情零上送）变成可执行边界**，不是放开护栏。E4 不做 statute/drafting/casebook，不改排序/召回/source selection/rerank 默认，不扩语料，不重建索引，不绕过 E3 直连检索底层。

### 1.1 地基决策（用户 2026-06-16 拍板）

- **脱敏与要素抽取默认在浏览器本地完成**：原始案情只存在于浏览器内存，绝不上送服务端；只有脱敏后的 `SearchProfile`（白名单字段 + 已脱敏 `query_text`）离开浏览器。这是生态红线①「录入端原始案情零上送」的 100% 达成方式。
- **服务端 AI 增强抽取是后期可选项，本期只冻结不接线**：当未来需要更强的 AI 要素抽取能力时，才考虑服务端增强；其调用前置是**用户显式确认 + 已完成本地脱敏**，且服务端增强**只接收已脱敏文本、用完即丢、零持久化零日志正文**。E4 阶段把它冻结为二级 flag `ENABLE_INTAKE_AI_EXTRACTION`（默认 `false`、预留接口、不接线、无 on 路径）。E4 只落地纯本地路径。

### 1.2 E4 新增 flag（在 E-1 五产品 flag 之外）

| flag | 含义 | 默认 | E4 阶段状态 |
| --- | --- | --- | --- |
| `ENABLE_INTAKE`（E-1 已冻结） | 案情录入端总开关：前端入口 + intake 端点是否可用 | `false` | E4 实现其 on 路径（本地脱敏 → SearchProfile → E3），但默认仍 `false` |
| `ENABLE_INTAKE_AI_EXTRACTION`（E4 新增） | 录入端**服务端 AI 增强抽取**子开关，正交于 `ENABLE_INTAKE` | `false` | **仅声明 + 预留接口，零接线、无 on 路径**；只有「用户确认 + 已脱敏」才允许调用（后期实现） |

> `ENABLE_INTAKE_AI_EXTRACTION` 与 `ENABLE_INTAKE` 正交：前者控制「是否允许服务端对已脱敏文本做增强抽取」，后者控制「录入端是否可用」。E4 阶段两者皆 `false`；E4 只接线 `ENABLE_INTAKE` 的本地路径，AI 增强子开关全程 off 不接线。任一默认 `true` 即 `NO_GO`。

### 1.3 E4 守的生态红线（在 M4/M5 全部红线之上）

1. **原始案情零上送**：原始口语化案情/当事人姓名/身份证/手机号/住址等只在浏览器内存，绝不进入任何后端请求体、持久层、日志、报告、导出、测试快照。
2. **跨产品只走契约对象**：录入端 → 检索只传 `SearchProfile`（白名单），检索 → 录入端只回 `CandidateRef`（无正文）；不直连检索底层。
3. **服务端增强（后期）仅接收已脱敏文本**：`ENABLE_INTAKE_AI_EXTRACTION` 即便后期开启，也只接收脱敏后文本、用完即丢、零持久化、日志不写正文。
4. 沿用 M4/M5 全部红线：仅锚定来源才展示 / 无来源降级 / 不落正文 / 不按 case_id 特判 / 不输出胜负概率 / flag 默认 false / 关闭回上一态 / 多租户强隔离 / 凭据不代管。

## 2. E4 分步计划（每步独立 Go/No-Go，前置须全 GO）

| 步骤 | 名称 | 目标 | 是否改运行时 |
| --- | --- | --- | --- |
| E4-1 | 入口合同与本地 only 边界 | 冻结 intake 产品包契约、`ENABLE_INTAKE_AI_EXTRACTION` 子 flag（默认 false）、原始案情零上送红线断言口径；零业务实现 | 否（仅声明 flag + 文档 + 可选契约常量/测试） |
| E4-2 | SearchProfile 产出与脱敏契约 | 实现「原始案情 → 脱敏 → SearchProfile」的纯函数（前端可复用 + 后端防御性校验）；本地脱敏规则 + 白名单收窄 | 是，但不接线端点/前端 |
| E4-3 | intake 后端包与消费 E3 服务 | 新建 `app/intake/` 产品包 + gated 端点；接收已脱敏 SearchProfile，经 E3 `InternalSearchService` 出 CandidateRef；默认 false 降级 | 是，新增 gated 端点 |
| E4-4 | 前端录入页（本地 only） | 新增录入端前端页：本地输入原始案情 → 本地脱敏预览 → 用户确认 → 只发 SearchProfile；入口受 `VITE_ENABLE_INTAKE` 门控 | 是，新增前端（gated） |
| E4-5 | 消费边界与护栏守门 | 加守门测试：原始案情零上送、intake 不直连检索底层、CandidateRef 白名单、AI 增强子开关默认 off 无接线、产品包不互相 import | 只加守门 |
| E4-6 | E4 验收与下一步门禁 | 生成 release gate + parameter version + 验收报告，判定是否允许进入 E5 法条检索 | 否 |

> 顺序依赖：E4-1 是合同冻结步（对标 E-1/E3-1），必须先做。E4-2 脱敏纯函数是 E4-3/E4-4 的共同依赖。E4-3（后端）与 E4-4（前端）严格串行（沿用「并行 bash 会冲突、必须串行验证」环境约束）。E4-5 守门固化纪律。E4-6 收官判定。

- 服务端在 `ENABLE_INTAKE_AI_EXTRACTION=false` 时出现任何 AI 增强 on 路径，或在任何时候接收未脱敏原始案情。
- 新建 statute/drafting/casebook 产品包，或 intake 与其他产品包互相 import。
- intake 绕过 `app.kernel.rag` 公开面直连 retrieval/rerank/summary，或复制检索主路径。
- CandidateRef 出现正文、summary/highlight 文本、matched_text、chunk_text、full_text、content、body；或缺 source_anchors 仍暴露下游。
- 新增未鉴权对外端点；intake 端点鉴权/租户隔离弱于 `/api/search`。
- `/api/search` 响应结构、错误码、降级行为无说明地改变；排序、召回、source selection、rerank 默认被改。
- 任一 E-1/M4/M5 flag、`ENABLE_INTAKE`、`ENABLE_INTAKE_AI_EXTRACTION` 默认被改为 `true`。
- qrels/label/relevance 或 query id/case id 特判进入运行时。
- 测试或产物写入真实长案情、裁判正文或真实 PII / 密钥。

## 11. E4 完成后的衔接

- E4 GO 后，E5 才允许开始做法条法规检索（statute/）。E5 复用同一向量库 + 法条标注，命中带锚点、法条↔类案互跳；同样经 E3 内部服务消费检索，不直连底层。
- SearchProfile / CandidateRef 契约（E-1 §3）在 E4 已首次落地消费；E5~E7 各产品只能用、不能私自加字段，要改字段须回文档 16/17 走合同变更。
- 原始案情零上送红线（E4 首次落为可执行边界）成为 E5~E8 常驻护栏；任何产品都不得把原始案情/PII 上送或持久化。
- `ENABLE_INTAKE_AI_EXTRACTION` 作为「服务端增强」的预留接口，后期实现时必须单独走一轮 Go/No-Go：前置「用户确认 + 已脱敏」、只接收脱敏文本、用完即丢、零持久化、日志不写正文，并补守门测试。
- 5 个产品 flag 中 `ENABLE_INTAKE` 在 E4 首次实现 on 路径但默认仍 false；statute/drafting/casebook 三个 flag 仍全程默认 false、不接线，待 E5~E7。
