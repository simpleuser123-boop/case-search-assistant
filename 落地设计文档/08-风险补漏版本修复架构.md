# 风险补漏版本修复架构

本文档承接 Day 3 第 8、9、10 步测试结果，用于定义下一轮「补漏修复版」的技术架构、执行顺序和验收门禁。该版本目标不是新增业务能力，而是把当前 MVP 从「能跑通」修到「可稳定内测、可评测决策、可回滚发布」。

## 1. 版本定位

| 项 | 内容 |
| --- | --- |
| 建议版本名 | M1.1 风险补漏修复版 |
| 版本目标 | 修复性能、评测、运行时一致性和 DB 健康风险 |
| 允许进入范围 | 本机和内部基础搜索验证 |
| 不允许进入范围 | 对外灰度、启用新排序、打开 LLM 改写、打开 LLM 摘要、打开扩展检索 |
| 默认策略 | 四个增强开关保持关闭：`ENABLE_QUERY_REWRITE=false`、`ENABLE_WEIGHTED_RERANK=false`、`ENABLE_SUMMARY=false`、`ENABLE_EXPANDED_SEARCH=false` |

## 2. 当前测试结论

### 2.1 已通过项

| 检查项 | 结果 |
| --- | --- |
| 后端单测 | `108 passed` |
| 前端单测 | `39 passed` |
| 前端生产构建 | 通过 |
| 真实 E2E | 通过，覆盖首页提交、结果页、详情抽屉、重搜、埋点脱敏 |
| 回滚演练 | 通过，最大恢复耗时 `21ms` |
| Chroma/Ollama | 可用，Chroma chunk 数 `9621` |

### 2.2 阻塞和风险

| 风险 | 证据 | 影响 | 当前决策 |
| --- | --- | --- | --- |
| 性能 P95 超线 | 性能 smoke `P95=3545ms`，慢点为 `embedding_duration_ms P95=3506ms` | 不满足 `<3s` 门槛 | 不进入外部灰度 |
| 离线评测阻塞 | LeCaRDv2 candidate corpus 缺失，rerank 无法对 qrels 评测 | 无法证明新排序优于基线 | `enable_new_rerank=false` |
| 数据集不匹配 | 产品检索返回 JuDGE/Chroma/BM25 case_id，LeCaRDv2 qrels 使用另一套 candidate id | 产品链路结果无法直接计算 Precision@5/NDCG@10 | 需要产品本地评测集或可靠 ID 映射 |
| 运行时不一致 | 源码 `/health` 有 `feature_flags`，当前 8000 端口旧进程响应没有该字段 | 灰度前无法确认运行的是当前源码 | 需要运行时预检和重启验收 |
| DB 不可达 | `/health` 中 `db_reachable=false` | 不影响基础搜索，但会阻塞历史、收藏、反馈、事件沉淀 | 下个版本先修健康检查或明确降级 |

## 3. 修复总原则

1. 不扩大业务范围：不做报告生成、收藏、历史、团队空间、外部部署。
2. 不默认启用增强能力：新排序、查询改写、摘要和扩展检索继续关闭。
3. 不重建向量索引，除非明确证明索引损坏或模型版本不一致。
4. 不删除现有数据，不清空 Chroma，不覆盖评测报告原件。
5. 原始案情不得写入日志、埋点、评测报告或 DB；只允许记录 `input_hash`、`input_length`、枚举原因和耗时。
6. 每个修复步骤必须有自动化验收，不能只靠手工观察页面。

## 4. 修复架构

### 4.1 运行时一致性层

目标：确保本机运行的 API 进程就是当前源码，且灰度前能从 `/health` 明确看到功能开关和依赖状态。

建议新增或增强：

| 模块 | 建议位置 | 责任 |
| --- | --- | --- |
| Runtime metadata | `apps/api/app/core/runtime.py` | 生成 `app_version`、`source_root`、`process_id`、`started_at`、`config_digest` |
| Health schema | `apps/api/app/api/health.py` | 返回 `feature_flags`、运行时元数据、DB/Ollama/Chroma 结构化依赖状态 |
| Runtime preflight | `apps/api/scripts/runtime_preflight.py` | 检查 live `/health` 与源码 TestClient `/health` 是否字段一致 |
| Windows process report | `apps/api/scripts/report_runtime_process.ps1` 或 Python 脚本 | 只报告占用 8000 的进程 PID、命令行和源码路径，不自动杀进程 |

验收标准：

- Live `GET http://127.0.0.1:8000/health` 返回 `feature_flags`。
- TestClient `/health` 与 live `/health` 的关键字段一致。
- `/health` 不泄露真实密钥值。
- 能明确看到当前 `process_id`、`source_root`、`started_at`。
- 若 live 进程不是当前项目路径，preflight 返回失败并提示重启。

### 4.2 性能稳定层

目标：把基础搜索链路 P95 拉回 `<3s`，并把冷启动耗时与稳定态耗时分开统计。

建议改造：

| 模块 | 建议位置 | 修复点 |
| --- | --- | --- |
| Embedding warmup | `apps/api/app/retrieval/embedding.py` 或启动事件 | API 启动后用固定短文本预热 Ollama bge-m3 |
| Query embedding timeout | `apps/api/app/retrieval/service.py` | embedding 超时后触发 BM25 兜底，不让前端白屏 |
| In-memory query vector cache | `apps/api/app/retrieval/embedding_cache.py` | 以 `input_hash + model_version` 为 key，短 TTL，只存内存，不落盘 |
| Performance smoke | `apps/api/scripts/day3_7_3_performance_smoke.py` | 增加 cold sample 和 warm sample 分组，避免首个冷启动污染稳定态 P95 |
| Stop-loss report | `docs/development/*performance*.json` | 输出具体慢阶段和建议关闭项 |

性能策略：

```text
请求进入
  -> 查询清洗
  -> 查询向量缓存命中：直接召回
  -> 未命中：调用 Ollama embedding
  -> embedding 超时或失败：BM25 兜底
  -> Chroma 召回成功：正常返回
  -> Chroma 失败：BM25 兜底
  -> 结果返回 degraded_reasons，不阻塞页面
```

验收标准：

- Warm P95 `<3s`。
- 首次 cold sample 单独记录，不纳入稳定态 P95。
- `embedding_duration_ms`、`retrieval_duration_ms`、`total_duration_ms` 均出现在响应和性能报告中。
- embedding 超时能返回 BM25 兜底结果或明确空状态，不返回 500。
- 前端错误态、降级态不白屏。

### 4.3 评测可信层

目标：让排序是否启用有可靠证据，而不是依赖感觉或不可比数据。

建议两条线并行：

| 线 | 目的 | 修复方式 |
| --- | --- | --- |
| LeCaRDv2 标准评测线 | 复现已有 BM25 baseline 和 qrels 指标 | 修复 `LECARDV2_CANDIDATE_CORPUS` 路径，检查 candidate doc 数和 qrels overlap |
| 产品本地评测线 | 评估当前 JuDGE/Chroma 产品语料 | 建立 20-50 条产品 query、人工相关 case 标注、可重复 Precision@5/NDCG@10 |

建议新增或增强：

| 模块 | 建议位置 | 责任 |
| --- | --- | --- |
| Eval corpus preflight | `apps/api/scripts/eval_corpus_preflight.py` | 检查 query、qrels、candidate corpus、ID overlap |
| Product eval set | `data/eval/product_eval_queries.jsonl`、`data/eval/product_eval_qrels.jsonl` | 存放当前产品语料可比评测集 |
| Product eval runner | `apps/api/app/eval/product_eval.py` | 对当前检索链路计算 Precision@5、NDCG@10、Top10 命中 |
| Bad case report | `data/eval/bad_cases_*.json` | 标注召回缺失、改写错误、案由误加权、段落匹配错误等原因 |

验收标准：

- LeCaRDv2 线：candidate corpus 可读，doc 数大于 0，qrels overlap 大于 0。
- 产品本地线：至少 20 条 query，至少 10 条有人工相关性标注。
- 输出 baseline 与 current 的 Precision@5、NDCG@10、Top10 hit rate。
- 新排序只有在 Precision@5 不下降、NDCG@10 正向且 Top10 命中接近或达到 60% 后，才允许进入灰度候选。
- 若评测仍阻塞，`ENABLE_WEIGHTED_RERANK` 继续保持关闭。

### 4.4 DB 健康层

目标：明确 DB 在当前版本中的职责。如果只做基础搜索，DB 可降级；如果进入历史、反馈、收藏或事件沉淀，DB 必须可用。

建议改造：

| 模块 | 建议位置 | 修复点 |
| --- | --- | --- |
| DB health detail | `apps/api/app/api/health.py` | 返回 DB driver、host、database、reachable、degraded_reason，不返回密码 |
| DB smoke | `apps/api/scripts/db_smoke.py` | 检查连接、迁移版本、基本读写能力 |
| Migration check | Alembic 或 SQLModel 初始化脚本 | 确认业务表存在，迁移可重复执行 |
| Event fallback | `apps/api/app/api/events.py` | DB 不可达时返回可理解降级，不丢主搜索链路 |

DB 版本策略：

| 场景 | DB 要求 | 行为 |
| --- | --- | --- |
| 基础搜索 | 可选 | DB 不可达时 `/health` 标记 degraded，但 `/api/search` 可继续 |
| 埋点持久化 | 必须 | DB 不可达时事件只返回降级或写本地脱敏缓冲，不写原始 query |
| 历史/收藏/反馈 | 必须 | 不允许进入下一业务版本前置开发 |

验收标准：

- `db_reachable=true`，或明确记录为基础搜索可接受降级。
- DB health 不泄露账号密码。
- DB smoke 能证明迁移和最小读写成功。
- 事件链路不写原始案情。

### 4.5 发布门禁层

目标：把测试、性能、评测、回滚和运行时预检聚合为一个 Go/No-Go 结果。

建议新增：

| 模块 | 建议位置 | 责任 |
| --- | --- | --- |
| Release gate script | `scripts/release_gate.ps1` 或 `apps/api/scripts/release_gate.py` | 顺序执行测试、性能、评测、回滚和 health preflight |
| Gate report | `docs/development/m1.1-risk-fix-gate-*.json` | 汇总所有门禁状态 |
| Gate summary | `docs/development/m1.1-risk-fix-gate-*.md` | 生成可读 Go/No-Go 结论 |

建议门禁：

| 门禁 | Go 条件 |
| --- | --- |
| 后端测试 | `pytest` 全通过 |
| 前端测试 | `npm test` 全通过 |
| 构建 | `npm run build` 通过 |
| 真实 E2E | Day3 E2E smoke 通过 |
| Runtime preflight | live `/health` 和源码一致，含 `feature_flags` |
| 性能 | warm P95 `<3s` |
| 评测 | 至少有一条可比评测线不阻塞 |
| 回滚 | 四个 flag 演练通过，恢复 `<60s` |
| 隐私 | 日志、埋点、报告不含原始 query |

## 5. 执行顺序

| 顺序 | 编号 | 任务 | 是否阻塞下步 | 主要交付物 |
| --- | --- | --- | --- | --- |
| 1 | R1 | 运行时一致性和 health 预检修复 | 是 | `/health` 增强、runtime preflight、旧进程识别 |

| 6 | R6 | 复测第 8、9、10 步并输出 Go/No-Go | 是 | 修复版验收报告 |

## 6. 分步骤系统提示词生成口径

### R1 系统提示词标题


范围：

- 只修 `/health`、运行时元数据、live/TestClient 一致性检查和端口进程报告。
- 不改检索排序、不改 Chroma、不改评测、不修 DB 业务表。

### R2 系统提示词标题

```text
类案检索助手 M1.1-R2：DB 健康检查与最小连接 Smoke
```

范围：

- 只修 DB health detail、DB smoke、DB 不可达时的降级口径。
- 不做历史、收藏、反馈、报告沉淀。

### R3 系统提示词标题

```text
类案检索助手 M1.1-R3：Embedding 性能、预热与 BM25 兜底修复
```

范围：

- 只修 embedding 预热、超时、短 TTL 内存缓存、性能 smoke 分组和降级返回。
- 不启用查询改写、不启用摘要、不启用新排序。

### R4 系统提示词标题

```text
类案检索助手 M1.1-R4：离线评测语料与产品本地评测集修复
```

范围：

- 只修 LeCaRDv2 corpus preflight、产品本地评测 query/qrels、bad case 输出。


范围：

- 只聚合已有测试、E2E、性能、评测、回滚和 runtime preflight。
- 不改业务逻辑，不改默认 feature flags。

### R6 系统提示词标题

```text
类案检索助手 M1.1-R6：补漏修复版第 8、9、10 步复测与结论
```

范围：

- 只复测联调、评测、性能、回滚和灰度门禁。
- 输出结论：内部基础搜索 Go/No-Go、外部灰度 Go/No-Go、新排序 Go/No-Go。

## 7. 最终 Definition of Done

M1.1 风险补漏修复版完成必须同时满足：

- live `/health` 返回当前源码的 `feature_flags` 和运行时元数据。
- 后端、前端、构建、真实 E2E 全通过。
- Warm P95 `<3s`，且慢阶段可定位。
- 评测不再停留在「无语料不可比」状态，至少产品本地评测线可跑。
- 回滚演练通过，恢复 `<60s`，不需要重建向量索引。
- DB 状态被修复为可达，或被明确标记为基础搜索可接受降级且不阻塞主链路。
- 所有日志、埋点和报告不包含原始案情。
- 默认仍不启用新排序、查询改写、LLM 摘要和扩展检索。

## 8. Go/No-Go 规则

| 结果 | 决策 |
| --- | --- |
| 主链路通过、性能达标、评测可比、回滚通过 | 可以进入内部增强能力灰度准备 |
| 主链路通过、性能达标、评测仍不证明新排序正向 | 只允许内部基础搜索，不启用新排序 |
| 性能 P95 仍超过 3s | 不对外灰度，继续修 embedding 或降级策略 |
| Runtime preflight 失败 | 不灰度，先重启或修复运行时一致性 |
| DB 不可达且下一版本需要历史/反馈/收藏 | 不进入该业务版本 |
| 回滚演练失败 | 不灰度上线 |
