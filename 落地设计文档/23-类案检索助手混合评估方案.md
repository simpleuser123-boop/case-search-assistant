# 类案检索助手混合评估方案

| 属性 | 内容 |
| --- | --- |
| 文档版本 | v0.1 |
| 适用项目 | 类案检索助手 |
| 方案定位 | 用于定义评分器标准、评估方案和后续评估系统建设 |
| 评估方式 | 规则评分 + qrels 离线评估 + LLM Judge + 法律人工评审 + 线上行为反馈 |
| 主要依据 | `应用闭环：Agent评估与迭代策略.pptx`、PRD、演进计划、数据层设计、M2 可信检索文档、现有 `data/eval` 评估产物 |

---

## 1. 评估目标

本项目评估的核心不是判断“AI 回答得像不像”，而是判断律师在真实类案检索任务中是否能更快找到事实相似、可核验、边界清楚的裁判文书。

评估体系需要回答四个问题：

1. 用户用自然语言描述案情后，系统是否能找到事实相似的类案。
2. 排序是否把真正可用的类案放在 Top5 / Top10 前列。
3. 摘要、高亮、相似理由、风险提示是否可回溯到来源文书片段。
4. 系统是否遵守法律产品边界：不承诺查全、不泄露隐私、不用 qrels 或人工标签做线上特判。

最终评估结论应能支持以下决策：

```text
这一版检索 / 排序 / 摘要 / 可信展示能力，是否比上一版更好？
是否达到内测 / 发布 / 进入下一阶段的门禁？
失败主要发生在召回、排序、摘要、来源锚点、性能，还是产品话术？
```

## 2. 评估架构

参考 `应用闭环：Agent评估与迭代策略.pptx` 中的四个核心组件，本项目评估闭环定义为：

| 组件 | 本项目落地 |
| --- | --- |
| 测试集 Dataset | 自然语言案情、关键词案情、复核遗漏、不利案例、无效输入、边界 case |
| 任务执行 Task Execution | 记录 query 改写、召回候选、重排、摘要、高亮、来源锚点、降级状态 |
| 评分器 Scorer | 规则评分器 + qrels 离线评分器 + LLM Judge + 法律人工评审 |
| 汇总与决策 Aggregation | Precision@5、NDCG@10、Top10 Hit Rate、锚点覆盖率、隐私/话术门禁、回归对比 |

评估系统不应只评最终结果页，而要记录完整 trace：

```text
query
-> query_rewrite
-> retrieval candidates
-> dedupe
-> rerank
-> summary / highlight
-> frontend response
-> user feedback
```

## 3. 评估范围分层

为避免一个总分掩盖问题，评估范围拆成五层：

| 层级 | 评估对象 | 核心问题 |
| --- | --- | --- |
| L1 数据质量 | 文书、chunk、元数据、索引 | 能不能被可靠检索 |
| L2 召回质量 | Top50 / Top100 候选池 | 相关案例有没有被召回 |
| L3 排序质量 | Top5 / Top10 结果 | 好案例是否排在前面 |
| L4 可信生成 | 摘要、高亮、相似理由、风险提示 | 是否忠于来源、可核验 |
| L5 产品体验 | 搜索完成率、点击、二次搜索、反馈 | 是否真的帮助律师更快判断 |

### 3.1 L1 数据质量

数据质量评估用于确认文书和索引是否具备被检索、被引用、被评估的基础条件。

| 指标 | 建议门槛 |
| --- | --- |
| 案号缺失率 | < 2% |
| 法院缺失率 | < 2% |
| 裁判日期缺失率 | < 5% |
| chunk 空文本率 | < 1% |
| 重复文书率 | < 3% |
| `本院查明` 识别成功率 | MVP >= 60%，后续 >= 80% |
| embedding 成功率 | >= 99% |
| 向量模型版本一致性 | 100% |

### 3.2 L2 召回质量

召回层主要判断“标准相关案例是否进入候选池”。如果相关案例没有被召回，后续排序无法补救。

| 指标 | 说明 |
| --- | --- |
| Recall@50 | 标准相关案例是否进入 Top50 候选池 |
| Recall@100 | 用于判断召回上限 |
| Candidate Count | 候选数量是否异常偏低 |
| Recall Miss Count | 有标准答案但候选池完全未覆盖的 query 数 |
| Retrieval Source Coverage | BM25、向量、改写 query、多路召回各自贡献 |

### 3.3 L3 排序质量

排序层判断好案例是否被放到用户最先看到的位置。

| 指标 | 说明 |
| --- | --- |
| Precision@5 | Top5 中可用类案比例 |
| Top10 Hit Rate | Top10 中是否命中可用类案 |
| NDCG@10 | 好案例是否排在前面 |
| MRR@10 | 第一个可用类案出现得有多早 |
| Duplicate Rate@10 | 同案、同 chunk 变体是否挤占结果 |
| Regression Query Count | 相比 baseline 出现回退的 query 数 |

### 3.4 L4 可信生成

可信生成层评估所有 AI 加工内容是否可核验。

| 指标 | 建议门槛 |
| --- | --- |
| 来源锚点覆盖率 | 100% |
| 无来源 AI 内容展示数 | 0 |
| 摘要忠实度 | LLM Judge + 人工抽检 |
| 高亮片段一致性 | 高亮必须来自命中 chunk |
| 风险提示来源覆盖 | 100% |
| 绝对查全话术违规 | 0 |

### 3.5 L5 产品体验

产品体验层用于判断检索结果是否真的帮助律师完成任务。

| 指标 | 说明 |
| --- | --- |
| 搜索完成率 | 请求成功返回结果的比例 |
| 详情点击率 | 用户是否愿意进一步查看结果 |
| 复制案号 / 原文跳转率 | 结果是否进入工作流 |
| 二次搜索率 | 高二次搜索率可能意味着首轮结果不满意 |
| 扩展检索点击率 | 用户是否有遗漏焦虑 |
| 相关 / 不相关反馈 | 只作为脱敏观测，不直接改线上排序 |

## 4. 测试集设计

当前 `data/eval/product_eval_queries.jsonl` 已具备产品评估集基础，但样本更偏案由或关键词检索。后续必须补充事实情节型自然语言 query，否则无法验证“口语案情 -> 高相似类案”的产品主张。

建议建立四类测试集：

| 测试集 | 规模建议 | 用途 |
| --- | ---: | --- |
| 常青回归集 | 80-120 条 | 每次排序、检索、Prompt 改动都跑 |
| 黄金人工集 | 30-50 条 | 法律人工标注，用于校准 LLM Judge |
| 专项坏例集 | 30-80 条 | 收录历史失败 case：召回 miss、案由相同但事实不相关、锚点缺失 |
| 发布验收集 | 150-300 条 | 发布前跑，含真实脱敏 query 和高风险场景 |

### 4.1 样本类型

| query 类型 | 示例方向 |
| --- | --- |
| 口语化案情 | “对方明知道产品有问题还继续卖，使用后受伤” |
| 法言法语案情 | “产品责任纠纷中销售者明知缺陷仍销售导致损害” |
| 新型事实 | AI 生成内容、平台交易、区块链道具、直播带货 |
| 争议焦点型 | 是否构成欺诈、是否承担连带责任、过错比例 |
| 负面 / 不利类案 | 找支持对方观点或风险提示的案例 |
| 低信息输入 | “这个案子我觉得能赢吗” |
| 噪声输入 | 人名、身份证号、长文本、错别字、混合口语 |

### 4.2 测试集版本管理

每个评估集必须记录：

```json
{
  "eval_set_version": "legal_case_eval_v1",
  "created_at": "2026-06-22",
  "query_count": 100,
  "qrels_count": 300,
  "source": "manual + real_desensitized + bad_cases",
  "scope": "M1/M2 regression",
  "privacy": {
    "raw_user_query_included": false,
    "desensitized": true
  }
}
```

测试集更新必须记录原因，避免评估结果变化无法归因。

## 5. 相关性标注标准

人工标注和 LLM Judge 统一采用 0-4 分。

| 分数 | 定义 | 是否算命中 |
| ---: | --- | --- |
| 4 | 高度类案：核心事实、争议焦点、裁判逻辑均高度相似，可优先引用 | 是 |
| 3 | 可用类案：主要事实和法律争点相似，对检索任务有实质参考价值 | 是 |
| 2 | 部分相关：案由或局部事实相似，但关键争点不同 | 否，可进低置信候选 |
| 1 | 弱相关：关键词、案由、主体类型相近，但事实相似度低 | 否 |
| 0 | 不相关、误召回、明显噪声结果 | 否 |

核心判定原则：

```text
案由相同不等于类案相似。
关键词命中不等于事实相似。
法院层级高不应覆盖事实不相似。
摘要说得合理但无来源，视为不可信。
```

### 5.1 单案例 100 分评分模型

| 维度 | 权重 | 判断重点 |
| --- | ---: | --- |
| 事实情节相似度 | 35 | 行为过程、因果链、主体关系、损害结果是否相似 |
| 法律争点一致性 | 20 | 争议焦点是否一致 |
| 裁判逻辑可迁移性 | 15 | 法院说理是否能迁移到用户案件 |
| 结果 / 立场可用性 | 10 | 是否支持或反向提示用户关注的主张 |
| 权威性与时效性 | 10 | 法院层级、审级、日期、地域是否有参考价值 |
| 来源可核验性 | 10 | 是否能回到 `case_id` 和 `source_chunk_id` |

分数换算：

| 100 分制 | relevance |
| --- | ---: |
| 85-100 | 4 |
| 65-84 | 3 |
| 40-64 | 2 |
| 15-39 | 1 |
| 0-14 | 0 |

## 6. 混合评分器设计

本项目不应使用单一大模型评分器，而应采用分层混合评分。

| 评分器 | 评什么 | 实现方式 |
| --- | --- | --- |
| 规则评分器 | schema、来源锚点、隐私、禁用话术、性能、重复结果 | 稳定自动化，作为强门禁 |
| qrels 离线评分器 | Precision@5、NDCG@10、Top10 Hit Rate、Recall@50 | 复用现有 `product_eval_qrels.jsonl` 并扩展 |
| LLM Judge | 事实相似度、争点一致性、摘要忠实度、风险提示合理性 | 基于固定 rubric，低 temperature，需人工校准 |
| 人工评审 | 黄金集标注、争议样本裁决、LLM Judge 校准 | 法律背景人员抽样 |
| 在线行为评分 | 点击、复制、二次搜索、反馈、扩展检索 | 产品观测信号，不直接作为真值 |

### 6.1 规则评分器

规则评分器适合做强约束、可复现、低成本的检查。

| 检查项 | 规则 |
| --- | --- |
| API schema | 必填字段完整，类型正确 |
| source anchors | AI 加工内容必须绑定 `case_id` 和 `source_chunk_id` |
| forbidden claims | 不得出现“已查全”“保证无遗漏”“查全率”等绝对话术 |
| privacy | 日志和报告不得保存原始 query、候选正文、chunk 正文 |
| duplicate | Top10 不得被同案重复结果挤占 |
| latency | P95 检索链路必须低于门槛 |
| feature flag | weighted rerank、expanded search 等必须可回滚 |

### 6.2 qrels 离线评分器

用于可重复衡量召回和排序。

核心输入：

```text
data/eval/*queries.jsonl
data/eval/*qrels.jsonl
retrieval run output
baseline run output
```

核心输出：

```text
Precision@5
NDCG@10
Top10 Hit Rate
Recall@50
MRR@10
Regression Query Count
Bad Case Report
```

### 6.3 LLM Judge

LLM Judge 用于语义维度判断，但不能直接作为真值。

建议评分维度：

| 维度 | 输出 |
| --- | --- |
| 事实情节相似度 | 0-35 |
| 法律争点一致性 | 0-20 |
| 裁判逻辑可迁移性 | 0-15 |
| 摘要忠实度 | pass / fail + reason |
| 风险提示合理性 | pass / fail + reason |
| relevance | 0-4 |

LLM Judge 输入必须限制为：

```text
脱敏 query
候选案例必要元数据
命中 source chunk
当前系统摘要 / 高亮
评分 rubric
```

不得让 Judge 基于常识或全量文书自由脑补。

### 6.4 人工评审

人工评审用于建立黄金标准和校准模型评分器。

建议机制：

1. 黄金集 30-50 条 query-result pair。
2. 每条至少一名法律背景评审标注。
3. 争议样本进入二次评审。
4. 人工理由只写判断依据，不写用户原始隐私信息。
5. 人工集每月或每个大版本重新抽样校准。

## 7. 核心指标与门槛

### 7.1 M1 检索 MVP 指标

| 指标 | 门槛 |
| --- | ---: |
| Top10 Hit Rate | >= 60% |
| Precision@5 | 不低于 baseline，目标提升 >= 10% |
| NDCG@10 | 不低于 baseline，且核心场景不回退 |
| Recall@50 | 监控召回上限，定位召回问题 |

| 指标 | 当前参考值 |

| NDCG@10 | 0.1954 |
| Precision@5 | 0.056 |
| baseline NDCG@10 | 0.134 |
| baseline Precision@5 | 0.032 |

## 8. 强门禁 NO-GO

以下问题不做扣分处理，而是直接 `NO_GO`：

1. AI 摘要、相似理由、风险提示无 `case_id` / `source_chunk_id`。
2. 出现“已查全”“保证无遗漏”“查全率 99%”等绝对承诺。
3. 用户原始案情被写入日志、报告、评测产物。
4. qrels、人工标签、query id、case id 被用于线上排序特判。
5. 新排序 Precision@5 或 NDCG@10 低于 baseline。
6. 关键场景 Top10 无任何可用案例。
7. 低置信候选挤占主结果排序。
8. 摘要或高亮内容无法从原文片段支持。
9. 默认打开仍未通过门禁的 weighted rerank 或 expanded search。
10. 评估产物包含候选正文、chunk 正文或用户隐私。

## 9. 评估运行机制

参考 PPT 中的“四层评估体系”，本项目按以下频率运行：

| 层级 | 触发时机 | 数据 | 目标 |
| --- | --- | --- | --- |
| 单元级评估 | 改 query rewrite、retrieval、rerank、summary 时 | 10-30 条专项样本 | 快速发现明显问题 |
| 回归评估 | 每次合并前 | 常青集 80-120 条 | 防止已有能力回退 |
| 发布评估 | 内测 / 上线前 | 发布集 150-300 条 | 判断是否达发布门禁 |
| 线上监控 | 上线后持续 | 脱敏真实流量抽样 | 发现测试集未覆盖失败模式 |

### 9.1 日常迭代协议

日常迭代可采用低成本协议：

```text
固定小测试集
单次运行
快速输出核心指标和 bad cases
只用于筛查明显问题
```

### 9.2 发布前协议

发布前必须采用更严格协议：

```text
固定测试集
固定索引版本
固定评分器版本
同批次跑 baseline 和 candidate
关键样本重复运行 3 次
输出均值、最差值、回归 query 列表
```

### 9.3 非确定性处理

| 非确定性来源 | 处理方式 |
| --- | --- |
| LLM 改写随机性 | 低 temperature；关键 query 重复运行 |
| 外部模型变化 | 记录 model/version；同批次比较 |
| 向量索引变化 | 固定 collection 与 embedding model version |
| LLM Judge 波动 | Judge temperature 设低；黄金集定期校准 |
| 线上真实数据波动 | 发布评估和线上监控分开看 |

## 10. LLM Judge 校准方案

LLM Judge 不能一上来就作为真值，必须先校准。

### 10.1 校准流程

1. 抽取 50 条 query-result pair。
2. 由法律人工按 0-4 分标注。
3. 用同一批样本跑 LLM Judge。
4. 计算一致率、严重偏差率。
5. 一致率低于 80% 不允许自动进入主评估。
6. 对偏差样本归因：偏好长文本、误把案由当事实、忽略争点、被摘要误导。
7. 每月或每次产品能力变化后重新校准。

### 10.2 一致性指标

| 指标 | 说明 |
| --- | --- |
| Exact Match | LLM relevance 与人工完全一致 |
| Adjacent Match | LLM 与人工差距 <= 1 |
| Severe Error Rate | 人工 <= 1 但 LLM >= 3，或人工 >= 3 但 LLM <= 1 |
| Pass/Fail Agreement | 是否同意“可用类案”判断 |

建议门槛：

| 指标 | 门槛 |
| --- | ---: |
| Pass/Fail Agreement | >= 80% |
| Severe Error Rate | <= 5% |
| Adjacent Match | >= 90% |

## 13. 阶段路线图

建议按四阶段推进，不一次做成复杂平台。

| 阶段 | 周期 | 关键产出 |
| --- | --- | --- |
| 阶段 1：最小闭环 | 1-2 周 | 50 条事实型 query、0-4 标签标准、qrels 指标、首版报告 |
| 阶段 2：混合评分 | 2-4 周 | 规则评分器、LLM Judge、人工黄金集、Judge 校准报告 |
| 阶段 3：嵌入研发流程 | 1-2 个月 | PR/合并前回归评估、baseline vs candidate 对比、NO-GO 门禁 |
| 阶段 4：线上闭环 | 持续 | 真实反馈脱敏入库、坏例自动入专项集、质量 dashboard |

### 13.1 阶段 1：最小闭环

目标是先拥有“量化对比”能力，而不是建立完美系统。

交付：

1. 新增事实型自然语言 query 50 条。
2. 人工标注 0-4 分 relevance。
3. 跑 baseline 与 current。
4. 输出 Precision@5、NDCG@10、Top10 Hit Rate。
5. 形成首版 bad case report。

成功标准：

```text
团队能够说清楚：
这一版比 baseline 好在哪里；
哪些 query 仍然失败；
失败是召回问题、排序问题，还是标注/数据问题。
```

### 13.2 阶段 2：混合评分

目标是降低人工成本，提高评估频率。

交付：

1. 规则评分器：schema、锚点、隐私、话术、性能。
2. LLM Judge：事实相似度、争点一致性、摘要忠实度。
3. 黄金人工集：30-50 条。
4. LLM Judge 校准报告。
5. 评估 JSONL 标准格式。

### 13.3 阶段 3：嵌入研发流程

目标是让评估成为开发工作流的一部分。

交付：

1. 每次检索、排序、摘要改动前后自动跑回归集。
2. baseline 与 candidate 同批次对比。
3. 生成 GO / NO-GO 结论。
4. feature flag 打开必须绑定评估结果。
5. bad case 自动沉淀到专项集。

### 13.4 阶段 4：线上闭环

目标是让真实用户反馈持续反哺测试集。

交付：

1. 脱敏记录相关 / 不相关反馈。
2. 监控二次搜索、扩展检索、详情点击、复制案号。
3. 线上失败 case 进入人工审核队列。
4. 审核后加入常青集或专项坏例集。
5. 建立质量 dashboard。

## 14. 当前项目的关键建议

1. 现有 `data/eval`、M1.3x robustness、Precision/NDCG/Top10 指标已经具备评估基础，但仍偏检索工程评测。
2. 下一步最重要的是补齐事实情节型自然语言评估集，否则无法证明产品主张。
3. M2 以后，评估重心必须从“搜得准”扩展为“搜得准 + 说得有来源 + 不制造查全幻觉”。
4. 评分器必须分层，法律类案检索不适合用单一 LLM Judge 兜底。
5. 第一版评估系统不需要复杂 UI，先把 `eval run -> JSON report -> bad cases -> gate decision` 跑稳定。
6. qrels、人工标签、用户反馈只能用于离线评估和后续分析，不允许直接进入线上排序特判。
7. 所有评估报告默认不得包含用户原始案情、候选正文、chunk 正文。
