## 统一产品口径

### 核心用户

MVP 只服务一个主用户：执业 3-8 年、民商事诉讼方向、经常承担类案检索任务的律师。文档沿用现有 PRD 中的「判例猎人王律」画像。

### 核心价值

用户用 100-300 字口语化案情描述，不再手动拆关键词，系统返回事实情节高度相似的裁判文书，并在结果列表中直接展示相似事实摘要、高亮片段、法院与审级信息。

### 研发主线

先解决「找得准」，再解决「查得全」，最后解决「读得快」和「可沉淀」。

1. 精准度：查询改写、语义召回、事实相似度排序、评测基线。
2. 信任感：引用来源、数据覆盖声明、低置信度候选、风险提示。
3. 阅读效率：关键事实摘要、裁判要旨、详情面板、对比视图。
4. 工作流：历史、收藏、导出、报告、团队协作。

### 当前范围判断

现有项目目前是文档型项目，尚未看到前端或后端代码工程。因此本组文档默认从 0 到 1 搭建工程。Demo 阶段技术栈建议调整为：React + TypeScript、Python FastAPI、DeepSeek API、LangChain、Chroma、本地 bge-m3（Ollama 部署）。向量模型经本地 Ollama 服务（`OLLAMA_BASE_URL`）调用，数据不出本机；Milvus 暂作为后续规模化候选，而不是 Demo 首选。

## 决策基线

| 决策项 | 当前建议 |
| --- | --- |
| MVP 北极星指标 | Top10 主观命中率 >= 60%，单次检索耗时 <= 10 分钟 |
| 工程优先级 | 检索质量 > 可追溯信任 > 阅读提效 > 工作流闭环 |
| 首期技术目标 | 可运行、可评测、可回滚；不追求复杂模型训练 |
| 首期不做 | 完整报告生成、团队协作、多用户画像模式、复杂法院/法官倾向分析 |
| 首期必须保留 | 降级检索、埋点、评测集、隐私保护、功能开关 |

## 推荐阅读顺序

1. 先读演进计划，确认做什么和暂时不做什么。
2. 再读技术架构和数据层，确认系统如何跑通。
3. 然后读前端架构和设计系统，确认用户界面如何落地。
4. 最后读功能拆解和开发步骤，进入排期与执行。
