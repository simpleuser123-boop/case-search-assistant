## 1. E6 总目标

E6 的目标是落地 E 系列**第三个产品能力包**：文书辅助工作台。它在「检索 → 法律依据」之后接上「依据 → 文书骨架」这一段闭环：用户把一次检索沉淀下来的 `CandidateRef`（类案）/ `StatuteRef`（法条，经互跳）组装成一份**文书结构骨架**，每个引用都带**可核验来源锚点**，导出时强制**免责头**——但工作台**只组装锚定来源、不起草结论性正文、不输出胜负判断**。

```text
一组 CandidateRef（类案，来自检索/清单）+ 可选 StatuteRef（法条，经 E5 互跳）
  -> 用户在工作台编排 structure_skeleton（段落标题，非正文）
  -> 每个引用继承 source_anchors（case_id + source_chunk_id）/ statute_anchors（text_id）
  -> 用户自填短字段（note / tag），不起草正文、不下结论
  -> 沉淀为 DraftDescriptor（只存骨架 + 引用 + 短字段，零起草正文）
  -> 导出走 M4-5 免责头机制：只含元数据 + 来源链接 + 用户备注 + 免责声明
```

E6 的本质是**第三个产品包落地 + 把 M4-6 轻量报告骨架向前推进一步**，不是放开护栏，也不是让 AI 起草法律文书。E6 不做 casebook（E7），不改排序/召回/source selection/rerank 默认，不改 `/api/search`、E3 `InternalSearchService`、intake、statute 端点行为，不绕过内核公开面直连检索底层，不 import 其它产品包。

### 1.1 文书工作台与「AI 起草」的关键差异（E6 地基判断）

- **只组装、不起草**：工作台的产出是「结构骨架 + 锚定引用 + 用户自填短字段」，**不是**一篇可直接提交的代理词/诉状/答辩状/法律意见书。`structure_skeleton` 是**段落标题清单**（如「一、基本案情」「二、争议焦点」「三、参考类案」），**不是段落正文**。模型不得自动起草任何段落正文、不得补全结论、不得输出胜负判断或诉讼结果预测。
- **引用必带锚点、继承 M4 边界**：骨架中引用的每个 `CandidateRef`/`StatuteRef` 必须可追溯到 `source_anchors`（case_id + source_chunk_id）/ `statute_anchors`（text_id）；无锚点的内容不进入文书骨架与导出交付物（沿用 M4「无来源不进交付物」红线）。
- **持久层只存元数据 + 引用 + 短字段**：`DraftDescriptor` 持久化时只存 `draft_id`、`structure_skeleton`（标题）、`candidate_refs`（引用）、可选 `statute_refs`（引用）、用户 `note`/`tag`、时间戳/状态/reason code；**绝不存起草正文、裁判文书正文、候选/chunk 正文、原始案情**（沿用 M4 持久层隐私升级）。
- **导出强制免责头**：导出文件复用 M4-5 机制，头部含数据覆盖说明 + 免责声明（不承诺已查全、提示需人工复核、不构成法律意见、不预测裁判结果）；导出内容只含元数据、来源链接、用户备注，不含正文、不含胜负概率、不含确定性法律结论。

### 1.2 E6 涉及的 flag（沿用 E-1 五产品 flag，不新增产品级 flag）

| flag | 含义 | 默认 | E6 阶段状态 |
| --- | --- | --- | --- |
| `ENABLE_DRAFTING`（E-1 已冻结） | 文书工作台总开关：前端入口 + drafting 端点是否可用 | `false` | E6 实现其 on 路径（组装 + 持久化 + 导出），但默认仍 `false` |
| `VITE_ENABLE_DRAFTING`（E-1 已冻结） | 前端镜像：文书工作台入口/路由是否渲染 | `false` | 同上，默认仍 `false` |

> E6 **不新增产品级 flag**（与 E4 新增 `ENABLE_INTAKE_AI_EXTRACTION` 不同，与 E5 一致）。文书工作台明确**不引入任何 AI 起草开关**——「不起草正文」是结构性红线，不能用 flag 放开。`ENABLE_INTAKE`/`ENABLE_INTAKE_AI_EXTRACTION`/`ENABLE_STATUTE_SEARCH`/`ENABLE_CASEBOOK`/`ENABLE_ECOSYSTEM`/`ENABLE_WEIGHTED_RERANK` 等默认全程仍 false。任一默认 `true` 即 `NO_GO`。

### 1.3 E6 守的红线（在 M4/M5 + 生态 3 红线之上）

1. **只组装锚定来源、不起草结论**：工作台只产出结构骨架 + 锚定引用 + 用户短字段；模型/端点不自动起草段落正文、不补全结论、不输出胜负概率或诉讼结果判断。`structure_skeleton` 只含标题不含正文。
2. **引用必带锚点、无锚点不进交付物**：骨架/导出引用的 `CandidateRef`/`StatuteRef` 必须有 `source_anchors`/`statute_anchors`；无锚点内容降级不进入文书骨架与导出文件。
3. **DraftDescriptor 持久层零正文**：持久化只存骨架标题 + 引用 + 用户短字段 + 时间戳/状态；不存起草正文、裁判文书正文、候选/chunk 正文、原始案情；出现起草正文型字段即 `NO_GO`。
4. **drafting 经内核 / 既有服务消费，不直连底层、不 import 其它产品包**：drafting 产品包只依赖 `app.kernel` 公开面（需要重新取锚点内容时经 `app.kernel.rag`）；不深引 retrieval/rerank/summary，不复制检索主路径，不 import intake/statute/casebook。
5. **导出强制免责头 + 多租户隔离**：导出复用 M4-5 免责头机制；DraftDescriptor 持久化沿用 M5 多租户强隔离 + 对象级鉴权 + 默认私有；不改既有检索/法条行为。
6. 沿用 M4/M5 全部红线 + 生态 3 红线：仅锚定来源才展示 / 无来源降级 / 不落正文（裁判文书）/ 不按 case_id 特判 / 不输出胜负概率 / flag 默认 false / 关闭回上一态 / 多租户强隔离 / 凭据不代管；原始案情零上送 / 跨产品只走契约对象 / 回流 100% 聚合无正文。
