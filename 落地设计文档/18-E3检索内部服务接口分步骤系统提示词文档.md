> 用途：为多产品生态 E 系列的第三步 E-3「检索内部服务接口」拆分独立 Codex 会话提示词。
> 范围严格承接 `落地设计文档/16-多产品生态全链路闭环设计文档.md`、`落地设计文档/17-E系列分步骤文档.md`、E-1 合同冻结产物与 E-2b 内核迁移产物。
> 使用方式：每次只复制一个步骤的「新会话标题」和「系统提示词」到新会话，用户消息再补一句「现在开始 E3-x」。
> 本文档只提供执行提示词，不代表已经实现 E3。

---

## 1. E3 总目标

E3 的目标是让「检索助手」从一个外部搜索 API，进一步沉淀为多产品生态可复用的**内部检索服务接口**：

```text
SearchProfile（脱敏、白名单输入）
  -> 检索内部服务（统一查询处理 / 召回 / 排序 / 来源锚点 / 护栏）
  -> CandidateRef[]（只含元数据与 source_anchors，不含正文）
```

E3 完成后，后续 E4 案情录入端、E5 法条检索、E6 文书工作台、E7 案件协作台只能通过 E3 暴露的内部服务消费检索能力，不能绕过内核公开面直连 `retrieval`、`rerank`、`summary` 或现有 `/api/search` 端点。

E3 的本质是**内部服务化 + 契约收窄 + 行为回归**，不是新增产品入口。E3 不做 intake/statute/drafting/casebook，不注册新的对外 API，不做前端导航，不改排序策略，不扩语料，不重建索引。

---

## 2. E3 分步计划

| 步骤 | 名称 | 目标 | 是否改运行时 |
| --- | --- | --- | --- |
| E3-0 | 入场复核与阻塞清零 | 确认 E-2b 前置条件、前端截断文件已恢复、E-2b gate 可作为 E3 基线 | 否 |
| E3-1 | 内部服务契约冻结 | 冻结 `SearchProfile -> CandidateRef` 的 Python 内部接口、模型、sanitize 规则和测试 | 只新增契约，不接线 |
| E3-2 | 检索执行服务适配层 | 把现有检索链路封装为内部服务，输出 CandidateRef，同时保留富结果供现有 API 使用 | 是，但不得改变外部行为 |
| E3-3 | `/api/search` 消费内部服务 | 让现有搜索 API 复用 E3 服务，做逐位回归和错误/降级一致性校验 | 是，要求行为零变化 |
| E3-4 | 消费边界与护栏守门 | 增加 import 边界、正文泄露、CandidateRef 白名单、无产品包/无端点守门测试 | 只加守门 |
| E3-5 | E3 验收与下一步门禁 | 生成 release gate、parameter version、验收报告，判定是否允许进入 E4 | 否 |
