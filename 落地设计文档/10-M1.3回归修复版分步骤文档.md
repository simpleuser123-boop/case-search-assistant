# M1.3 回归修复版分步骤文档

本文档承接 `M1.2 Final Verification`：

- `docs/development/m1.2-final-verification-20260609-195835.md`
- `docs/development/m1.2-release-gate-20260609-195835.json`

M1.2 的正式结论是：基础搜索继续可用，但新 weighted rerank 不是灰度候选，不能进入 M2 可信检索准备。M1.3 因此不是 M2 功能实施版，而是 M1 阶段内的第二轮回归修复版。

M1.3 的目标是把 M1.2 遗留的 `METRIC_REGRESSION`、`RECALL_MISS` 和固定回归集退化样本处理到可进入下一阶段的门槛，再重新判断是否允许进入 M2 readiness。

---

## 1. 当前起点

依据 M1.2 最终验收，当前状态如下：

| 项 | 当前状态 |
| --- | --- |
| 基础搜索 | `GO`，继续可用 |
| M1.2 总门禁 | `CONDITIONAL_GO` |
| 新 weighted rerank 灰度候选 | `NO_GO` |
| 是否进入 M2 可信检索准备 | `NO_GO` |
| `ENABLE_WEIGHTED_RERANK` 默认状态 | `false`，必须继续关闭 |
| final product-local Precision@5 | baseline `0.032`，current `0.040` |
| final product-local NDCG@10 | baseline `0.134`，current `0.1371` |
| final product-local Top10 hit rate | baseline `0.48`，current `0.52` |
| 灰度候选 Top10 门槛 | `>= 0.60` |
| 固定回归 before -> after | `IMPROVED=3`，`STABLE=18`，`REGRESSED=4` |
| 固定回归 after vs baseline | `IMPROVED=5`，`STABLE=11`，`REGRESSED=9` |
| final bad cases | `METRIC_REGRESSION=9`，`RECALL_MISS=11` |
| 标准 rerank current 线 | blocked，缺少 LeCaRDv2 candidate corpus |

因此，M1.3 的第一优先级不是做可信检索功能，而是回答：

1. 哪些 query 的退化必须先消除。
2. 剩余 `RECALL_MISS` 是召回池缺失、候选合并截断、排序压低，还是 qrels/data mismatch。
3. 能否在不伤害成功样本的前提下把 Top10 hit rate 推到 `0.60`。
4. M1.3 结束时是否可以正式进入 M2 readiness。

---

## 2. 版本目标

### 2.1 主目标

把 M1.2 的“基础搜索可用但新排序不可灰度”推进到“回归集无明确退化，Top10 hit rate 达到灰度候选门槛，并能重新判断 M2 入口”。

### 2.2 成功标准

M1.3 结束时，至少要同时满足：

| 指标 | 门槛 |
| --- | --- |
| product-local eval | 可重复运行，无 blocked item |
| Precision@5 | `current >= baseline` |
| NDCG@10 | `current > baseline` |
| Top10 hit rate | `current >= 0.60` |
| 固定回归 before -> after | `REGRESSED=0` |
| 固定回归 after vs baseline | `REGRESSED=0` |
| final bad cases | `METRIC_REGRESSION=0`，`RECALL_MISS` 显著下降并有剩余解释 |
| 性能 | warm P95 `< 3s`，错误率 `0` 或有明确非阻断解释 |
| 回滚 | `ENABLE_WEIGHTED_RERANK=false` 可回到 `score_mode=base_retrieval`，恢复 `< 60s` |
| 默认开关 | `ENABLE_WEIGHTED_RERANK=false`，不因评测通过而直接默认开启 |

### 2.3 分层结论

M1.3 最终必须输出三类结论：

| 结论项 | 允许状态 |
| --- | --- |
| 基础搜索是否继续可用 | `GO` 或 `NO_GO` |
| 新 weighted rerank 是否允许成为灰度候选 | `GO` 或 `NO_GO` |
| 是否允许进入 M2 可信检索准备 | `GO` 或 `NO_GO` |

不允许写成“建议观察”“大致变好”“可以试试”。

### 2.4 本版本不做

- 不做 M2 的来源引用、引用锚点、覆盖声明、低置信度候选产品化扩面。
- 不做 M3 的裁判要旨摘要、事实对比、阅读提效扩展。
- 不做 M4 的检索历史、收藏、导出、类案报告。
- 不做多用户、团队协作、权限、商业化能力。
- 不为了通过门禁修改 qrels、删除坏样本、伪造指标、手工修结果。
- 不把 `ENABLE_WEIGHTED_RERANK` 默认改成 `true`。

---

## 3. 执行原则

1. 先消除明确退化，再追求整体提升。
2. 每个修复都必须能映射到具体 query id、case id、指标、标签或排序位置。
3. 召回修复和排序修复分开验收，避免混合归因。
4. 成功样本保护优先；不能为了修高价值 bad case 大面积伤害 typical success。
5. 新排序只作为离线候选或受控灰度候选，不直接默认上线。
6. 原始案情文本不得进入日志、报告正文或新增 JSON 明文字段。
7. 每一步都保留 baseline 对照和回滚路径。

---

## 4. 分步骤总览

建议把 M1.3 拆成 7 个独立步骤，每一步可以单开一个 Codex 会话执行：

| 步骤 | 标题 | 目标 |
| --- | --- | --- |
| M1.3-1 | 回归样本冻结与归因重标注 | 冻结 M1.2 final 证据，明确 4/9 退化和 11 个 recall miss 的主因 |
| M1.3-2 | 回归门禁与评测 runner 加固 | 把“无明确退化”做成硬门禁，防止只看 aggregate 指标 |
| M1.3-3 | 剩余 RECALL_MISS 定点修复 | 只修相关案例没进候选或被召回截断的问题 |
| M1.3-4 | 排序退化定点修复 | 只修固定回归中的排序退化，不扩大 rerank 能力 |
| M1.3-5 | 参数候选对比与定版 | 多候选参数横向评测，选出无退化版本或明确撤回 |
| M1.3-6 | 灰度候选预演复核 | 只复核开关、API、性能、回滚和候选门槛，不对外放量 |
| M1.3-7 | M1.3 验收与 M2 入口结论 | 输出 Go/No-Go，决定是否进入 M2 readiness |

## 5.1 步骤 M1.3-1：回归样本冻结与归因重标注

### 新会话标题

```text
类案检索助手 M1.3-1 回归样本冻结与归因重标注
```

### 目标

冻结 M1.2 final 证据，把 `REGRESSED`、`METRIC_REGRESSION`、`RECALL_MISS` 的交叉关系重标注清楚，形成 M1.3 的唯一修复清单。

### 开始前必须读取

- `docs/development/m1.2-final-verification-20260609-195835.md`
- `docs/development/m1.2-release-gate-20260609-195835.json`
- `docs/development/m1.2-regression-run-final-20260609-195835.json`
- `docs/development/m1.2-product-bad-cases-product-chain-final-20260609-195835.json`
- `docs/development/m1.2-parameter-version-final-20260609-195835.json`
- `data/eval/m1_2_regression_set_20260609.json`
- `data/eval/product_eval_queries.jsonl`
- `data/eval/product_eval_qrels.jsonl`

### 主要任务

1. 列出固定回归集中：
   - before -> after 的 `REGRESSED=4`
   - after vs baseline 的 `REGRESSED=9`
   - final bad cases 中 `METRIC_REGRESSION=9`
   - final bad cases 中 `RECALL_MISS=11`
2. 为每个 query 打主因标签：
   - `RECALL_POOL_MISS`
   - `CANDIDATE_MERGE_GATING`
   - `RERANK_SUPPRESSION`
   - `GUARD_OVER_CORRECTION`
   - `QUERY_MAPPING_NO_GAIN`
   - `QRELS_OR_DATA_MISMATCH`
   - `SUCCESS_SAMPLE_PROTECTION`
3. 标记每个 query 的修复优先级：
   - P0：导致 Top10 miss 或 fixed regression 的硬阻断
   - P1：Top10 hit 但 NDCG 明显低于 baseline
   - P2：只影响解释或边界样本
4. 输出 M1.3 唯一修复清单，后续步骤只能处理清单内问题。

### 产物

- `docs/development/m1.3-regression-triage-*.md`
- `docs/development/m1.3-regression-triage-*.json`

### 验收

- 能明确回答每个退化 query 属于召回、合并、排序、guard 过度、qrels/data mismatch 还是成功样本保护问题。
- 不包含原始 query、案情正文、候选正文或 chunk 正文。
- 不修改评测样本、qrels、参数或业务代码。

### 止损

如果无法给出可靠归因，不进入修复代码步骤；先补分析字段或 runner 输出。

## 5.2 步骤 M1.3-2：回归门禁与评测 runner 加固

### 新会话标题

```text
类案检索助手 M1.3-2 回归门禁与评测 runner 加固
```

### 目标

把 M1.3 的硬门禁写进评测与 release gate：aggregate 指标变好但存在明确 query 退化时，不允许通过灰度候选。

### 开始前必须读取

- M1.3-1 产物
- `apps/api/scripts/m1_2_regression.py`
- `apps/api/app/eval/product_eval.py`
- `apps/api/app/eval/result_format.py`
- `apps/api/tests/test_m1_2_regression.py`
- `apps/api/tests/test_eval_pipeline.py`

### 主要任务

1. 增加或复用 M1.3 regression runner 输出：
   - `beforeVsAfterRegressedCount`
   - `afterVsBaselineRegressedCount`
   - `top10MissCount`
   - `metricRegressionCount`
   - `recallMissCount`
   - `grayCandidateHardGatePassed`
2. 明确灰度候选硬门禁：
   - `Top10 hit rate >= 0.60`
   - before -> after `REGRESSED=0`
   - after vs baseline `REGRESSED=0`
   - `METRIC_REGRESSION=0`
3. 保证 runner 不写原始 query 或候选正文。
4. 增加 focused tests 覆盖“aggregate 变好但 per-query 退化仍失败”的情况。

### 产物

- `docs/development/m1.3-regression-gate-*.md`
- `docs/development/m1.3-regression-gate-*.json`
- 如需改代码，只限评测 runner、gate 逻辑和测试。

### 验收

- 评测脚本能稳定输出 M1.3 硬门禁结果。
- 有测试证明存在任何明确退化时，`weightedRerankGrayCandidate=false`。
- 不改变搜索线上逻辑、召回策略、rerank 权重或 feature flag 默认状态。

### 止损

如果 gate 加固导致旧产物不可比，必须同时保留 M1.2 格式读取兼容，不得通过破坏历史对比来“清零退化”。

## 5.3 步骤 M1.3-3：剩余 RECALL_MISS 定点修复

### 新会话标题

```text
类案检索助手 M1.3-3 剩余 RECALL_MISS 定点修复
```

### 目标

只处理 M1.3-1 清单中的剩余召回问题，避免大范围扩大候选池导致排序噪声。

### 开始前必须读取

- M1.3-1、M1.3-2 产物
- `docs/development/m1.2-recall-repair-20260609.md`
- `docs/development/m1.2-recall-repair-20260609.json`
- `apps/api/app/retrieval/service.py`
- `apps/api/app/retrieval/merge.py`
- `apps/api/app/retrieval/bm25_fallback.py`
- `apps/api/app/query_processing/term_mapping.py`
- `apps/api/app/eval/product_eval.py`

### 主要任务

1. 对每个 `RECALL_MISS` query 复盘：
   - relevant case 是否在原 query vector topK
   - 是否在 mapped/rewrite vector topK
   - 是否在 cleaned BM25 topK
   - 是否在 expanded BM25 topK
   - 是否被 merge、dedupe、TopK 或候选 gating 丢失
2. 只允许定点修复：
   - 对特定失效模式补充候选池 admission rule
   - 对 BM25 命中但未合并的样本做受控 supplement
   - 对 query mapping 无收益样本补充 recall-only 映射
3. 禁止默认扩大所有 query 的候选池。
4. 输出修复前后每个 query 的 candidate presence 和 Top10 状态。

### 产物

- `docs/development/m1.3-recall-repair-*.md`
- `docs/development/m1.3-recall-repair-*.json`
- 必要时包含召回层小范围代码修复和测试。

### 验收

- `RECALL_MISS` 数量下降。
- Top10 hit rate 有实质提升，目标向 `0.60` 靠近。
- fixed regression 的 typical success 样本不新增退化。
- 性能 smoke 不明显恶化，warm P95 仍 `< 3s`。

### 止损

如果召回修复提升少数 bad case 但新增任何 fixed regression，则撤回该策略或转为受控实验，不进入默认路径。

### 新会话标题


### 目标

修复固定回归中的排序退化，尤其是 M1.2 guarded fusion 对部分成功样本或边界样本的过度保护问题。

### 开始前必须读取

- M1.3-1 到 M1.3-3 产物
- `docs/development/m1.2-rerank-fusion-repair-20260609.md`

### 主要任务

1. 对 `RERANK_SUPPRESSION`、`GUARD_OVER_CORRECTION`、`SUCCESS_SAMPLE_PROTECTION` query 做分数拆解。
2. 只允许小范围 guard 修正：
   - 放宽 no-fact-support guard 的适用条件
   - 增加强 vector / 多源命中保护
   - 限制 weak-signal-only boost，避免回到 M1.2 初始误伤

   - baseline
   - M1.2 current before
   - M1.2 guarded after

- fixed regression before -> after `REGRESSED` 明显下降，目标为 `0`。
- after vs baseline `REGRESSED` 明显下降，目标为 `0`。
- `METRIC_REGRESSION` 明显下降，目标为 `0`。
- 不重新引入 key paragraph / authority weak-signal 过度抬分。

### 止损

如果排序修复让 Top10 hit rate 上升但 fixed regression 仍有明确退化，不允许进入灰度候选，只保留为候选参数。

## 8. M1.3 Definition of Done

M1.3 完成必须同时满足：

- 有冻结后的 M1.3 regression triage。
- 有硬门禁版 regression/gate 输出。
- `RECALL_MISS`、`METRIC_REGRESSION` 和 fixed regression 明确下降或有不可修复解释。
- 至少一个候选参数版本通过全部硬门禁，或明确全部候选 `NO_GO`。
- 新 weighted rerank 是否允许成为灰度候选有正式门禁。
- 是否进入 M2 readiness 有正式 Go/No-Go。
- 默认 feature flags 没有被悄悄打开。
- 没有把 M2、M3、M4 功能提前混进来。
- 没有把原始案情文本写进日志、报告正文或新增 JSON 明文字段。

## 9. M2 入口条件

只有 M1.3-7 同时给出以下结论时，下一轮才允许进入 M2 可信检索准备：

| 条件 | 必须满足 |
| --- | --- |
| 基础搜索 | `GO` |
| weighted rerank 灰度候选 | `GO`，或明确 M2 不依赖该排序且 baseline 精准度已达标 |
| product-local Top10 hit rate | `>= 0.60` |
| fixed regression | `REGRESSED=0` |
| bad case | `METRIC_REGRESSION=0`，剩余 `RECALL_MISS` 有解释和止损 |
| 性能 | warm P95 `< 3s` |
| 回滚 | `< 60s`，无需重建索引 |
| 隐私 | 无原始案情/候选正文/chunk 正文写入 |
| 默认开关 | `ENABLE_WEIGHTED_RERANK=false`，除非另有灰度开关策略明确批准 |

如果任何一项不满足，下一轮继续做 M1.3.x 回归修复，不进入 M2 实施。
