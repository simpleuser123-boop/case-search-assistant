## 2. 开发目标与范围

### 2.1 MVP 目标

3 天内跑通以下主链路：

```text
案情输入 -> 查询清洗 -> 查询改写 -> 向量召回 -> 合并去重 -> 事实相似度排序 -> 摘要高亮 -> 结果展示 -> 详情查看 -> 埋点评测
```

上线内测前必须满足：

| 指标 | 门槛 |
| --- | --- |
| Top10 主观命中率 | >= 60% |
| Precision@5 | 不低于基线，目标提升 >= 10% |
| P95 检索链路 | < 3s |
| 搜索完成率 | > 95% |
| 无结果降级 | 100% 覆盖 |
| 回滚能力 | 排序规则可回到基线向量排序 |

### 2.2 MVP 必做范围

| 编号 | 功能 | 交付物 |
| --- | --- | --- |
| F01 | 自然语言搜索框 | 首页输入框、字数统计、示例案情、草稿恢复 |
| F02 | 查询清洗与校验 | 输入校验、`query_session_id`、`input_hash`、脱敏日志 |
| F03 | 查询改写 | DeepSeek 改写、法律要素、检索变体、规则兜底 |
| F04 | 向量召回 | 本地 bge-m3（Ollama）、Chroma 检索、BM25 兜底 |
| F05 | 合并去重 | 多路召回合并、按 `case_id` 去重、召回来源记录 |
| F06 | 事实相似度排序 | 向量分、法律要素分、案由分、段落分、权重配置 |
| F07 | 搜索结果列表 | 结果卡片、相似度、摘要、高亮、法院/审级/日期 |
| F08 | 关键摘要与高亮 | 原文片段摘要、来源 chunk、摘要失败降级 |
| F09 | 案例详情抽屉 | 元数据、摘要、裁判要旨、来源、原文链接 |
| F10 | 无结果与错误降级 | 空状态、网络错误、LLM 降级、向量库降级 |
| F11 | 基础埋点 | 搜索、渲染、点击、详情、重搜、无结果、退出 |
| F12 | 评测集与离线评测 | 评测 query、人工标注、Precision@5、NDCG@10、报告 |

### 2.3 暂不做范围

- 完整类案报告生成。
- 团队空间、权限、协作。
- 法官/法院倾向分析。
- 大规模生产级向量基础设施。
- 承诺「已查全」「保证无遗漏」的绝对话术。

## 4. Day 0：环境、数据与门禁准备

目标：让开发从「可以真实检索和评测」开始，而不是只完成 UI 或接口空壳。

### 4.1 锁定配置和密钥

建立 `.env.example`，至少包含：

```env
DEEPSEEK_API_KEY=
EMBEDDING_PROVIDER=ollama
EMBEDDING_MODEL=bge-m3
OLLAMA_BASE_URL=http://localhost:11434
CHROMA_COLLECTION=case_chunks_bge_m3_v1
CHROMA_PERSIST_DIR=./data/chroma
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/case_search
REDIS_URL=redis://localhost:6379/0
LOG_LEVEL=INFO
```

要求：

- 启动时校验 DeepSeek 密钥是否存在（不打印值），并校验本地 Ollama 服务（`OLLAMA_BASE_URL`）可达。
- query embedding 和文书 embedding 必须使用同一 provider、模型名、维度和距离度量。
- 首次 embedding 调用后固化向量维度（bge-m3 应为 1024），并写入 collection 元数据。
- Windows 本机 Chroma 持久化目录使用 ASCII 路径，避免中文项目路径下 HNSW 文件缺失导致 collection 不可查询。

### 4.2 初始化基础服务

任务：

1. 初始化 `apps/web`：React、TypeScript、Vite、Tailwind CSS、React Router、TanStack Query、Vitest。
2. 初始化 `apps/api`：FastAPI、Pydantic、SQLAlchemy 或 SQLModel、Alembic、pytest。
3. 新增 Docker Compose：PostgreSQL、Redis，Chroma 先使用本地持久化目录或轻量服务模式。
4. 后端提供 `GET /health`。
5. 前端提供空白首页和基础路由。

验收：

- 前端本地能启动并打开首页。
- 后端 `/health` 返回正常。
- PostgreSQL 可连接。
- Chroma 持久化目录可读写。
- 测试命令能执行空测试或示例测试。

### 4.3 准备裁判文书样本

任务：

1. 准备 100-500 份公开裁判文书样本。
2. 清洗标题、案号、法院、审级、案由、日期、来源 URL。
3. 识别 `本院查明`、`本院认为`、`裁判结果` 等段落。
4. 将文书切分为 600-1000 中文字符 chunk，相邻 chunk 保留 80-120 字重叠。
5. 每个 chunk 绑定 `case_id`、`chunk_id`、`chunk_type`、`start_offset`、`end_offset`。

验收：

| 检查项 | 门槛 |
| --- | --- |
| 案号缺失率 | < 2% |
| 法院缺失率 | < 2% |
| 裁判日期缺失率 | < 5% |
| chunk 空文本率 | < 1% |
| 重复文书率 | < 3% |
| `本院查明` 识别成功率 | MVP >= 60% |

### 4.4 建立向量索引

任务：

1. 使用本地 bge-m3（Ollama）为 chunk 生成 embedding。
2. 写入 Chroma collection：`case_chunks_bge_m3_v1`。
3. metadata 至少包含 `case_id`、`chunk_id`、`chunk_type`、`case_cause`、`court_level`、`trial_level`、`judgment_year`、`region`。
4. 记录 `EmbeddingModelVersion`：provider、model_name、vector_dimension、normalize、distance_metric、collection_name。

验收：

- embedding 成功率 >= 99%。
- collection 元数据能校验模型版本和向量维度。
- 5 条手工 query 可以从 Chroma 返回 chunk 候选。
- 不同 embedding 模型不得写入同一个 collection。

### 4.5 准备评测集和兜底映射

任务：

1. 准备至少 20 条典型 query。
2. 至少 10 条 query 有标准相关案例或人工相关性标注。
3. 准备至少 15 组法律术语映射或兜底映射，例如口语词、法律术语、案由提示、置信度。
4. 建立基线检索报告：纯向量排序或简单 BM25。

Go 条件：

- 有可检索数据。
- 有可重复跑的评测 query。
- DeepSeek API 可用；本地 Ollama bge-m3 服务可达。
- 基线结果可输出 Precision@5 和 NDCG@10。

No-Go 行动：

- 数据不足：只做 Demo，不宣称质量达标。
- LLM 不可用：关闭查询改写，使用原文检索和规则映射。
- embedding 不可用：临时降级为 BM25/关键词检索，不宣称 RAG 语义召回可用。
- Chroma 未就绪：先用内存 FAISS 或进程内向量索引验证链路。

## 5. Day 1：后端检索链路

目标：后端完成从请求到排序结果的闭环，并具备降级和评测能力。

### 5.1 API 骨架

实现接口：

| 接口 | 用途 |
| --- | --- |
| `POST /api/search` | 标准检索 |
| `POST /api/search/expand` | 扩展检索和低置信度候选 |
| `GET /api/cases/{case_id}` | 案例详情 |
| `POST /api/events` | 脱敏埋点事件 |
| `GET /health` | 健康检查 |

验收：

- OpenAPI 文档可访问。
- 所有接口有统一错误格式。
- 每次搜索生成 `query_session_id`。
- 日志包含阶段耗时，不包含原始案情。

### 5.2 查询清洗与改写

任务：

1. 清洗多余空白，统一中英文标点。
2. 拒绝空白、纯标点、过短无效 query。
3. 生成 `input_hash`，不持久化原始 query。
4. 调用 DeepSeek 输出法律要素、2-3 条 query variants、案由提示。
5. 对 LLM 返回做 JSON schema 校验。
6. 增加 5 秒超时和格式异常降级。

验收：

- 10 条典型 query 中至少 8 条能输出法律要素。
- LLM 超时、JSON 解析失败时仍能进入原文检索。
- 改写结果不得删除原始关键事实。
- 改写耗时和降级原因写入脱敏日志。

### 5.3 多路召回

任务：

1. 对原 query 和改写 query variants 生成 embedding。
2. 使用 Chroma 召回：原文 Top 50，改写变体每条 Top 30。
3. 支持案由、年份、段落类型软过滤。
4. Chroma 异常时触发 BM25 兜底。
5. 候选按 `case_id` 去重，保留最高 chunk 分、命中 chunk、召回来源。

验收：

- 检索 P95 < 1s。
- 每个候选包含 `case_id`、`chunk_id`、`vector_score`、`retrieval_source`。
- 同一案号不重复展示。
- 候选不足时触发宽松召回。

### 5.4 事实相似度排序

MVP 排序公式：

```text
final_score =
  0.55 * vector_similarity
+ 0.20 * legal_element_overlap
+ 0.10 * case_cause_match
+ 0.10 * key_paragraph_match
+ 0.05 * authority_signal
```

任务：

1. 实现向量相似分。
2. 实现法律要素重合分。
3. 实现案由软加权。
4. 实现 `本院查明`、`本院认为` 等关键段落加权。
5. 法院层级、审级、日期只作为弱信号。
6. 排序权重放入配置文件，支持回滚到纯向量排序。

验收：

- Precision@5 不低于基线。
- NDCG@10 有正向提升或至少不退化。
- 每条结果返回分数组成，便于 bad case 分析。
- 案由加权不能把事实明显不相关案例推入 Top3。

### 5.5 摘要、高亮和来源锚点

任务：

1. 优先从 matched chunk 抽取 2-3 句事实摘要。
2. 高亮与输入案情相似的短语或片段。
3. 每个摘要和高亮绑定 `source_chunk_id`。
4. LLM 摘要作为可选增强，失败时回退原文片段。

验收：

- 摘要失败不影响搜索响应。
- 无来源锚点的 AI 摘要不展示。
- 结果卡片能拿到摘要、高亮和来源 chunk。

### 5.6 后端测试

至少覆盖：

- 输入校验。
- 查询改写 schema。
- LLM 超时降级。
- embedding 模型版本校验。
- Chroma 召回。
- BM25 兜底。
- 合并去重。
- 排序权重。
- 摘要来源锚点。
- 隐私日志不含原始 query。

## 6. Day 2：前端页面与联调

目标：用户能从首页完成一次搜索，查看结果，打开详情，并在异常状态下得到清晰反馈。

### 6.1 首页搜索

任务：

1. 实现 `SearchComposer`。
2. 支持多行案情输入、字数统计、空输入禁用、超过 500 字弱提示。
3. 支持示例案情点击填入。
4. 支持 Ctrl+Enter 或按钮提交。
5. 使用 localStorage 保存未提交草稿。

验收：

- 输入 150-300 字案情后可提交。
- 刷新页面后恢复未提交草稿。
- 移动端输入字号 >= 16px，避免系统缩放。
- 前端不向埋点上传原始案情。

### 6.2 搜索结果页

任务：

1. 实现顶部紧凑搜索框。
2. 实现 `ResultOverview`：结果数、耗时、数据覆盖、是否降级。
3. 实现 `ResultList` 和 `ResultCard`。
4. 卡片展示标题、案号、法院、审级、案由、日期、相似度、摘要、高亮。
5. 实现骨架屏、加载中、成功、结果少、无结果、网络异常、降级提示。

验收：

- mock 数据和真实 API 都能渲染。
- 375px、768px、1440px 布局正常。
- 加载、hover、错误状态不造成明显布局跳动。
- 相似度不能只靠颜色表达，必须有文本。

### 6.3 案例详情抽屉

任务：

1. 点击结果卡片打开 `CaseDetailDrawer`。
2. 调用 `GET /api/cases/{case_id}`。
3. 展示案号、法院、审级、案由、日期、完整摘要、裁判要旨、来源片段、原文链接。
4. 桌面端右侧抽屉，移动端全屏抽屉。
5. 原文链接不可达时展示说明。

验收：

- 打开、关闭、重试可用。
- 抽屉打开后焦点进入抽屉，关闭后回到触发卡片。
- 无来源的生成内容不展示。

### 6.4 扩展检索与低置信度候选

MVP 可做轻量版：

1. 主结果少于 5 条时展示「可能相关候选」入口。
2. 用户点击后调用 `POST /api/search/expand`。
3. 低置信度候选单独分组展示。
4. 文案使用「部分相关，仅供复核」，不得使用「已查全」。

验收：

- 主结果充足时默认不打扰。
- 主结果少或无结果时有下一步动作。
- 扩展检索状态可埋点。

### 6.5 前端埋点

事件：

| 事件 | 触发 |
| --- | --- |
| `search_submit` | 用户提交搜索 |
| `search_result_render` | 结果渲染完成 |
| `result_card_click` | 点击结果卡片 |
| `case_detail_view` | 打开详情 |
| `search_refine` | 修改 query 后重搜 |
| `search_zero_result` | 无结果 |
| `extended_search_trigger` | 点击扩展检索 |
| `page_exit` | 用户离开页面 |

验收：

- 开发环境能看到事件。
- 事件包含 `query_session_id`。
- 事件不包含原始案情文本。

### 6.6 前端测试

至少覆盖：

- 输入校验和字数统计。
- 搜索框提交状态。
- 结果卡片渲染。
- 骨架屏。
- 空状态。
- 网络错误重试。
- 详情抽屉。
- 移动端布局。
- 埋点参数脱敏。

## 7. Day 3：评测、集成、灰度和回滚

目标：用评测和回滚决策控制上线风险，避免「感觉更好」但实际退化的排序上线。

### 7.1 端到端联调

联调路径：

1. 首页输入案情。
2. 提交搜索。
3. 后端完成查询改写、召回、排序、摘要。
4. 前端展示结果列表。
5. 点击第一条结果。
6. 打开详情。
7. 修改 query 重搜。
8. 触发扩展检索。
9. 上报埋点。

必须覆盖异常：

- LLM 超时。
- 向量库异常。
- 无结果。
- 摘要失败。
- 详情原文不可达。
- 网络中断后重试。

验收：

- 所有异常都有用户可理解反馈。
- 不丢失用户输入。
- 服务端日志可定位失败阶段。

### 7.2 离线评测

任务：

1. 跑基线排序。
2. 跑新排序。
3. 计算 Precision@5、NDCG@10、Top10 主观命中率。
4. 输出 bad case 清单。
5. 标记每个 bad case 的原因：召回缺失、改写错误、案由误加权、段落匹配错误、摘要误导。

上线门槛：

| 结果 | 决策 |
| --- | --- |
| Precision@5 不下降，NDCG@10 正向，Top10 主观命中率接近或达到 60% | 可以灰度启用新排序 |
| 指标模糊或样本不足 | 只上线日志和评测管道，不启用新排序 |
| Precision@5 下降或出现高风险错排 | 不上线新排序，回到基线 |

### 7.3 性能与稳定性检查

检查指标：

- `rewrite_duration_ms`
- `embedding_duration_ms`
- `retrieval_duration_ms`
- `rerank_duration_ms`
- `summary_duration_ms`
- `total_duration_ms`
- LLM 超时率
- 向量库错误率
- API P95/P99
- 降级触发次数

验收：

- P95 检索链路 < 3s，若摘要生成导致超时，摘要改为异步或关闭。
- API 错误率在内测可接受范围内。
- 降级检索不会造成前端白屏。

### 7.4 灰度发布

任务：

1. 配置 feature flags：查询改写、加权排序、摘要生成、扩展检索。
2. 小范围启用给内部或 5 名目标律师。
3. 监控错误率、耗时、点击率、二次搜索率、无结果率。
4. 收集每次搜索的脱敏会话和结果点击。

灰度策略：

- 评测正向：启用新排序。
- 评测模糊：仅启用日志和评测，不启用新排序。
- 评测负向：保留代码但 feature flag 关闭。

### 7.5 回滚演练

回滚开关：

| 开关 | 回滚后行为 |
| --- | --- |
| `ENABLE_QUERY_REWRITE=false` | 原 query 直接检索 |
| `ENABLE_WEIGHTED_RERANK=false` | 回到纯向量排序 |
| `ENABLE_SUMMARY=false` | 展示原文片段 |
| `ENABLE_EXPANDED_SEARCH=false` | 隐藏扩展检索入口 |

验收：

- 1 分钟内恢复基础搜索。
- 回滚后前端仍可正常展示。
- 回滚事件写入日志。
- 回滚不需要重建向量索引。

## 10. 测试清单

### 10.1 后端

- 输入为空、纯标点、超长文本。
- DeepSeek 超时、返回非 JSON、字段缺失。
- 本地 bge-m3（Ollama）embedding 调用失败或服务不可达。
- query embedding 与文书 embedding 模型版本不一致。
- Chroma 正常召回和异常降级。
- BM25 兜底。
- 多路候选按 `case_id` 去重。
- 排序权重配置化和回滚。
- 摘要来源锚点。
- 日志脱敏。

### 10.2 前端

- 搜索框默认、聚焦、输入、超长、提交、禁用状态。
- 示例案情填入。
- localStorage 草稿恢复。
- 结果列表成功渲染。
- 骨架屏、空状态、错误 Banner。
- 降级检索提示。
- 详情抽屉桌面和移动端。
- 扩展检索入口。

### 10.3 E2E

1. 打开首页。
2. 输入或填入示例案情。
3. 提交搜索。
4. 看到结果概览和结果列表。
5. 点击结果卡片。
6. 查看详情和来源。
7. 关闭详情。
8. 修改 query 重搜。

10. 验证埋点和后端日志。

## 11. MVP Definition of Done

MVP 完成必须同时满足：

- 用户能从首页完成一次搜索并看到结果。
- 结果列表包含相似度、摘要、高亮、法院、审级、日期。
- 可以打开案例详情并看到来源片段或原文链接。
- 无结果、网络异常、LLM 异常、向量库异常、摘要异常都有降级。
- 搜索链路耗时和关键行为有脱敏埋点。
- 评测集可以重复运行。
- 排序规则和摘要生成可以通过配置回滚。
- 用户原始案情不落库、不进埋点。
- 生成内容必须绑定 `case_id` 和 `source_chunk_id`，否则不展示。
