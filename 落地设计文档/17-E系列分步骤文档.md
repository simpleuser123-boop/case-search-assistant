# E-1 生态入口合同

## 1. E-1 定位

E-1 对应文档 16 §6 执行计划表的第一步，是整个 E 系列的「合同冻结步」，对标 M1-1 / M5-1 入口合同：**只冻结契约与开关，零业务实现、零行为变化**。它存在的意义是：在动任何代码（E-2 抽内核）之前，先把「共享平台怎么分层、跨产品传什么、5 个产品各自的开关叫什么、默认什么状态、守哪些生态级红线」用文档 + 机器可校验的产物固定下来，让后续 E-2~E-9 每一步都有可比对的冻结基线。

E-1 不新建任何产品能力包（intake/statute/drafting/casebook 都不建），不抽取共享内核（那是 E-2），不暴露检索服务接口（那是 E-3）。E-1 结束时，代码行为与 M5-10 末态**逐位一致**。

### 1.1 E-1 目标

1. **冻结共享平台分层契约**：明确「前端层 / 产品能力包层 / 共享内核层」三层边界与依赖方向（能力包只依赖内核、不互相 import）。
2. **冻结 4 个跨产品契约对象的字段白名单**：SearchProfile / CandidateRef / DraftDescriptor / CaseFolder，逐字段标注类型、是否必填、是否锚点、sanitize 时主动丢弃的非白名单键。
3. **冻结 5 个产品 feature flag**：声明开关位、默认全 `false`、关闭语义（前端入口不渲染 + 后端安全降级），写入 `config.py` 与 `.env.example`（仅声明，不接线任何业务）。
4. **冻结生态级隐私红线**：把文档 16 §5 的三条跨产品红线（录入端原始案情零上送 / 跨产品只走契约对象 / 回流聚合无正文）落成可校验断言口径。
5. **产出机器可校验的 release gate + parameter version**，供 E-2~E-9 每步复比对。

### 1.2 E-1 落地基调（沿用 M5-1 同款）

- **先合同，零实现**：只新增 flag 声明 + 契约文档 + 产物 + （可选）契约白名单常量与其单元测试；不接线任何端点、不建产品包、不抽内核。
- **默认全关**：5 个产品 flag 默认 `false`；任何 flag 默认 `true` 即 `NO_GO`。
- **行为零变化**：E-1 后复跑 M5-10 同款回归集必须逐位一致（同 passed 数、无新增失败、排序与默认行为不变）。
- **可回退**：E-1 只增不改；移除新增 flag 声明即回到 M5-10 末态。

## 2. 共享平台分层契约（E-1 冻结口径）

```
┌─────────────────────────────────────────────────────────────┐
│ 前端层 apps/web（多入口路由 / 共享组件库）                     │
│  录入端 │ 检索助手 │ 法条检索 │ 文书工作台 │ 案件协作台        │
├─────────────────────────────────────────────────────────────┤
│ 产品能力包层 apps/api/app/<product>/（互不直接 import）        │
│  intake/ │ (检索沿用现有) │ statute/ │ drafting/ │ casebook/  │
├─────────────────────────────────────────────────────────────┤
│ 共享内核层（所有产品依赖，护栏只实现一次）                      │
│  RAG核心: retrieval/ rerank/ query_processing/ summary/       │
│  身份与租户: account/ team/ permission/ sharing/              │
│  契约与护栏: M4-1 字段白名单 / source_anchors 校验 / 禁用文案守门 │
│  数据: pipeline/ + Chroma 向量库 + case_store/                │
└─────────────────────────────────────────────────────────────┘
```

冻结约束（E-1 只把这些写成文档与产物，不实现）：

1. 产品能力包**只能依赖共享内核**，不能互相 import；跨产品数据流转走 §3 的契约对象，不走代码直连。
2. 护栏（来源锚点校验、字段白名单 sanitize、禁用文案守门、多租户过滤）全部在共享内核实现，能力包调用不重写。
3. 每个产品一个总开关 flag，默认 `false`；关闭某产品 = 前端入口不渲染 + API 端点安全降级，其余产品与基础搜索不受影响。

## 3. 四个契约对象字段白名单（E-1 冻结）

所有契约对象受 M4-1 字段白名单约束；sanitize 时主动丢弃任何非白名单键（含潜在正文）。下表即 E-1 要冻结进文档与产物的权威口径。

### 3.1 SearchProfile（录入端 → 检索助手）

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `case_cause` | str(短) | 否 | 案由（结构化要素） |
| `region` | str(短) | 否 | 地域 |
| `trial_level_preference` | str(短) | 否 | 审级倾向 |
| `dispute_focus_keywords` | list[str(短)] | 否 | 争议焦点关键词 |
| `query_text` | str | 是 | **已脱敏**的查询文本（供检索） |

红线：原始口语化案情**不在内**（仅浏览器本地）；只携带结构化白名单字段 + 已脱敏 query。出现 `raw_case` / `raw_query` / 任何正文型键 = `NO_GO`。

### 3.2 CandidateRef（检索助手 → 法条/文书/协作台）

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `case_id` | str(短) | 是 | 案例标识 |
| `case_number` | str(短) | 否 | 案号 |
| `court` | str(短) | 否 | 法院 |
| `trial_level` | str(短) | 否 | 审级 |
| `case_cause` | str(短) | 否 | 案由 |
| `judgment_date` | str(短) | 否 | 裁判日期 |
| `source_anchors` | list[{case_id, source_chunk_id, anchor_type?}] | 是 | 来源锚点 |

红线：**不含候选正文 / chunk 正文 / 裁判文书全文**；下游要展示正文须自行实时取且不落库。出现 `full_text`/`content`/`chunk_text` 等 = `NO_GO`。

### 3.3 DraftDescriptor（文书工作台 → 协作台）

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `draft_id` | str(短) | 是 | 草稿标识 |
| `structure_skeleton` | list[str(短)] | 是 | 文书结构骨架（段落标题，非正文；单项 ≤ 60 字防塞正文） |
| `candidate_refs` | list[CandidateRef] | 是 | 引用的检索结果（缺锚点项 fail-closed 丢弃） |
| `statute_refs` | list[StatuteRef] | 否 | 引用的法条（经 E5 互跳；缺锚点项 fail-closed 丢弃）── **E6-1 合同变更追加** |
| `note` | str(短) | 否 | 用户自填短备注 |
| `tag` | str(短) | 否 | 用户自填短标签 |

> **合同变更登记（E6-1 追加可选 statute_refs，2026-06-18）**：E-1 §3.3 冻结的核心五字段（`draft_id` / `structure_skeleton` / `candidate_refs` / `note` / `tag`）**保持不变**，仍由 `whitelist.DRAFT_DESCRIPTOR_FIELDS` 与 `test_e1_contracts` 守门（逐位一致）。E6 文书工作台需要引用经 E5 互跳而来的法条 `StatuteRef`，故经文档 16 §4 与本节登记，为 `DraftDescriptor` 追加**可选** `statute_refs` 字段（沿 StatuteRef §3.5 登记范式，**非擅自加字段**）。E6 权威白名单冻结为独立常量 `DRAFT_DESCRIPTOR_E6_FIELDS`（= 核心五字段 + `statute_refs` + 持久层元数据 `created_at`/`updated_at`/`owner_user_id`/`team_id`/`visibility`，同 CaseFolder 由后端补，默认 `visibility=private`），落地于 `app/kernel/guardrails/contracts/drafting_contract.py`（纯数据 + 纯函数，E6-1 不接线）。该常量**不进** `CONTRACT_FIELD_WHITELIST`（E-1 四对象口径不动，与 `STATUTE_REF_FIELDS` 独立冻结同理）。

红线：**只存引用与骨架，不存起草正文**；`structure_skeleton` 是段落标题清单（每项做长度上限校验，超限即拒绝并记 reason code），不是段落正文；`candidate_refs`/`statute_refs` 引用 100% 有锚点（缺锚点 fail-closed 丢弃，不进交付物）；导出走 M4-5 免责头机制。出现起草正文型 / 裁判正文型 / PII 型 / 胜负结论型字段 = `NO_GO`。

### 3.4 CaseFolder（协作台内归集）

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `case_folder_id` | str(短) | 是 | 案件聚合标识 |
| `owner_user_id` | str(短) | 是 | 归属用户 |
| `team_id` | str(短)\|null | 否 | 归属团队（空=单用户私有） |
| `visibility` | enum(private/team) | 是 | **默认 private** |
| `search_profile_summary` | dict(脱敏白名单子集) | 否 | SearchProfile 摘要（脱敏） |
| `candidate_refs` | list[CandidateRef] | 否 | 关联检索结果清单 |
| `draft_descriptors` | list[DraftDescriptor] | 否 | 关联文书草稿描述 |
| `created_at` / `updated_at` | datetime | 是 | 时间戳 |

红线：默认 `visibility=private`；对象级鉴权；持久层只存元数据/引用/短字段。默认非私有 = `NO_GO`。

### 3.5 StatuteRef（法条检索 → 结果消费方 / 法条↔类案互跳）── E5-1 新增登记

> **合同变更登记（2026-06-17）**：原 §3.1~§3.4 四契约对象不覆盖「法条检索结果」，E5 法条检索端 `statute/` 落地需要新契约对象。经文档 16 §4.1.1 与本节登记，新增 `StatuteRef` 为第 5 个跨产品契约对象——**走合同变更，非擅自加字段**。同步落地为 `app/kernel/guardrails/contracts/statute_contract.py`（纯数据 + 纯函数，E5-1 不接线）。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `statute_id` | str(短) | 是 | 法条标识 |
| `law_name` | str(短) | 是 | 法名 |
| `article_no` | str(短) | 否 | 条号 |
| `statute_anchors` | list[{text_id, law_name?, article_no?, anchor_type?}] | 是 | **必填非空**，每条至少 `text_id`（指向法条语料） |
| `article_text` | str | 否 | 条文文本：**只来自法条语料、带 text_id 锚点**，不得由模型生成 |
| `source_corpus` | str(短) | 否 | 法条语料来源（如 `judge_law_corpus`） |
| `effective_status` | str(短) | 否 | 时效状态（如 `current`），结构化非正文 |
| `related_case_refs` | list[CandidateRef] | 否 | 法条→类案互跳（CandidateRef 同款，**零裁判正文**） |

红线：

- **条文必锚定语料、不得由模型生成**：`article_text`（若有）只来自法条语料库且带 `text_id` 锚点；模型不杜撰/改写/续写；命中无锚点降级不展示（契约层以「`statute_anchors` 必填非空、每条至少 `text_id`」表达「无锚点不展示」）。出现 `generated_article`/`llm_text`/`paraphrased_article` 等模型生成条文型键 = `NO_GO`。
- **互跳只走契约对象**：`related_case_refs` 只承载 `CandidateRef` 白名单七字段、必有 `source_anchors`、**不含裁判正文**；`CandidateRef` 字段不因互跳被增删。
- 拒绝裁判正文型键（`full_text`/`content`/`chunk_text`/`summary_text`/`highlight_text`/`matched_text`）与 PII 型键；`extra=forbid`。出现即 `NO_GO`。
- flag：受 `ENABLE_STATUTE_SEARCH`（E-1 已冻结，默认 `false`）门控，E5 不新增产品级 flag；关闭时法条端点 403 降级、前端无入口。

## 4. 五个产品 feature flag（E-1 冻结，默认全 false）

E-1 在 `config.py` 与 `.env.example` 新增以下开关声明（仅声明，不接线业务）。命名沿用 `ENABLE_<PRODUCT>` 体例，前端镜像 `VITE_ENABLE_<PRODUCT>`：

| flag | 产品 | 默认 | 关闭语义 |
| --- | --- | --- | --- |
| `ENABLE_ECOSYSTEM` | 生态总开关（跨产品入口/导航） | `false` | 关闭即无任何跨产品入口，回 M5-10 单产品末态 |
| `ENABLE_INTAKE` | 案情智能录入端 | `false` | 前端无录入入口；无 SearchProfile 产出端点 |
| `ENABLE_STATUTE_SEARCH` | 法条法规检索 | `false` | 前端无法条入口；法条端点 403 降级 |
| `ENABLE_DRAFTING` | 文书辅助工作台 | `false` | 前端无文书入口；草稿端点 403 降级 |
| `ENABLE_CASEBOOK` | 案件协作工作台 | `false` | 前端无案件入口；CaseFolder 端点 403 降级 |

> 注：检索助手核心无新增 flag（沿用 M1~M5 既有开关）。`ENABLE_ECOSYSTEM` 作为生态总开关，语义上是「是否暴露跨产品导航/枢纽」，与 4 个产品子开关正交：子开关控制单个产品是否可用，总开关控制是否出现「多产品」这件事本身。E-1 阶段七者皆 `false`，全部不接线。

约束：

- 7 个开关（含前端镜像）默认全 `false`；`.env.example` 不得把任一默认改为 `true`。
- E-1 不为任何 flag 实现 on 路径（无端点、无前端入口），仅声明 + 默认 off + （可选）flag 读取的单元测试。
- 不得新增任何会改变主排序 / source selection / rerank 默认开关的逻辑。

# E-2 共享内核抽取

## 1. E-2 定位

E-2 对应文档 16 §6 执行计划表的第二步，是 E 系列「抽地基」的第一块。E-1 只冻结了契约与开关、零运行时改动；E-2 第一次真正动后端结构：把已散落在 `apps/api/app/` 下的 RAG 核心、身份与租户、契约与护栏、数据层，**显式收敛成「共享内核层」**，并把检索助手（现有搜索链路）改写为「内核消费方」，让护栏只在内核实现一次。E-2 的存在意义是：在 E-4~E-7 各产品包落地之前，先把内核边界焊死，否则后面每个产品都会各自直连 `retrieval/`、`rerank/` 等模块，护栏必然漂移。

E-2 的第一性约束是**行为零变化**：抽内核是纯结构重整，不改任何运行时语义。E-2 结束时，检索助手对外行为（排序、source selection、rerank 默认、护栏判定、多租户隔离、性能）与 **E-1 末态逐位一致**，比对基线为 `docs/development/e1-release-gate-20260615-075029.json`。

按既定决策，E-2 采用「**先逻辑后物理**」两子步落地，把高风险的物理文件迁移与低风险的边界冻结分开，各自独立 Go/No-Go：

| 子步 | 内容 | 物理移动 | 回归口径 |
| --- | --- | --- | --- |
| **E-2a** | 逻辑内核边界冻结：定义内核公开面（kernel surface）、声明哪些模块属内核、加 import 边界守门测试（能力包禁止互相 import、禁止绕过公开面深引内核内部）。**零文件移动**。 | 无 | 与 E-1 末态逐位一致 |
| **E-2b** | 物理迁移：把内核模块真正移入 `app/kernel/` 子目录，旧路径留 re-export shim 兼容；检索链路改为从 kernel 公开面消费。 | 有（带 shim） | 与 E-2a 末态逐位一致 |

> 拆分理由：物理迁移会触及大量 import 语句，回归面大、与「行为零变化」张力高；E-2a 先用最低风险方式把边界与消费纪律立起来并通过守门测试，E-2b 再在已验证的边界内做物理搬迁，旧路径 shim 保证任何遗漏的引用仍可用。任一子步 NO_GO 都不污染另一子步的基线。

### 1.1 E-2 目标

1. **冻结共享内核公开面（kernel surface）**：明确内核对外暴露的稳定入口（`app/kernel/__init__.py` 或等价聚合），其余内核内部模块视为实现细节，能力包与检索链路只通过公开面消费。
2. **声明内核成员清单**：依据文档 16 §3.1，把现有模块归为四类内核成员——RAG 核心（`retrieval/ rerank/ query_processing/ summary/`）、身份与租户（`account/ team/ permission/ sharing/`）、契约与护栏（`contracts/` + 来源锚点校验 + 禁用文案守门 + 多租户过滤）、数据（`pipeline/ case_store/`）。
3. **建立 import 边界守门**：用 focused 测试断言「未来的产品能力包不得互相 import」「非内核代码不得绕过公开面深引内核内部子模块」「内核不得反向 import 任何产品包」，把文档 16 §3 的依赖方向变成可执行断言。
4. **检索助手改为内核消费方**：现有搜索端点/服务对内核的引用改走公开面（E-2a 逻辑层先收敛引用入口，E-2b 物理迁移后切到 `app/kernel`），行为零变化。
5. **产出 release gate + parameter version**，作为 E-3~E-9 的新比对基线（E-2b 末态）。

### 1.2 E-2 落地基调

- **结构重整，零语义变化**：只动「模块归属与引用路径」，不动任何排序/检索/rerank/护栏/租户判定逻辑。
- **公开面唯一**：护栏（锚点校验、白名单 sanitize、禁用文案守门、多租户过滤）只在内核实现一次，消费方不得重写或绕过。
- **行为零变化回归**：E-2a 复跑 E-1 同款回归集须逐位一致；E-2b 复跑 E-2a 同款回归集须逐位一致。
- **可回退**：E-2a 移除新增的 surface 聚合与守门测试即回 E-1 末态；E-2b 因旧路径留 shim，回退 = 删除 `app/kernel/` 物理目录、引用切回原路径。

## 2. 共享内核成员与公开面（E-2 冻结口径）

### 2.1 内核成员清单（依据现有真实目录 + 文档 16 §3.1）

下表是 E-2 要冻结的内核成员归属。左列为现有 `apps/api/app/` 下真实包，**E-2a 不移动它们**，只声明归属与公开面；E-2b 才物理迁移进 `app/kernel/`。

| 内核分组 | 现有模块（`app/` 下真实路径） | 对外公开面（消费方允许调用的入口） |
| --- | --- | --- |
| RAG 核心 | `retrieval/`、`rerank/`、`query_processing/`、`summary/` | 检索服务入口、rerank 服务、查询改写、摘要服务（各包 `service.py` 暴露面） |
| 身份与租户 | `account/`、`team/`、`permission/`、`sharing/` | 账号/会话、团队隔离、对象级鉴权、共享，及 `team/isolation.py`、`permission/access.py` |
| 契约与护栏 | `contracts/`（E-1 已建）+ 来源锚点校验 + 禁用文案守门 + 多租户过滤 | `contracts.whitelist` 的 sanitize/白名单常量；锚点校验、禁用文案、租户过滤的统一调用面 |

> 说明：`billing/`、`bulk_import/`、`tendency_analysis/`、`tendency_gate/`、`eval/` 不属于本次内核公开面冻结对象——`eval/` 是离线评测（不进运行时），其余为 M5 商业化/沉淀能力，E-2 不动其归属，留待后续步骤判定。E-2 只冻结上表四组「所有产品都要共用」的内核。

### 2.2 公开面契约（E-2 冻结）

1. **唯一入口**：内核对外只通过公开面（`app/kernel/__init__.py` 聚合 re-export，或 E-2a 阶段等价的 surface 声明）暴露稳定符号；消费方不得 `from app.retrieval.xxx import 内部符号` 式深引内核实现细节。
2. **护栏单点**：来源锚点校验、字段白名单 sanitize、禁用文案守门、多租户过滤只在内核实现一次；消费方调用，不重写、不旁路。
3. **依赖方向单向**：消费方（检索链路、未来产品包）→ 内核公开面 → 内核内部；内核**不得反向** import 任何产品包；未来产品能力包之间**不得互相** import。
4. **行为不变**：公开面只是「把现有可调用入口收敛命名」，不改变任何被调函数的签名语义与运行时行为。

## 5. E-2 与后续步骤的衔接

- E-2 是结构重整步，**不动任何运行时行为**；E-2a 以 E-1 release gate 为基线、E-2b 以 E-2a release gate 为基线，逐子步守「行为逐位一致」。
- E-2 收官后，内核公开面（kernel surface）成为 E-3~E-9 唯一允许的内核消费入口；E-3 暴露的检索内部服务、E-4~E-7 各产品包，都只能经公开面消费内核，不得深引内核内部、不得互相 import。
- import 边界守门测试（E-2a 建、E-2b 续）是 E-4~E-7 每个产品包落地时的常驻护栏：新产品包一旦互相 import 或绕过公开面，守门测试即红。
- 5 个产品 flag 仍全程默认 false、不接线；E-2 不实现任何 flag 的 on 路径。
