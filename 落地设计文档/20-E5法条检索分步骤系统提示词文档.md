## 1. E5 总目标

E5 的目标是落地 E 系列**第二个产品能力包**：法条法规检索端。它在「案源 → 检索」之后接上「检索 → 法律依据」这一段闭环：用户从一段查询（或一条已检索到的类案 `CandidateRef`）出发，命中相关**法条法规条文**，每条命中都带**可核验来源锚点**，并支持**法条 ↔ 类案双向互跳**。

```text
查询文本（已脱敏 SearchProfile.query_text）或 一条 CandidateRef（类案）
  -> 经 app.kernel.rag 内核检索能力（不直连底层）
  -> 命中法条条文（StatuteRef：law_name / article_no / 条文锚点）
  -> 每条命中带 statute_anchors（指向法条语料 text_id，可核验、不杜撰）
  -> 法条 <-> 类案互跳：StatuteRef 关联 CandidateRef[]，CandidateRef 关联 StatuteRef[]
     （互跳只传契约对象：类案侧只回 CandidateRef 无正文，法条侧只回 StatuteRef）
```

E5 的本质是**第二个产品包落地 + 复用既有向量库与法条标注**，不是放开护栏，也不是重建索引或扩语料。E5 不做 drafting/casebook，不改排序/召回/source selection/rerank 默认，不改 `/api/search` 与 E3 `InternalSearchService` 行为，不绕过内核公开面直连检索底层。

### 1.1 法条与类案的关键差异（E5 地基判断）

- **法条条文是公开法律文本**：与裁判文书正文（受 CandidateRef「零正文」红线约束）不同，法条条文本身是公开、可公示的法律内容，**允许在结果中展示**——但有两条硬约束：(a) 展示的条文必须**来自法条语料库、带 `text_id` 锚点**，模型**不得杜撰、改写、续写**任何条文；(b) 命中无锚点则降级不展示（沿用「仅锚定来源才展示 / 无来源降级」红线）。
- **法条 ↔ 类案互跳只走契约对象**：从法条跳类案，只回 `CandidateRef`（无正文，带 source_anchors）；从类案跳法条，只回 `StatuteRef`（带 statute_anchors）。任何一侧都不得借互跳泄露对侧的正文型内容。
- **复用既有标注，不重建案件索引**：既有 chunk 元数据已带 `law_articles`（源自 JuDGE「Law Articles」），E5 用它做「类案 → 法条」的关联种子；法条语料（如 JuDGE `law_corpus.jsonl` 451 条 `text_id/text/name`）+ qrels（案件→法条）作为法条语料与评测标注的现成种子。E5 当前覆盖**刑事法条**；民事/行政法条扩充与 M5-7 路线 B 扩语料是同一依赖，合并一份语料两用，**本期不强求扩充**，覆盖范围在产物中如实标注。

### 1.2 E5 涉及的 flag（沿用 E-1 五产品 flag，不新增产品级 flag）

| flag | 含义 | 默认 | E5 阶段状态 |
| --- | --- | --- | --- |
| `ENABLE_STATUTE_SEARCH`（E-1 已冻结） | 法条法规检索总开关：前端入口 + statute 端点是否可用 | `false` | E5 实现其 on 路径（法条检索 + 互跳），但默认仍 `false` |
| `VITE_ENABLE_STATUTE_SEARCH`（E-1 已冻结） | 前端镜像：法条检索入口/路由是否渲染 | `false` | 同上，默认仍 `false` |

> E5 **不新增产品级 flag**（与 E4 新增 `ENABLE_INTAKE_AI_EXTRACTION` 不同）。法条语料若需独立索引开关，可用内部构建脚本参数或 data-layer 配置控制，不引入新的对外业务 flag；任何新开关默认必须 false。`ENABLE_INTAKE`/`ENABLE_INTAKE_AI_EXTRACTION`/`ENABLE_DRAFTING`/`ENABLE_CASEBOOK`/`ENABLE_ECOSYSTEM`/`ENABLE_WEIGHTED_RERANK` 等默认全程仍 false。任一默认 `true` 即 `NO_GO`。

### 1.3 E5 守的红线（在 M4/M5 + 生态 3 红线之上）

1. **法条条文必须锚定语料、不得杜撰**：任何展示的法条条文都来自法条语料库且带 `text_id` 锚点；模型不生成、不改写、不补全条文；无锚点则降级不展示。
2. **法条 ↔ 类案互跳只走契约对象**：法条侧只产 `StatuteRef`，类案侧只回 `CandidateRef`（无正文）；互跳不泄露对侧正文型内容。
3. **statute 经内核检索能力消费，不直连底层**：statute 产品包只依赖 `app.kernel` 公开面（检索经 `app.kernel.rag`），不深引 retrieval/rerank/summary，不复制检索主路径，不 import 其它产品包。
4. **不改既有检索行为、不重建案件索引**：复用既有向量库 + `law_articles` 标注；不改 `/api/search`、`/api/search/expand`、E3 `InternalSearchService` 行为；不改排序/召回/source selection/rerank 默认；不改 data/processed 既有产物或评测基线。
5. 沿用 M4/M5 全部红线 + 生态 3 红线：仅锚定来源才展示 / 无来源降级 / 不落正文（裁判文书）/ 不按 case_id 特判 / 不输出胜负概率 / flag 默认 false / 关闭回上一态 / 多租户强隔离 / 凭据不代管；原始案情零上送 / 跨产品只走契约对象 / 回流 100% 聚合无正文。

- E5-1 到 E5-6 均 GO。
- E5 全套测试 + E1~E4 同款回归全通过。

- 仅 intake + statute 产品包、include_router=14、无前端默认入口、ENABLE_STATUTE_SEARCH 默认 false、无新增对外业务 flag。
- 法条索引与案件索引隔离；案件产物/评测基线未改；/api/search、E3、intake 行为零变化；前端 build 可验证通过。

CONDITIONAL_GO：

收尾时列出新增/修改文件、完整验证结果、E5 overall、法条覆盖范围、是否允许进入 E6，以及下一步标题：

## 11. E5 止损线

出现以下任一情况，当前子步骤必须 NO_GO 或 BLOCKED，不得继续后续步骤：

- E4 入场未 GO（`allow_enter_e5` 非 true），且当前步骤不是 E5-1。
- 法条条文由模型生成/改写/补全，或展示无 `text_id` 锚点的条文（违反「锚定不杜撰 / 无来源降级」）。
- 裁判正文 / chunk 正文 / 裁判文书全文经 `StatuteRef` 或法条↔类案互跳泄露；或互跳产出的 `CandidateRef` 缺 `source_anchors` 或含正文。
- `StatuteRef` 白名单出现裁判正文型 / PII 型字段；`CandidateRef` 因互跳被擅自加字段；未走合同变更就新增/改契约对象。
- 新建 `drafting`/`casebook` 产品包，或 `statute` 与其它产品包互相 import。
- `statute` 绕过 `app.kernel.rag` 公开面直连 retrieval/rerank/summary，或复制检索主路径，或循环导入。
- 重建 / 修改案件索引、`cases.jsonl`/`chunks.jsonl`/既有评测基线；法条索引与案件索引未物理隔离。
- 新增未鉴权对外端点；`statute` 端点鉴权/租户隔离弱于 `/api/search`。
- `/api/search`、`/api/search/expand`、E3 `InternalSearchService`、intake 端点行为无说明地改变；排序、召回、source selection、rerank 默认被改。
- 任一 E-1/M4/M5/INTAKE flag、`ENABLE_STATUTE_SEARCH` 默认被改为 `true`；或新增任何默认 `true` 的对外业务 flag。
- `qrels`/`label`/`relevance` 进入运行时（仅离线评测可用），或 `query id`/`case id` 特判进入运行时。
- 原始案情/PII 进入任何后端请求体、持久层、日志、报告、JSON、导出、测试快照，或进入浏览器存储/URL/网络请求体。
- 测试或产物写入真实长案情、裁判正文、真实 PII / 密钥；法条 fixture 嵌入真实长篇正文型数据。

## 12. E5 完成后的衔接

- E5 GO 后，E6 才允许开始做文书工作台（drafting/）。E6 复用 `CandidateRef` + `StatuteRef`（类案 + 法条依据）拼装文书骨架，产出 `DraftDescriptor`（只引用骨架不存正文，文档16 §3.3），同样经内核公开面消费、不直连底层。
- `StatuteRef` 契约（E5-1 引入并登记）成为 E6~E7 可用的法律依据引用对象；各产品只能用、不能私自加字段，要改字段须回文档 16/17 走合同变更。
- 「法条条文锚定语料、不杜撰、无锚点不展示」红线（E5 首次落为可执行边界）成为 E6~E8 常驻护栏；任何产品展示法律条文都必须带 `text_id` 锚点、来自语料。
- 法条↔类案互跳模式（只走契约对象、两侧不泄露对侧正文）为 E6 文书引用、E7 协作台归集、E8 回流提供互跳范式。
- 5 个产品 flag 中 `ENABLE_STATUTE_SEARCH` 在 E5 首次实现 on 路径但默认仍 false；`ENABLE_DRAFTING`/`ENABLE_CASEBOOK` 两个 flag 仍全程默认 false、不接线，待 E6~E7。
- 法条语料民事/行政缺口与 M5-7 路线 B 扩语料是同一依赖：后续扩语料时应合并一份语料两用（类案检索 + 法条检索），并各自补离线评测与守门，不影响 E5 已冻结的契约与边界。
