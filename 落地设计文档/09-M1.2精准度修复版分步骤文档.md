# M1.2 精准度修复版分步骤文档

本文档承接 `2026-06-09` 的 `M1.1 Final Verification` 与 `08-风险补漏版本修复架构.md`，用于把下一轮版本收敛为一个明确目标：先把“检索准不准”修到可评测、可回归、可决策，再讨论 M2 的可信检索与阅读提效。

M1.2 不是新功能版本。它的任务是修复当前精准度问题，补齐可比评测，建立 bad case 闭环，并在此基础上决定是否允许重新打开 `ENABLE_WEIGHTED_RERANK`，以及是否具备进入下一阶段的条件。

---

## 1. 当前起点

依据当前验收记录：

- `docs/development/m1.1-final-verification-20260609-123523.md`
- `落地设计文档/08-风险补漏版本修复架构.md`

当前已知结论：

| 项 | 当前状态 |
| --- | --- |
| 内部基础搜索 | 可用 |
| 对外灰度 | 不允许 |
| 新 rerank | 不允许启用 |
| 当前 product eval Top10 hit rate | `0.44` |
| 当前 product eval NDCG@10 | `0.1248` |
| baseline product eval Top10 hit rate | `0.48` |
| baseline product eval NDCG@10 | `0.134` |
| rerank 标准评测 | blocked，缺少可比 candidate corpus |

因此，M1.2 的第一优先级不是“继续加前台功能”，而是解决以下三个问题：

1. 当前排序为什么比 baseline 更差。
2. 哪些 bad case 是召回问题，哪些是 rerank 问题，哪些是评测口径问题。
3. 新排序何时才有资格重新进入灰度候选。

---

## 2. 版本目标

### 2.1 主目标

把当前系统从“基础搜索可用，但精准度证据不足”修到“精准度改动可被稳定评测、可被解释、可被回滚”。

### 2.2 成功标准

M1.2 结束时，至少要同时满足：

| 指标 | 门槛 |
| --- | --- |
| product eval 可重复运行 | 必须 |
| bad case 有原因标签 | 必须 |
| 当前方案 Precision@5 | 不低于 baseline |
| 当前方案 NDCG@10 | 不低于 baseline，目标正向提升 |
| 当前方案 Top10 hit rate | 从 `0.44` 提升，目标接近或达到 `0.60` |
| 新 rerank 是否启用 | 由评测结论决定，不允许拍脑袋开启 |

### 2.3 本版本不做

- 不做检索历史、收藏、导出、类案报告。
- 不做团队空间、权限、协作。
- 不做法院/法官倾向分析。
- 不做 M2 的来源引用、覆盖声明、低置信度候选产品化扩面。
- 不为了追求指标临时修改评测口径、伪造候选或手工“修结果”。

---

## 3. 执行原则

1. 先评测闭环，后算法修复。
2. 先识别 bad case 类型，后做权重调整。
3. 每一步都要能独立验收，不在同一轮里混入后续步骤。
4. 所有精准度改动都必须保留回滚能力。
5. 原始案情文本不得进入日志、埋点、评测报告正文。

---

## 4. 分步骤总览

建议把 M1.2 拆成 8 个独立步骤，每一步都可以单开一个 Codex 会话执行：

| 步骤 | 标题 | 目标 |
| --- | --- | --- |
| M1.2-1 | 基线冻结与 bad case 盘点 | 锁定当前基线，形成问题清单 |
| M1.2-2 | 可比评测线修复 | 打通标准评测线和产品本地评测线 |
| M1.2-3 | 查询理解与术语映射补强 | 修“用户怎么说”和“系统怎么理解”之间的损耗 |
| M1.2-4 | 候选召回修复 | 修“相关案例根本没进候选集”的问题 |
| M1.2-5 | rerank 融合修复 | 修“候选进来了但排序错了”的问题 |
| M1.2-6 | bad case 回归与参数定版 | 用回归集压实改动效果 |
| M1.2-7 | 精准度灰度预演 | 只预演开关和指标，不直接对外 |
| M1.2-8 | M1.2 验收与结论 | 输出 Go/No-Go 和下一阶段建议 |

## 5. 分步骤说明

## 5.1 步骤 M1.2-1：基线冻结与 bad case 盘点

### 新会话标题

```text
类案检索助手 M1.2-1 基线冻结与 bad case 盘点
```

### 目标

把当前 `M1.1` 的评测输出冻结下来，明确“问题到底出在哪”。

### 主要任务

1. 读取现有 `product_eval`、`rerank_eval`、`bad_cases`、性能和 gate 报告。
2. 生成一份 `M1.2 baseline snapshot`，记录：
   - baseline 与 current 的 Precision@5、NDCG@10、Top10 hit rate
   - 当前 feature flags 状态
   - 当前 blocked 项
3. 对现有 bad case 做第一轮归类，至少分成：
   - `RECALL_MISS`
   - `QUERY_UNDERSTANDING_GAP`
   - `RERANK_OVERWEIGHT`
   - `CAUSE_BIAS`
   - `KEY_PARAGRAPH_MISS`
   - `DATASET_MISMATCH`
4. 输出 Top 问题分布，作为后续步骤优先级依据。

### 产物

- `docs/development/m1.2-baseline-snapshot-*.md`
- `docs/development/m1.2-bad-case-clusters-*.json`

### 验收

- 能明确回答“当前退化主要来自召回、排序还是评测口径”。
- baseline 和 current 指标可被重复引用。

### 止损

- 如果现有 bad case 不足以支持分布分析，先补齐标注框架，不进入下一步调权重。

---

## 5.2 步骤 M1.2-2：可比评测线修复

### 新会话标题

```text
类案检索助手 M1.2-2 可比评测线修复
```

### 目标

解决“无法可靠证明新排序优于 baseline”的问题。

### 主要任务

1. 检查并修复 LeCaRDv2 评测线的 candidate corpus 可用性。
2. 如果标准评测线仍然不能直接评当前产品链路，建立产品本地评测线：
   - `product_eval_queries`
   - `product_eval_qrels`
   - 可重复 runner
3. 明确两条评测线的角色：
   - 标准评测线：检索学术/标准口径对照
   - 产品评测线：当前真实产品语料决策依据
4. 输出统一评测结果格式，避免后续脚本各写各的。

### 产物

- `data/eval/product_eval_queries*.jsonl`
- `data/eval/product_eval_qrels*.jsonl`
- `docs/development/m1.2-eval-preflight-*.md`
- 可重复执行的评测命令

### 验收

- 至少有一条评测线可以稳定跑出 current vs baseline 对比。
- blocked 原因不再是“没有 candidate corpus / 无法比较”这种基础问题。

### 止损

- 如果 LeCaRDv2 与当前产品语料天然不相交，就停止强凑统一口径，明确以产品本地评测线为发布决策依据。

## 5.3 步骤 M1.2-3：查询理解与术语映射补强

### 新会话标题

```text
类案检索助手 M1.2-3 查询理解与术语映射补强
```

### 目标

减少“用户描述进来了，但系统没理解到正确法律事实模式”的损耗。

### 主要任务

1. 从 bad case 中抽取高频口语表达和失败模式。
2. 补充或修正术语映射、行为模式映射、案由提示规则。
3. 区分高信心映射和低信心映射：
   - 高信心映射可参与加权
   - 低信心映射只扩展召回，不直接抬分
4. 校验改写结果不能丢失原始关键事实。
5. 对比“原 query 直搜”与“改写后召回”的收益与副作用。

### 产物

- 映射表或规则文件的版本化更新
- 失败 query -> 修复策略对照表

### 验收

- 典型 bad case 中，至少一批 `QUERY_UNDERSTANDING_GAP` 能被映射修复。
- 改写或映射不会把明显无关案由大面积带入候选集。

### 止损

- 如果映射只带来召回噪声，不进入默认路径，只保留为受控实验能力。

---

## 5.4 步骤 M1.2-4：候选召回修复

### 新会话标题

```text
类案检索助手 M1.2-4 候选召回修复
```

### 目标

修复“好案例根本没进候选集”的问题。

### 主要任务

1. 对 `RECALL_MISS` bad case 做召回路径复盘：
   - 原 query 向量召回
   - 改写 query 向量召回
   - BM25 召回
   - 合并去重后是否丢失
2. 调整候选生成策略，优先考虑：
   - 向量 + BM25 混合召回
   - 关键段落类型优先
   - 案由/年份/层级软过滤
3. 检查 TopK、去重和宽松召回阈值是否过早截断。
4. 对召回结果增加可分析字段，便于后续判断候选来源。

### 产物

- 候选召回对比报告
- 召回层参数或策略修复

### 验收

- 典型 `RECALL_MISS` 案例能进入候选集。
- 候选集变大后不能明显拖垮性能和排序稳定性。

### 止损

- 如果召回修复使噪声暴涨，先保守控制召回范围，再进入 rerank 修复。

---

## 5.5 步骤 M1.2-5：rerank 融合修复

### 新会话标题

```text
类案检索助手 M1.2-5 rerank 融合修复
```

### 目标

修复“候选进来了，但被错误排序压下去”的问题。

### 主要任务

1. 逐项复盘当前分数组成：
   - vector similarity
   - legal element overlap
   - case cause match
   - key paragraph match
   - authority signal
2. 找出过拟合或过重加权项，特别是：
   - 案由权重过重
   - 权威性弱信号被放大
   - 段落匹配误导
3. 为关键权重增加上限、保护规则或回退规则。
4. 保留 baseline 排序对照，确保随时可回滚。
5. 输出每类 bad case 修复前后的排序变化。

### 产物

- rerank 权重修复
- 分数组成对比报告

### 验收

- Precision@5 不低于 baseline。
- NDCG@10 至少不退化。
- Top3 不再出现明显“案由像但事实不相似”的错排。

### 止损

- 如果 rerank 修完仍不稳定，保留召回修复成果，但 `ENABLE_WEIGHTED_RERANK` 继续关闭。

---

## 5.6 步骤 M1.2-6：bad case 回归与参数定版

### 新会话标题

```text
类案检索助手 M1.2-6 bad case 回归与参数定版
```

### 目标

避免“修了 A，坏了 B”，把修复结果压进回归集。

### 主要任务

1. 建立固定回归集：
   - 高价值 bad case
   - 典型成功 case
   - 易受权重影响的边界 case
2. 对每次参数改动跑：
   - baseline
   - current before
   - current after
3. 输出每个 query 的前 10 变化。
4. 对回归结果打标签：
   - 改善
   - 持平
   - 退化
   - 不可比
5. 固化本轮可接受参数版本。

### 产物

- `m1.2-regression-report-*`
- 固化后的参数版本记录

### 验收

- 回归集可重复执行。
- 高价值 bad case 的改善具有可追踪证据。

### 止损

- 如果参数只能改善少数 case 但大面积伤害整体，撤回该轮参数。

## 5.7 步骤 M1.2-7：精准度灰度预演

### 新会话标题

```text
类案检索助手 M1.2-7 精准度灰度预演
```

### 目标

只预演“如果打开新排序，会怎么监控、怎么回退”，不是直接对外放量。

### 主要任务

1. 检查与精准度相关的 feature flags。
2. 设计灰度观察指标：
   - Top10 hit rate
   - Precision@5
   - NDCG@10
   - 二次搜索率
   - 无结果率
   - 降级触发率
3. 验证开关切换后：
   - 前端不白屏
   - `/api/search` 返回字段稳定
   - 回滚到 baseline 路径可用
4. 形成“启用条件”和“立即回退条件”。

### 产物

- `docs/development/m1.2-gray-readiness-*.md`

### 验收

- 有明确灰度条件，而不是“感觉比之前好一些”。
- 新排序是否允许进入候选，有明确门槛。

### 止损

- 如果评测结果仍模糊，结论只能是“继续关闭新排序，只保留评测与日志能力”。

## 5.8 步骤 M1.2-8：M1.2 验收与结论

### 新会话标题

```text
类案检索助手 M1.2-8 验收与结论
```

### 目标

用统一门禁给出 M1.2 的正式结论。

### 主要任务

1. 聚合测试、评测、性能、回滚、灰度预演结果。
2. 输出三类结论：
   - 基础搜索是否继续可用
   - 新 rerank 是否允许成为灰度候选
   - 是否可以进入 M2 可信检索准备
3. 输出未解决问题列表和下一版建议。

### 产物

- `docs/development/m1.2-final-verification-*.md`
- `docs/development/m1.2-release-gate-*.json`

### 验收

- 有明确 Go/No-Go。
- 有明确默认开关状态。
- 有明确下一阶段入口条件。

### 止损

- 如果评测仍不能证明正向，M1.2 结论应为“继续关闭新排序，不进入能力扩面”。

## 6. 执行顺序

建议严格按顺序执行：

1. `M1.2-1 基线冻结与 bad case 盘点`
2. `M1.2-2 可比评测线修复`
3. `M1.2-3 查询理解与术语映射补强`
4. `M1.2-4 候选召回修复`
5. `M1.2-5 rerank 融合修复`
6. `M1.2-6 bad case 回归与参数定版`
7. `M1.2-7 精准度灰度预演`
8. `M1.2-8 验收与结论`

不建议跳步。特别是：

- 没有 `M1.2-2` 的可比评测线，不要直接做 `M1.2-5` 调权重。
- 没有 `M1.2-6` 的回归集，不要直接做 `M1.2-7` 灰度预演。

## 8. M1.2 Definition of Done

M1.2 完成必须同时满足：

- 有冻结后的 baseline snapshot。
- 有可重复执行的产品本地评测线。
- bad case 有稳定的原因标签体系。
- 至少一轮精准度修复能拿出 current vs baseline 对比证据。
- 新 rerank 的启用与否有正式门禁，不依赖主观判断。
- 所有改动保留回滚能力。
- 没有把 M2、M3、M4 的功能提前混进来。
