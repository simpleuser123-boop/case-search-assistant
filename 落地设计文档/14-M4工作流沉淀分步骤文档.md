| --- | --- |

| M3 阅读提效完成 | `GO` |

## 1. M4 定位

M4 对应 `01-演进计划总览.md` 中的 `M4 工作流沉淀`，在 `07-开发步骤.md` 中落到 `Sprint 5：沉淀与导出`，覆盖功能拆解中的 `F16 检索历史草稿`、`F17 收藏与导出类案清单`、`F18 类案报告生成（轻量模板）` 以及团队复用能力评估。

### 1.1 版本目标

让一次检索不再是"用完即走"，而是可以被保留、复用、产出交付物：

1. 提供本地/会话级检索历史和草稿恢复，不把用户原始案情上送后端。
2. 提供收藏能力，收藏的是案例的元数据和引用，不是正文。
3. 支持把当前选择的若干案例组装成"类案清单"，作为本次工作的可复用集合。
4. 支持把类案清单导出为文件（如 Markdown / CSV），导出内容只含元数据和来源链接，不含裁判文书正文。
5. 支持基于清单生成"轻量报告模板"，报告骨架由用户/系统填入元数据和来源锚点，不自动起草法律意见、不输出胜负结论。
6. 评估团队复用的前置条件（账号、共享、权限），只做能力评估与最小开关，不在 M4 实现完整团队空间。

### 1.2 成功标准

M4 结束时，至少要同时满足以下条件，才允许讨论进入 M5 商业化扩展：

| 指标 | 门槛 |
| --- | --- |
| 持久层隐私 | 历史、草稿、收藏、清单、导出、报告的持久化内容 `100%` 只含元数据、`case_id`、来源引用、用户标签/备注等结构化字段，不含原始 query、案情正文、候选正文、chunk 正文或裁判文书长正文 |
| 用户输入隐私 | 用户原始案情不上送后端、不写入服务端持久层；本地草稿/历史可由用户清除 |
| 来源锚点继承 | 清单、导出、报告中引用的案例侧 AI 加工内容必须可追溯到 `case_id` 和 `source_chunk_id`，无锚点内容不进入交付物 |
| 收藏/清单边界 | 收藏与清单只保存引用与用户自填字段，不复制并存储案例正文，不据此改变主排序 |
| 导出边界 | 导出文件只含元数据、来源链接、用户备注；不含正文、不含胜负概率、不含确定性法律结论 |
| 报告边界 | 报告为模板骨架 + 元数据 + 来源锚点；不自动起草代理词/诉状/答辩状，不输出诉讼结果判断 |
| 文案边界 | 全程不使用"已查全""保证无遗漏""查全率"，不输出胜诉/败诉概率或确定性法律结论 |
| 团队复用边界 | 仅做能力评估、数据结构预留和最小可关闭开关，不实现完整权限/团队空间/批量导入 |
| 默认排序 | 不改变主结果排序、source selection、rerank 默认开关和 M2/M3 默认行为 |
| 离线评测隔离 | qrels、label、relevance 只用于离线评测，不进入历史、收藏、清单、导出或报告决策 |
| 性能 | 主搜索 warm P95 继续 `< 3s`；M4 沉淀能力可懒加载、失败降级，不造成主链路白屏 |
| 回滚 | 新增 M4 能力可通过 feature flag 或安全降级关闭；关闭后基础搜索、M2、M3 行为不变 |
| 正文泄露 | 任何产物（报告、JSON、日志、测试快照、导出文件、埋点）不得包含正文型内容 |

### 1.3 本版本不做

- 不做 M5 的完整团队空间、角色权限、批量导入、法院/法官倾向分析。
- 不自动起草代理词、诉状、答辩状或庭审提纲。
- 不输出胜诉概率、败诉概率、风险定级分数或确定性法律结论。
- 不承诺绝对覆盖，不使用"已查全""保证无遗漏""查全率"等话术。
- 不默认开启 `ENABLE_WEIGHTED_RERANK`。
- 不改变 M1/M2/M3 已验证的线上排序、召回、source selection、扩展检索或阅读辅助默认行为。
- 不让 qrels、label、relevance 参与历史、收藏、清单、导出、报告或排序决策。
- 不根据 query id、case id 做排序、收藏、清单、导出或报告特判。
- 不把用户原始案情、候选正文、chunk 正文或裁判文书长正文写入持久层、开发报告或导出文件。
- 不把收藏/清单/历史变成绕过来源锚点和隐私门禁的"正文缓存"。

## 2. 执行原则

1. 先做工作流沉淀合同（含持久层数据边界），再做历史、收藏、清单、导出、报告的具体能力。
2. 任何写入持久层的字段都必须先在合同中声明为"允许持久化"，未声明的字段一律不落库。
3. 用户原始案情只在本地草稿/历史中临时保存，可清除；不上送后端持久层。
4. 收藏、清单、导出、报告引用案例时，只引用元数据和来源锚点，不复制正文。
5. M4 能力默认是检索之后的沉淀辅助，不反向影响主搜索排序、召回和 source selection。
6. 每个沉淀模块都必须有隐藏、降级、清除或关闭路径。
7. 文案必须表达"工作记录""可复核引用""待人工确认"，不能表达"结论已确定"或"已查全"。
8. 每一步都要输出 Go/No-Go；任何隐私、来源、绝对话术、排序污染、持久层正文落库门禁失败都必须 `NO_GO`。

## 3. 分步骤总览

建议把 M4 拆成 8 个独立步骤，每一步可以单开一个会话执行：

| 步骤 | 标题 | 目标 |
| --- | --- | --- |
| M4-1 | 工作流沉淀入口合同 | 冻结 M3 GO 证据，定义 M4 API/UI/持久层/日志/feature flag 合同与持久化字段白名单 |
| M4-2 | 检索历史与草稿恢复 | 提供本地/会话级历史与草稿，不上送原始案情，可清除 |
| M4-3 | 案例收藏能力 | 收藏案例的元数据与引用，可增删、可清除，不存正文、不改排序 |
| M4-4 | 类案清单组装 | 把当前选择的若干案例组装成可复用清单，受控、可编辑、可删除 |
| M4-5 | 类案清单导出 | 把清单导出为文件（Markdown/CSV），只含元数据、来源链接、用户备注 |
| M4-6 | 轻量报告模板生成 | 基于清单生成报告骨架，含元数据与来源锚点，不起草、不下结论 |
| M4-7 | 团队复用能力评估 | 评估账号/共享/权限前置条件，预留数据结构与最小开关，不实现团队空间 |
| M4-8 | M4 验收与 M5 入口结论 | 汇总门禁、测试、隐私、性能、回滚和下一阶段 Go/No-Go |

## 4.2 步骤 M4-2：检索历史与草稿恢复

### 目标

提供本地/会话级检索历史和草稿恢复，让用户能快速回到上一次或最近几次检索。用户原始案情只在浏览器本地保存、可清除，不上送后端持久层。

4. 隐私规则：
   - 后端日志和埋点只记录脱敏字段（hash、count、status、reason code）。

## 4.3 步骤 M4-3：案例收藏能力

### 新会话标题

```text
类案检索助手 M4-3 案例收藏能力
```

### 目标

允许用户收藏感兴趣的案例，便于后续复用。收藏保存的是案例元数据和来源引用，不复制并存储裁判文书正文，不据此改变主排序。

### 开始前必须读取

- M4-1 产物
- M4-2 产物
- `落地设计文档/04-数据层设计.md`
- `apps/api/app`
- `apps/web/src`

### 主要任务

1. 定义 `case_favorite` 数据结构：
   - `case_id`
   - `case_number`
   - `court` / `trial_level` / `case_cause` / `judgment_date`
   - `source_anchors`（可选，用于回跳详情）
   - 用户自填 `note` / `tag`（短字段）
   - `created_at` / `favorite_status`
2. 收藏边界：
   - 只保存白名单内的元数据、引用和用户自填短字段。
   - 不复制并存储案例正文、摘要正文、chunk 正文。
   - 收藏可增、可删、可清空。
3. 展示规则：
   - 结果卡片、详情抽屉、对比视图可收藏/取消收藏。
   - 收藏列表展示元数据，并可回跳到案例详情。
