M5 的目标是把"单个律师能把检索沉淀下来反复使用"推进为"律所/团队能安全地共享、协作复用类案工作产物，并在此基础上形成可持续的商业化闭环"。

## 1. M5 定位

M5 对应 `01-演进计划总览.md` 中的 `M5 商业化扩展`（面向律所采购与专业场景），承接 `07-开发步骤.md` 的 `Sprint 5：沉淀与导出` 之后的「团队空间、权限、协作」「法官/法院倾向分析」后续 Sprint，覆盖功能拆解中的 `F17 收藏与导出类案清单`（依赖项中的"账号体系"部分）、`F19 法院/法官倾向分析`，以及 `M4-7 团队复用能力评估` 中标记为 `not_ready` 的五项能力载体：

1. `account_system` 账号体系（用户身份与认证）
2. `list_sharing` 清单/收藏共享
3. `permission_tiering` 权限分级
4. `team_workspace_isolation` 团队空间与数据隔离
5. `bulk_import` 批量导入

加上 F19 法院/法官倾向分析与商业化/计费闭环。

### 1.1 版本目标

让产品从"单用户工具"升级为"团队可采购、可协作、可付费"的专业服务：

1. 建立账号体系：引入服务端用户身份与认证，把当前匿名单用户态映射到具名 `user_id`。
2. 建立团队空间与数据隔离：引入 `team_id`，存储与查询按团队强隔离，跨团队默认不可见。
3. 建立权限分级：owner / editor / viewer 角色与对象级访问控制。
4. 支持团队内共享：把历史、收藏、清单、报告从本地迁移/同步到服务端按 owner 归属的存储，默认私有、显式共享。
5. 支持批量导入：既有清单/案例批量进入团队空间，导入即元数据/引用 only，绝不引入正文。
6. 提供法院/法官倾向分析（F19）：仅在数据质量门禁达标时提供，以统计透明、可追溯、不预测个案结果为前提。
7. 形成商业化闭环：套餐、试用、计费、续费意愿采集；不在工具内代填任何支付凭据。

### 1.2 落地基调（本版本沿用 M4 的谨慎基调）

M5 的实现深度遵循「**先合同 + 骨架，敏感项谨慎落地**」：

- 账号、鉴权、隔离、计费先做合同与最小骨架，每一步独立 Go/No-Go。
- 凭据类（密码、SSO 令牌、支付卡号、银行账户）**绝不在工具内代填、代管、代存明文**；只设计安全的存储与校验流程，由用户/平台侧完成敏感输入。
- 跨用户可见性**默认关闭**；任何默认开启跨用户可见性即触发 `NO_GO`。
- 每个新能力都有 feature flag（默认 `false`）与降级/关闭路径；关闭后回到 M4 末态（单用户、纯前端沉淀）。

### 1.3 成功标准

M5 结束时，至少要同时满足以下条件，才允许宣称商业化扩展可用：

| 指标 | 门槛 |
| --- | --- |
| 多租户隔离 | 跨团队、跨用户数据 `100%` 强隔离；`team_id`/`owner_user_id` 为空时行为与 M4 单用户私有完全一致 |
| 访问控制 | 对清单/收藏/报告的每一次读写都经过对象级鉴权；越权访问被拒绝并审计 |
| 凭据安全 | 不在工具内代填/代管支付或登录凭据；密码不明文存储；令牌不写入日志/产物 |
| 共享默认私有 | 所有可共享对象默认 `visibility=private`；共享需显式授权动作 |
| 持久层隐私 | 服务端持久层 `100%` 只含元数据、`case_id`、来源引用、用户自填短字段；不含原始 query、案情正文、候选/chunk/裁判文书正文 |
| 来源锚点继承 | 团队共享/导入/报告中案例侧 AI 内容 `100%` 可追溯 `case_id` + `source_chunk_id`，无锚点不进入交付物 |
| 倾向分析边界 | F19 仅在数据质量门禁达标时输出；标注样本量与覆盖范围；不预测个案胜负、不输出确定性结论 |
| 文案边界 | 全程不使用"已查全""保证无遗漏""查全率"，不输出胜诉/败诉概率或确定性法律结论 |
| 默认排序 | 不改变主结果排序、source selection、rerank 默认开关和 M2/M3/M4 默认行为 |
| 离线评测隔离 | qrels、label、relevance 只用于离线评测，不进入账号、团队、共享、导入、倾向分析或排序决策 |
| 性能 | 主搜索 warm P95 继续 `< 3s`；M5 能力可懒加载、失败降级，不造成主链路白屏 |
| 回滚 | 新增 M5 能力均可通过 feature flag 或安全降级关闭；关闭后基础搜索、M2/M3/M4 行为不变 |
| 正文泄露 | 任何产物（持久层、报告、JSON、日志、测试快照、导出文件、埋点）不得包含正文型内容 |

### 1.4 本版本不做

- 不在工具内代填、代管或代存任何支付凭据、银行账户、登录密码或 SSO 令牌。
- 不默认开启跨用户/跨团队数据可见性。
- 不实现先于账号体系与鉴权中间件的共享或团队功能（顺序依赖必须满足）。
- 不让 F19 倾向分析预测个案诉讼结果、输出胜负概率或确定性法律结论。
- 不在数据质量门禁未达标时上线 F19。
- 不默认开启 `ENABLE_WEIGHTED_RERANK`。
- 不改变 M1/M2/M3/M4 已验证的线上排序、召回、source selection、扩展检索、阅读辅助或沉淀默认行为。
- 不让 qrels、label、relevance 参与账号、团队、共享、导入、倾向分析或排序决策。
- 不根据 query id、case id 做排序、共享、导入或倾向分析特判。
- 不把用户原始案情、候选正文、chunk 正文或裁判文书长正文写入服务端持久层、开发报告或导出文件。

## 2. 执行原则

1. 先做商业化扩展合同（含多租户数据边界、鉴权边界、凭据安全边界），再做账号、隔离、权限、共享、导入、倾向分析、计费的具体能力。
2. 严格遵守能力顺序依赖：账号体系 → 团队空间隔离 → 权限分级 → 共享 → 批量导入；前一步未 GO 不得开始后一步。
3. 任何写入服务端持久层的字段都必须先在合同中声明为"允许持久化"；未声明字段一律不落库；M4 字段白名单继续生效，只新增 owner/visibility/team_id/role 等结构化关系字段，不放宽正文禁令。
4. 凭据类敏感输入由用户/平台侧完成；工具只设计安全流程，不代填、不代管明文，不把令牌/密码写入任何产物或日志。
5. 跨用户可见性默认关闭；共享必须显式授权；隔离默认单租户私有，`team_id` 为空等同当前单用户私有行为。
6. F19 倾向分析以数据质量门禁为前置，统计透明、标注样本量、可追溯到来源，不预测个案、不下结论。
7. M5 能力默认是检索/沉淀之上的扩展，不反向影响主搜索排序、召回和 source selection。
8. 每个能力都必须有隐藏、降级、清除或关闭路径；关闭后回到 M4 末态。
9. 每一步都要输出 Go/No-Go；任何隐私、来源、绝对话术、排序污染、持久层正文落库、跨用户越权、默认开启共享、凭据代管门禁失败都必须 `NO_GO`。

## 3. 分步骤总览

建议把 M5 拆成 10 个独立步骤，每一步可以单开一个会话执行。前 6 步严格按能力顺序依赖推进：

| 步骤 | 标题 | 目标 |
| --- | --- | --- |
| M5-1 | 商业化扩展入口合同 | 冻结 M4 GO 证据，定义 M5 多租户/鉴权/凭据安全/共享/计费合同与服务端持久化字段白名单 |
| M5-2 | 账号体系与认证骨架 | 引入服务端用户身份与认证骨架，把匿名单用户态映射到 `user_id`；凭据不在工具内代管明文 |
| M5-3 | 团队空间与数据隔离 | 引入 `team_id`，存储与查询按团队强隔离，跨团队默认不可见 |
| M5-4 | 权限分级与对象级访问控制 | owner/editor/viewer 角色与鉴权中间件，对清单/收藏/报告做对象级访问控制 |
| M5-5 | 沉淀同步与团队共享 | 把本地沉淀迁移/同步到服务端按 owner 归属存储，默认私有、显式共享 |
| M5-6 | 批量导入 | 既有清单/案例批量进入团队空间，导入即元数据/引用 only，锚点完整性校验 |
| M5-7 | 法院/法官倾向分析（F19）数据门禁 | 评估并实现 F19 的数据质量门禁，未达标不上线分析 |
| M5-8 | 法院/法官倾向分析（F19）展示与边界 | 在门禁达标前提下输出统计透明、可追溯、不预测个案的倾向分析 |
| M5-9 | 商业化闭环（套餐/试用/计费/续费意愿） | 设计套餐与计费流程；不在工具内代填支付凭据；采集续费意愿 |
| M5-10 | M5 验收与商业化就绪结论 | 汇总门禁、测试、隐私、安全、性能、回滚和商业化就绪 Go/No-Go |

- 持久层、报告、日志、JSON、导出文件、共享对象、导入结果或测试快照泄露正文。
- 出现密码明文、SSO/OAuth 令牌、支付卡号、银行账户、CVV 或政府证件号落库/入日志，或工具内代填支付/登录凭据。
- 跨团队/跨用户数据串读，或对象级越权读写未被拦截。
- 跨用户可见性默认开启，或共享默认非私有。
- 用户原始案情被上送服务端持久层。
- 共享/导入/报告/F19 引用无来源 AI 内容。
- F19 在数据门禁未达标时上线，或输出个案胜负预测/概率/确定性法律结论。
- 出现"已查全""保证无遗漏""查全率"等绝对话术。
- qrels、label、relevance 进入账号、团队、共享、导入、倾向分析或排序。
- query id / case id 被用于运行时特判。
- 任一 M5 flag、`ENABLE_WEIGHTED_RERANK` 或 M4 flag 被默认打开。
- M4-8 核心门禁回归。
- M5 能力无法回滚，或导致主链路不可用。

### 如果 GO

M5 是当前演进路线图的末段里程碑。M5-10 GO 后，下一阶段建议进入**运营与规模化**（数据规模化迁移 Milvus、F19 持续扩样、商业化指标跟踪与迭代），可另起里程碑文档：

```text
类案检索助手 M6-1 运营与规模化入口合同
```

### 如果 NO_GO

报告中必须列出：

- 失败步骤。
- 失败门禁。
- 受影响能力。
- 是否需要回滚。
- 是否阻止宣称商业化就绪。
- 下一轮建议从哪个 M5 子步骤继续。
